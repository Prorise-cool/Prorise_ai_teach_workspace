## 一、项目背景

### 1.1 赛事约束
- **赛事名称**：教育智能体应用创新赛（高职组）
- **平台要求**：腾讯云智能体开发平台（仅用基础设施）
- **提交截止**：2026/4/25（约 5 周）
- **高职组定位**：基础教学智能体
- **必交材料**：方案说明书、演示视频（≤3分钟）、答辩 PPT、可访问链接

### 1.2 现有基础
- 已有 **ManimToVideoClaw** 项目，完整的视频生成流水线
- 支持：分镜生成 → Manim 代码 → 渲染 → TTS → 上传
- 多 TTS 支持：豆包、百度、Spark、Kokoro

### 1.3 开发策略
- **不是在示例项目上开发**，而是全新开发
- 吸收 ManimToVideoClaw 的精华和能力

## 二、产品定义

### 2.1 核心问题

**用户痛点**：
1. 理解抽象概念（数学公式/算法逻辑太抽象）
2. 学习效率低（看书/看课太慢）
3. 缺乏个性化（现有视频不够针对性）
4. 教师备课负担（制作高质量讲解视频太耗时）

**差异化价值**：
1. 动画质量（3Blue1Brown 级别）
2. 生成速度（输入 → 分钟级出视频）
3. 教学效果（更懂教学法）
4. 多风格 Agent（选择老师 = 选择 Agent）

### 2.2 目标用户

| 用户类型 | 核心场景 |
|----------|----------|
| 高职学生 | 课后复习、考前冲刺 |
| 高职教师 | 备课、制作教学视频 |
| 自学者 | 按需学习、快速理解 |

### 2.3 输入与输出

**输入方式**：
- 文字输入（题目/知识点）
- 图片上传（拍照识别）
- 语音输入（语音提问）

**输出**：
- 带配音的讲解视频
- 多种老师风格可选

### 2.4 MVP 功能范围

**必须包含**：
- ✅ 用户系统（登录/注册/个人中心）
- ✅ 单题生成（核心流程跑通）
- ✅ 学习记录（历史/收藏/重看）
- ✅ 播放体验（播放器/进度/倍速）

**后期迭代**：
- 批量生成
- 风格模板
- 学习分析
- 作业系统

## 三、技术架构（对齐 OpenMAIC 融合方案）

### 3.1 架构原则

> **当前阶段**：ToC 用户端 MVP
> **架构目标**：支持无缝扩展 ToB 管理端

### 3.2 前端 Monorepo 架构（前端分离策略）

```
前端 Monorepo
├── packages/
│   ├── ui-common/      ← 共享 UI 组件库（待建）
│   ├── auth/          ← 统一登录/RBAC 上下文
│   ├── api-client/    ← 对接两个后端 API
│   ├── user/          ← ToC 用户端（当前阶段开发）
│   └── admin/         ← ToB 管理端（架构预留）
```

**当前阶段**：只开发 `packages/user`（ToC 用户端）
**扩展 ToB**：复用 `ui-common` + `auth` + `api-client`，新增 `packages/admin`

### 3.3 后端架构

| 服务 | 技术栈 | 端口 | 说明 |
|------|--------|------|------|
| **RuoYi-Vue-Plus (Java)** | Spring Boot | 8080 | 管理服务（租户/用户/RBAC）- 架构预留 |
| **OpenMAIC-FastAPI (Python)** | FastAPI + LangGraph | 8090 | 功能微服务（当前阶段使用） |

### 3.4 OpenMAIC-FastAPI 内部结构

```
OpenMAIC-FastAPI (Python, 8090)
├── packages/
│   ├── openmaic/      ← LangGraph + 多 Agent 流水线
│   │   ├── agents/    ← 严肃老师 | 幽默老师 | 耐心老师 | 高效老师
│   │   └── pipeline/  ← 分镜 → 代码 → 渲染 → TTS
│   └── manim/         ← Manim 渲染器（合并自 ManimToVideoClaw）
└── services/
    └── [TTS 多态支持]
```

**部署优化**：ManimToVideoClaw 原 4 端口 → 合并为单 FastAPI 应用

### 3.5 技术选型

| 层级 | 技术选择 | 说明 |
|------|----------|------|
| **前端** | React 19 + Vite + TypeScript | Monorepo 统一技术栈 |
| **后端** | FastAPI + LangGraph | Python 生态，多 Agent 支持 |
| **管理后端** | Java Spring Boot (RuoYi) | 架构预留 ToB |
| **动画引擎** | Manim | 吸收 ManimToVideoClaw 精华 |
| **云平台** | 腾讯云 | 赛事要求基础设施（COS/云函数/数据库） |
| **TTS** | 多态设计 | 可插拔，选择 TTS = 选择老师 |

## 四、多 Agent 设计（对齐架构）

### 4.1 老师风格类型

| Agent 类型 | 语音特点 | 视觉风格 | 内容深度 | 互动频率 |
|------------|----------|----------|----------|----------|
| **严肃老师** | 稳重、清晰 | 简洁、专业 | 深入、严谨 | 少 |
| **幽默老师** | 活泼、有趣 | 多彩、动感 | 轻松、案例多 | 中 |
| **耐心老师** | 温和、慢速 | 温暖、柔和 | 细致、步骤多 | 高 |
| **高效老师** | 快速、干练 | 极简、高效 | 精炼、直击要点 | 少 |

### 4.2 风格维度

每种 Agent 在以下四个维度有差异化配置：
1. **语音风格**：语速、语调、节奏
2. **视觉风格**：动画节奏、配色、字体
3. **内容风格**：讲解深度、案例选择、步骤拆解
4. **互动风格**：习题数量、提问频率、鼓励语

### 4.3 选择机制

- **生成前选择**：用户主动选择老师类型
- **个性化匹配**：根据学习历史智能推荐

## 六、商业模式与阶段策略

### 6.1 当前阶段（赛事 ToC MVP）

| 多租户管理 | RuoYi 后端预留 | SaaS 商业模式 |

## 九、关键决策记录（对齐 OpenMAIC 融合架构）

| 决策点 | 选择 | 原因 |
|--------|------|------|
| **开发阶段** | ToC MVP | 当前赛事聚焦，ToB 架构预留 |
| **前端架构** | React Monorepo | 统一技术栈，ToC/ToB 隔离部署 |
| **后端架构** | FastAPI + LangGraph | 吸收 OpenMAIC 精华，多 Agent 支持 |
| **管理后端** | RuoYi-Vue-Plus (Java) | 架构预留 ToB，赛事只做 ToC |
| **OpenMAIC 处理** | 提取代码重构 | 价值在 LangGraph + 流水线，非 Next.js |
| **ManimToVideoClaw** | 合并为单 FastAPI | 4 端口 → 1 端口，降低复杂度 |
| **Agent 模式** | 多 Agent | 选择老师 = 选择 Agent，风格差异化 |
| **TTS 方案** | 多态设计 | 可插拔，选择 TTS = 选择老师风格 |
