## P0: LLM Stream 524 超时 — manim_gen 阶段间歇性返回 None

### 现象
- 管道 manim_gen 阶段报错: `Code generation LLM call returned None`
- 有时成功（vtask_20260415105804_bf54607c 成功生成 10 sections），有时失败
- 失败时，worker 耗时 ~8 分钟后返回 None

### 最小可行性测试结果
```
短 prompt stream:     OK (返回 '好')
短 prompt non-stream:  OK (返回 4786 tokens 代码)
长 prompt stream:     FAIL — 524 A Timeout Occurred
长 prompt non-stream: OK (之前验证过能返回完整代码)
```

### 根因
`synai996.space` 是一个 CDN 代理，对**大 payload 的 stream 连接建立**返回 524 超时。
Code generation 的 prompt 很大（system prompt + API codebook + user prompt，合计上万字符），
CDN 在建立 SSE stream 连接时等太久直接返回 524，此时还没收到任何 chunk，partial recovery 也救不了。

### 代码路径
```
_call_stream_primary() → httpx.stream() → 524（无 chunk 接收）
  ↓ 返回 (None, {})
_call_non_stream() → 可能也超时
  ↓
3次重试全挂 → _call_openai_compatible 返回 (None, usage_info)
  ↓
code_generator.py:82 → ValueError("Code generation LLM call returned None")
```

### 涉及文件
- `packages/fastapi-backend/app/features/video/pipeline/engine/gpt_request.py`
  - `_call_stream_primary()` — stream 调用，长 prompt 时 524
  - `_call_non_stream()` — non-stream fallback
  - `_call_openai_compatible()` — 重试逻辑：stream first → non-stream fallback → 3次重试

### 可能的修复方向
1. **智能 stream 策略**: 对大 prompt（>某个阈值字符数）跳过 stream，直接 non-stream
2. **缩短 prompt**: API codebook 注入太大，可以压缩或按需注入
3. **换 provider**: 用更稳定的 LLM 代理（不经过 CDN 或 CDN 超时配置更大）
4. **stream 建立超时分离**: 给 stream 连接建立阶段设短超时，建立成功后用长超时读数据

### Provider 信息
```
base_url: https://synai996.space
model:    gemini-3.1-pro-high
path:     /v1/chat/completions
timeout:  300s
```
