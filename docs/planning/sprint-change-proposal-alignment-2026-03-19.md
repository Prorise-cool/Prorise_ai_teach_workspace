# Sprint Change Proposal：市场分析与架构对齐

**日期**: 2026-03-19
**触发问题**: 市场分析结论与讨论架构不一致
**状态**: ✅ 已对齐

---

## 一、问题摘要

### 1.1 原始不一致

| 维度 | 市场研究报告 | 头脑风暴总结 | 架构讨论 |
|------|-------------|-------------|---------|
| **目标客户** | B端为主 | C端+教师 | ToC用户端 + ToB管理端 |
| **商业模式** | 院校采购 | 免费+订阅 | 两个独立端 |
| **核心价值** | 教学效率+规模化 | 个性化学习 | 多租户+RBAC |

### 1.2 问题根因

- 市场研究发现了 VideoTutor 的 B 端能力空白
- 头脑风暴阶段聚焦 ToC 产品功能
- OpenMAIC 架构讨论引入 ToB 管理端复杂性
- **三个阶段结论未统一**

---

## 二、对齐方案

### 2.1 核心原则

> **当前阶段**：只做 ToC（赛事 MVP）
> **架构目标**：支持无缝扩展 ToB

### 2.2 对齐后的产品定位

**一句话定位**：
> "面向高职学生的 AI 教学视频生成智能体（ToC），架构预留 ToB 扩展能力"

**阶段策略**：

| 阶段 | 目标 | 核心工作 |
|------|------|----------|
| **赛事阶段** | ToC MVP | 用户端 + OpenMAIC-FastAPI |
| **赛后拓展** | ToB 管理端 | RuoYi 后端 + packages/admin |

### 2.3 对齐后的架构

```
前端 Monorepo
├── ui-common/      ← 共享 UI 组件（ToC + ToB 共用）
├── auth/          ← 统一登录上下文
├── api-client/    ← 对接后端 API
├── user/          ← 【当前开发】ToC 用户端
└── admin/         ← 【架构预留】ToB 管理端

后端
├── RuoYi-Vue-Plus (Java, 8080)  ← 【架构预留】ToB 管理服务
└── OpenMAIC-FastAPI (Python, 8090) ← 【当前使用】ToC 功能服务
```
