# RuoYi 开发联调常见坑与防呆清单

> **状态**：已更新
> **负责人**：Story Owner / 后台模块 Owner
> **最后更新**：2026-04-05
> **适用范围**：`packages/RuoYi-Vue-Plus-5.X/`、`packages/ruoyi-plus-soybean/`、`packages/student-web/` 与 `xm_*` 业务表相关 Story
> **关联事实源**：`_bmad-output/implementation-artifacts/1-5-用户配置系统（个人简介与学习偏好）.md`

---

## 文档目标

这份文档只做一件事：把 `RuoYi` 业务开发、代码生成、管理端联调和学生端对接里最容易反复踩的坑，收敛成一份长期可复用的防呆清单。

适用原则：

1. 先看事实源，再改代码。
2. 先对齐数据库和契约，再跑生成器。
3. 生成器只负责骨架，不负责业务真相。
4. 后台管理端、学生端、后端接口必须按同一套字段语义联动。

---

## 核心结论

1. 数据库字段类型、默认值、字典值和注释不对齐，后面的 `Bo / Vo / Service / 前端搜索 / 字典回显` 一定会连锁出错。
2. `RuoYi` 代码生成器只能解决标准 CRUD 骨架，凡是“关联字段、JSON 字段、URL 字段、登录态自服务接口、真实业务搜索条件”，都要手工补。
3. 管理端页面不能把数据库存储形式当成最终展示形式。JSON 不是直接展示文本，URL 也不一定是 `ossId`。
4. 学生端 onboarding 的页面流转，必须按“业务采集完整性”设计，不能被某个表单字段是否填写绑架。

---

## 第一类坑：数据库没有对齐，后面全错

### 1. 字段类型和默认值没先对齐

典型表现：

- SQL 里是 `tinyint(1)`，Java 里写成 `Long`、`Boolean`、`Integer` 混用。
- 默认值、空值语义不清楚，导致新增记录和更新记录行为不一致。
- 字段注释没有写清真实业务含义，生成器导入后标签和控件类型全猜错。

这次 `Story 1.5` 的直接教训：

- `is_completed` 虽然库里是 `tinyint(1)`，但后台查询、字典展示、前端搜索都必须统一桥接。
- `teacher_tags` 在库里是 `varchar`，但业务语义是 JSON 数组字符串，不是普通字符串。
- `avatar_url` 的真实语义是“头像地址”，不等于“OSS 主键”。

防呆规则：

1. 建表 SQL 落仓后，先检查字段类型、默认值、注释、索引，再导入生成器。
2. 业务字段注释必须直接写清“存什么”和“怎么用”，不要只写模糊中文名。
3. 只要字段真实语义不是纯文本，就要在 Story / 模块文档里明确写出来。

---

## 第二类坑：字典值以为是 `true/false`，实际上系统用的是 `Y/N`

典型表现：

- 管理端查询框选择“是/否”后，后端报类型转换错误。
- 列表里 `DictTag` 不回显。
- 前端把 `true / false` 直接提交给 `Long` 或 `tinyint` 字段。

这次 `Story 1.5` 的直接教训：

- `sys_yes_no` 在 RuoYi 标准字典里的键值是 `Y / N`，不是 `true / false`。
- 但我们的业务表 `is_completed` 存的是 `0 / 1`，所以展示和搜索必须做桥接转换。

防呆规则：

1. 任何系统字典先查标准 SQL，不要凭印象写值。
2. 只要数据库值和字典值不是同一套编码，就在前后端各自做显式转换。
3. 搜索表单的数据类型必须和后端查询对象一致，不能把字符串直接喂给数字字段。

---

## 第三类坑：生成器导了表，不代表业务就能直接用

典型表现：

- 生出来的 CRUD 页面能开，但一搜索、一编辑、一展示就错。
- 生成器把“必填”直接固化成前端共享校验，新增和编辑都被同一条规则卡住。
- 以为生成器会自动补关联字段，结果列表只能看到 `userId`。

这次 `Story 1.5` 的直接教训：

- `userName` / `nickName` 不是表字段，生成器不会自动产出关联展示。
- `avatarUrlUrl` 这种翻译字段也不会自动补。
- `isCompleted` 一旦在生成器里被错误勾成前端必填，管理员编辑未完整配置的记录就无法保存。

防呆规则：

1. 生成器只当骨架脚手架，不当最终交付物。
2. 导表后必须写一份“生成后手工调整清单”，并和 Story 同步。
3. 下列场景默认需要手工补代码：
   - 关联展示字段
   - 登录用户自服务接口
   - JSON 字段读写
   - URL / 文件字段兼容处理
   - 非表字段搜索
   - 字典值桥接

---

## 第四类坑：把 URL 当 OSS ID，用错组件

典型表现：

- 后台编辑抽屉一打开就报 `the backend request error`
- 管理端图片列不显示图片，只显示一段字符串。
- `MinIO` 管理页能预览，但前端页面还是报错。

这次 `Story 1.5` 的直接教训：

- `avatar_url` 可能直接存的是完整 URL，也可能是翻译前的 OSS 值。
- 如果后台编辑组件强行使用 `OssUpload` 或 OSS 翻译逻辑，遇到纯 URL 就会发错请求。

防呆规则：

1. 先确认数据库里存的是“资源 ID”还是“可直接访问 URL”。
2. 列表展示优先做“兼容解析”：
   - 先用翻译后的 URL。
   - 没有翻译结果时，再判断原值是不是完整 URL。
3. 编辑抽屉遇到 URL 型字段，优先使用“文本输入 + 预览”，不要默认上 OSS 组件。

---

## 第五类坑：JSON 字段直接原样渲染

典型表现：

- 页面上显示 `["humorous","logical"]`
- 字典组件不回显。
- 搜索和导出结果不可读。

这次 `Story 1.5` 的直接教训：

- `teacher_tags` 存的是 JSON 数组字符串。
- 管理端列表必须先解析，再按 `user_teacher_tag` 字典标签渲染。

防呆规则：

1. 只要字段注释里出现“JSON 数组字符串”，就不要直接拿原值渲染。
2. 前端写入前统一 `JSON.stringify`，读取后统一 `JSON.parse` + 容错。
3. 列表页、详情页、导出 VO 必须按“业务语义”展示，不按“存储格式”展示。

---

## 第六类坑：学生端流程被单个字段绑架

典型表现：

- 只要填了简介，就直接跳过后续偏好采集。
- 表里只有 `bio`，没有 `personality_type` / `teacher_tags`，导致后续任务透传缺数据。

这次 `Story 1.5` 的直接教训：

- “填写简介”只是资料补充，不等于“完成用户配置”。
- 首次进入时，点击“下一步”必须继续进入偏好采集。
- 只有显式点击“跳过”，或复访时本来就已有偏好数据，才允许直接导览。

防呆规则：

1. 页面跳转条件必须绑定“业务完成条件”，不要绑定单字段是否为空。
2. onboarding 相关 Story 必须把“首次进入”和“复访已有数据”拆开写清。
3. 只要表里还有关键字段未采集，就不能把这个流程视为完成。

---

## 第七类坑：上传问题以为是存储桶，其实是服务端 multipart 临时目录

典型表现：

- `MinIO` 页面能预览、能下载，但前端上传接口还是 500。
- 误以为是桶路径或对象不存在，实际是服务端还没把文件落到业务逻辑就失败了。

这次 `Story 1.5` 的直接教训：

- 头像上传 500 的根因不是 `MinIO` 桶，而是服务端 multipart 临时目录不可写。
- 最终通过新增 `MultipartTempDirConfig` 修复。

防呆规则：

1. 上传报 500 时，先区分“请求到达业务前失败”还是“对象存储写入失败”。
2. 先看服务端临时目录、磁盘权限和 multipart 配置，再看桶和对象路径。
3. 只要上传链路依赖本地临时目录，就要给出显式可写目录配置。

---

## 第八类坑：只改后端或只改前端，没有把整条链路走通

典型表现：

- 后端接口改了，管理端 typings 没更新。
- 管理端能查，学生端还按旧契约读写。
- Story 文档还是旧流程，后面的人继续按旧文档返工。

这次 `Story 1.5` 的直接教训：

- 这条链路至少涉及：
  - SQL
  - RuoYi 后端
  - Soybean 管理端
  - student-web 学生端
  - Story / docs 文档

防呆规则：

1. 改表字段或业务语义时，按下面顺序逐层检查：
   - 数据库脚本
   - 代码生成器配置
   - 后端 `Bo / Vo / Service / Controller`
   - 管理端 API typings
   - 管理端列表 / 搜索 / 编辑
   - 学生端 API 适配 / 页面流程
   - Story 与手册文档
2. 任何一层没同步，都不算真正收口。

---

## 推荐联调顺序

1. 先执行 SQL，确认表结构、字典和默认值。
2. 再导表到生成器，校准字段配置。
3. 生成后马上补手工调整项，不要等联调时再补。
4. 先跑后台管理端列表 / 搜索 / 编辑。
5. 再跑学生端 onboarding、上传和登录后回跳。
6. 最后回写 Story、开发手册和验收记录。

---

## 提交前自检清单

1. 数据库字段类型、默认值、字典值是否和代码一致。
2. `Bo / Vo / typings` 是否都同步。
3. JSON 字段是否做了解析与序列化。
4. URL 字段是否错误套用了 OSS 组件或翻译逻辑。
5. 搜索表单提交值类型是否和后端查询对象一致。
6. onboarding 流程是否仍按业务完成度而非单字段跳转。
7. Story 和 docs 是否同步更新。
