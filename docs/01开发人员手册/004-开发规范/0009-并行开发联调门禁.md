# 并行开发联调门禁

> **状态**: 已冻结
> **负责人**: 架构负责人 / SM / 模块 Owner
> **最后更新**: 2026-03-29

---

## 目标

- 明确 `mock 可开发`、`真实接口可联调`、`可合并`、`可发布` 不是同一层门禁。
- 允许前后端在契约稳定后并行推进，禁止再回到“前端等后端”或“后端等视觉稿”的口头依赖。
- 为 PR 审查、联调和发布提供统一判断标准。

## 六层门禁

| 门禁层级 | 进入条件 | 允许行为 | 禁止行为 | 退出条件 |
|------|------|------|------|------|
| 契约冻结 | 字段、错误码、状态枚举、示例 payload、数据宿主明确 | 前后端并行拆分任务；创建 adapter / mock | 未冻结 schema 就直接写真实实现 | 消费方可不依赖口头说明开始开发 |
| mock 可运行 | adapter 与 mock handler 已打通；关键状态可演示 | 前端页面、后端 schema、测试夹具并行推进 | 用静态假数据冒充完整闭环 | 页面或服务能演示核心状态流转 |
| 测试通过 | 单元 / 集成 / 文档检查通过 | 转入自审与 Draft PR 收口 | 未跑验证就宣称“基本没问题” | 自测证据可回写到 PR 或 Story |
| 联调前门禁 | 真实接口、权限、错误态、状态机与 mock 口径一致 | 前后端联调、修 seam、补适配层 | 联调时再改结构、再补 schema | 联调不再依赖猜字段或临时口头约定 |
| 合并前门禁 | AC 覆盖、测试证据、文档索引、审查意见闭环 | Ready for review、Squash merge | 带 Blocker / Major 问题直接合并 | PR 满足审查标准与 GitHub Flow 约束 |
| 发布前门禁 | P0 E2E、P0 Bug、性能、状态恢复与长期数据链路验证通过 | 打 Tag、发版、提交里程碑 | 把“已合并但未验证”当成可发布 | 具备上线 / 交付证据 |

## 并行开发基本规则

1. 前端可以基于 `adapter + mock handler` 先行，但前提是契约已冻结。
2. 后端可以先以 schema、示例、测试夹具交付，不要求等待正式页面完成。
3. 真实接口联调前，禁止在联调现场临时改字段名、状态值或错误码。
4. Draft PR 用于持续暴露进度，不用于隐藏大批量一次性改动。

## 常见禁止写法

1. “等后端接口好了前端再做”
2. “先把页面画出来，联调时再改结构”
3. “先不定义 schema，到时候按返回改”
4. “先做 happy path，错误态后面补”
5. “先放 Redis，后面再落库”
6. “先直接调厂商 SDK，后面再抽象 Provider”

## 合并前最小检查

1. 当前变更是否只承载一个 Story 或一个可独立回滚的子任务。
2. Story / Issue / PR / 测试证据是否彼此可追踪。
3. mock、真实接口、文档与测试是否还保持同一口径。
4. 是否存在“代码完成了，但文档和索引没人知道”的情况。

## 发布前最小检查

1. P0 E2E 场景通过。
2. 任务状态、进度、完成、失败、恢复路径验证通过。
3. 长期业务数据与文件归档路径验证通过。
4. 所有阻塞性问题关闭，不带未决高风险进入发布。
