# 代码审查标准

> **状态**: 已初始化
> **负责人**: 模块 Owner / Reviewer
> **最后更新**: 2026-03-27

---

## 审查流程

1. 开发者完成本地自检后创建 Draft PR。
2. 开发者先进行自审，确认 PR 描述、验收标准、自测证据完整。
3. 转为 Ready for review 后，由 Reviewer 进行正式审查。
4. 如存在阻塞问题，开发者修复后再次请求审查。
5. 所有阻塞问题关闭、测试证据充分后才允许合并。

### 触发条件

- 所有代码类 PR
- 配置、脚本、数据库脚本变更
- 影响接口契约、权限、认证、 Redis 、 SSE 的文档与代码变更
- 对 Story 验收标准有实质影响的 UI 改动

### Story 门禁引用

- Story Ready / Done 主规范：`0008-story-交付门禁与-dor-dod.md`
- 并行开发与联调收口门禁：`0009-并行开发联调门禁.md`
- 故事级执行清单：`../007-测试策略/0005-故事级验收清单.md`

## 审查清单

### 通用清单

1. 变更是否与当前 Story / Issue 强关联。
2. 改动是否足够小，是否易于回滚。
3. 命名、目录、分层是否符合现有模式。
4. 是否补齐测试、文档和必要中文注释。
5. 是否把逻辑验证页、原型页或线框态错误标记为“正式页面已完成”。
6. `packages/student-web/` 测试文件是否统一收敛到 `src/test/` 镜像目录。

### 架构清单

1. FastAPI 是否仍然只承担功能服务职责。
2. 长期业务数据是否错误写入 Redis 而非 RuoYi。
3. 标准业务 CRUD 是否错误落在 FastAPI 而非 RuoYi。
4. FastAPI 与 RuoYi 的交互是否通过防腐层，而非直接耦合。

### 接口与状态清单

1. API 响应是否保持 `{code, msg, data}` / `{code, msg, rows, total}`。
2. 状态码与错误提示是否一致，`401`、`403`、`500`、`601` 是否明确。
3. Redis Key 是否符合命名规范， TTL 是否明确。
4. SSE 事件类型与 Payload 是否符合架构约束。

### Story 1.1 专项清单

1. 注册与登录是否复用 RuoYi 认证能力，而不是重造一套账户体系。
2. 未授权状态是否能稳定回到可恢复的未登录状态。
3. 认证成功后是否回到原始页面上下文。
4. 是否避免出现无穷跳转、双重登录态或前后端判断不一致。

### Epic 1 前端页面专项清单

1. 当前 `HomePage`、`AuthDialog`、退出按钮等是否仍属于逻辑验证载体，而非被错误宣称为正式首页 / 正式认证页面。
2. Story 1.4 / Story 1.6 的正式页面待办、视觉收口与原型文案清理要求是否已在文档或 PR 描述中保留。
3. 业务文件是否按规范补齐中文注释，尤其是 auth seam、cleanup 分支、受保护访问与降级路径。
4. 测试是否统一进入 `packages/student-web/src/test/`，是否仍存在继续扩散的同层 `*.test.ts(x)`。
5. 当前页面中的 Story 说明文字、调试态、token 片段等原型痕迹是否被错误带入正式交付。

## 问题分级

### Blocker

必须修复，否则不得合并：

- 架构边界违规
- 安全缺陷
- 认证与权限错误
- 数据丢失、错误持久化
- Story 验收标准未满足
- 原型页 / 逻辑验证页被当作正式页面交付
- 缺少强制中文注释，导致关键业务边界不可维护

### Major

原则上应在合并前修复：

- 明显可维护性问题
- 缺少关键测试
- 异常处理与回退路径不足
- API 契约与文档不一致
- Story Ready / Done 门禁未满足，但被错误推进到 review
- 测试目录散乱，继续在业务目录同层新增 `*.test.ts(x)`
- 正式页面待办、原型清理范围或 Epic 1 执行边界未回写到文档

### Minor

可在本 PR 或后续 PR 修复：

- 命名不一致
- 注释不清晰
- 非关键样式与结构瑕疵

## 审查标准

1. **正确性**：逻辑与验收标准一致，没有明显回归。
2. **边界性**：实现没有越过 FastAPI / RuoYi / Redis / Provider 的职责边界。
3. **可维护性**：结构清晰，命名一致，没有隐藏副作用。
4. **安全性**：认证、授权、敏感信息处理和错误暴露符合要求。
5. **可验证性**：具备自测步骤、日志或测试证明。

## 最佳实践

1. 审查意见聚焦行为风险，不做纯风格争论。
2. 评论必须尽量给出文件、上下文和期望修正方向。
3. 若发现 Story 本身不清楚，应先阻塞需求，而不是让实现阶段硬猜。
4. 文档 PR 也要检查是否更新了相关索引与引用。
5. 审查通过不代表可以跳过测试；测试证据依然是合并前置条件。
6. 若 PR 涉及 Story 门禁、联调门禁或发布门禁，必须显式核对 `0008`、`0009` 与 `0005` 三份文档。
