## GitHub Flow 对接方式

实施阶段默认与 GitHub Flow 绑定执行：

1. `Create Story` 或现有 Story 文档确认后，先创建对应 GitHub Issue。
2. 基于 Issue / Story 拉出短分支，例如 `feature/story-1-1-auth-entry`。
3. `Dev Story` 阶段通过 Draft PR 持续暴露实现进度。
4. `Code Review` 阶段以 GitHub PR 为载体执行，审查结论回写到 PR。
5. 审查通过后以 `Squash and merge` 合回 `master`。

### Story 进入开发前的最小前置条件

1. PRD、架构、Epic / Story 已完成并相互对齐。
2. `004-开发规范` 与 `005-环境搭建` 已初始化。
3. GitHub 仓库已按 `0002-Git工作流.md` 启用受保护分支与 PR 流程。
4. 对于 `story 1-1`，已明确认证由 RuoYi 承载，业务前端承载路径采用架构默认方案。

## WDS 模块 — Wikki 设计系统

> **WDS 与 BMM 的关系**：WDS 和 BMM 是互补的并行轨道，可以同时运行。
> - **WDS** 专注于"设计什么"——用户体验、交互流程、视觉规格
> - **BMM** 专注于"如何实现"——技术架构、开发任务、测试

## 并行执行指南：WDS 与 BMM 协同工作

### 核心区别

| 维度 | BMM (主工作流) | WDS 模块 |
|------|---------------|----------|
| **定位** | 产品开发全生命周期管理 | UX/UI 设计专项 |
| **核心产物** | PRD、架构文档、Epics/Stories、Sprint 计划 | Project Brief、Trigger Map、UX Scenarios、Page Specs |
| **输出目录** | `planning_artifacts` / `implementation_artifacts` | `design_artifacts` |
| **主导代理** | Mary (BA)、John (PM)、Winston (Architect)、Amelia (Dev) | Saga (Strategic Analyst)、Freya (UX Designer) |

### 并行执行模型

```

│                        项目启动                                      │
└─────────────────────────────────────────────────────────────────────┘
                                 │
            ┌────────────────────┴────────────────────┐
            ▼                                         ▼
    ┌───────────────┐                        ┌───────────────┐
    │   WDS 轨道    │                        │   BMM 轨道    │
    ├───────────────┤                        ├───────────────┤
    │ Phase 0-1     │                        │ Phase 1-2     │
    │ Project Brief │                        │ PRD + Arch    │
    │ Trigger Map   │                        │               │
    └───────┬───────┘                        └───────┬───────┘
            │                                         │
            ▼                                         │

            ▼                                 │ Phase 3-4     │

    │ Design        │────────────────────────│ Story Cycle   │
    │ Delivery      │                        │ Implementation│
```

### 推荐的并行启动顺序

1. **同时启动**：
   - WDS: 唤醒 Saga → 开始 Project Brief / Trigger Mapping
   - BMM: 开始 Market Research / Technical Research（如果需要）

2. **设计完成后**：
   - WDS 产出 `design_artifacts/E-PRD/Design-Deliveries/`
   - BMM 的 Sprint Planning 基于设计交付物进行开发任务分解

3. **持续迭代**：
   - WDS Phase 3 (Agentic Development) 可以与 BMM Story Cycle 并行
   - 每轮设计迭代后可触发新的开发冲刺

### 共享资源

两个模块共享以下资源：
- **`project_knowledge`**：指向 `docs/` 目录的项目知识
- **`output_folder`**：都输出到 `_bmad-output/` 目录
- **Stakeholder Context**：利益相关者信息对两个模块都可用
