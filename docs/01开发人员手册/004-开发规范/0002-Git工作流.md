## 工作流选择

本仓库采用 **GitHub Flow**，并沿用当前仓库默认主线分支 `master` 作为唯一长期分支。

- 远程仓库：`origin = https://github.com/Prorise-cool/Prorise_ai_teach_workspace`
- 主线分支：`master`
- 合并策略：默认 `Squash and merge`
- 分支生命周期：短分支、短周期、单目标

## 分支策略

1. 所有开发分支都从最新的 `origin/master` 拉出。
2. 禁止长期存在 `develop`、`release` 之类的平行主线分支。
3. 一个分支只承载一项明确目标，通常对应一个 Story、一个缺陷或一类纯文档改动。

### 分支命名

推荐格式：

- `feature/story-1-1-auth-entry`
- `feature/story-1-3-auth-consistency`
- `fix/auth-401-loop`
- `docs/init-dev-standards`
- `chore/setup-fastapi-shell`

如已建立 GitHub Issue，可在前缀中带编号：

- `feature/123-story-1-1-auth-entry`

## 提交规范

提交信息采用中文 Conventional Commits 风格：

```text
类型: 简短描述
```

允许的类型：

- `feat`
- `fix`
- `refactor`
- `docs`
- `test`
- `chore`

示例：

- `feat: 完成 story 1-1 登录弹窗骨架`
- `fix: 修复认证态失效后重复跳转`
- `docs: 初始化 GitHub Flow 与环境搭建手册`

## 合并流程

### 标准节奏

1. 从 Story / Issue 建立分支。
2. 本地完成最小可运行改动后立即推送远端，并创建 Draft PR。
3. 在 Draft PR 中持续更新实现、测试证据与风险说明。
4. 满足合并门禁后，转为 Ready for review。
5. 至少获得 `1` 次有效审批后，使用 `Squash and merge` 合并回 `master`。
6. 合并后删除功能分支。

### PR 门禁

1. 必须关联 Story、 Epic 或 Issue。
2. 必须填写 PR 模板中的验收、自测和风险说明。
3. 必须通过本地自检与必要测试。
4. 必须解决全部阻塞性评审意见。
5. 单个 PR 默认只交付一个 Story 或一个可独立回滚的子任务。

### 标签规则

PR 至少要有一个有效标签，当前仓库统一采用以下标签体系：

- `new-feature`
- `bugfix`
- `documentation`
- `refactor`
- `performance`
- `maintenance`
- `ci`
- `dependencies`
- `breaking-change`

## 版本管理

1. 日常 Story 开发直接通过 GitHub Flow 合并到 `master`。
2. 需要里程碑发布时，基于 `master` 打 Tag。
3. `release-drafter` 使用 PR 标签生成版本说明，因此 PR 标签必须准确。
4. 非里程碑阶段不额外维护发布分支。

## GitHub 仓库初始化要求

### 已在仓库内落地

1. `.github/PULL_REQUEST_TEMPLATE.md`
2. `.github/ISSUE_TEMPLATE/feature_request.md`
3. `.github/ISSUE_TEMPLATE/ISSUE_TEMPLATE.md`
4. `.github/ISSUE_TEMPLATE/story-task.md`
5. `.github/workflows/auto-assign.yml`
6. `.github/workflows/pr-labels.yml`
7. `.github/release-drafter.yml`

### 需要在 GitHub 仓库设置页启用

1. 将 `master` 设置为受保护分支。
2. 禁止直接 push 到 `master`。
3. 开启 `Require a pull request before merging`。
4. 至少要求 `1` 位 reviewer 审批。
5. 开启 `Require branches to be up to date before merging`。
6. 开启 `Automatically delete head branches`。
7. 若团队成员明确后，再补 `CODEOWNERS` 与自动 reviewer。

## 冲突解决

1. 提交前先同步主线：

```bash
git fetch origin
git rebase origin/master
```

2. 如果发现工作树中存在与你当前任务无关的未提交改动，停止覆盖，先确认来源。
3. 冲突优先在本地解决，不在 GitHub Web Editor 上做结构性冲突处理。
4. 解决冲突后必须重新跑受影响范围测试。
5. 多 CLI / 多代理并行时，只提交当前任务直接产出的文件。

## Story 1.1 启动方式

`story 1-1` 默认按照下面的 GitHub Flow 路径推进：

1. 建立 `story-task` Issue，记录验收标准与边界。
2. 从 `master` 拉出 `feature/story-1-1-auth-entry`。
3. 先交付业务前端壳层与认证入口最小闭环，再补双后端一致性衔接。
4. 通过 Draft PR 持续暴露实现进度，避免大批量一次性合并。
