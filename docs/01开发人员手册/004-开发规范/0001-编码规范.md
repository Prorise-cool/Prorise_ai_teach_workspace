## 文档目标

本文档定义 `story` 开发阶段必须遵守的最小编码基线，用于把 PRD、架构文档、 Epic / Story 与代码实现对齐。

## 通用规范

1. 先遵循架构，再写代码。所有实现必须服从 `_bmad-output/planning-artifacts/architecture/` 中已经确认的 `[Decision]` 与 `[Rule]`。
2. 每次提交保持小而可验证。默认一条 Story 对应一个功能分支和一个 PR。
3. 函数、类、模块保持单一职责，避免跨层堆逻辑。
4. 优先复用现有模式，禁止为一次性需求引入额外抽象。
5. 任何跨模块依赖都必须先确认边界归属，不能直接绕过防腐层。
6. 新增或修改公共行为时，必须同步补测试与文档。

## Python 编码规范

1. 文件名使用 `snake_case.py`，类名使用 `PascalCase`，函数与变量使用 `snake_case`。
2. FastAPI 仅承担功能服务职责，不定义长期业务实体，不提供完整业务 CRUD。
3. 所有 FastAPI 路由统一挂在 `/api/v1` 下，长耗时能力统一抽象为 `tasks` 资源。
4. 接口响应格式必须与 RuoYi 对齐：

```json
{
  "code": 200,
  "msg": "success",
  "data": {}
}
```

5. Redis 只承载运行时状态、认证态与事件缓存，所有运行态 Key 必须显式设置 TTL。
6. Provider、任务编排、外部调用逻辑必须通过明确的接口或适配层组织，不能把第三方 SDK 直接散落到业务逻辑里。
7. 异常处理统一转成可恢复、可观测的业务错误，不向前端暴露原始堆栈。

## TypeScript 前端规范

### React 业务前台

1. 学生端业务前台按架构目标落在 `packages/student-web/`，技术栈为 React + Vite + TypeScript。
2. 文件名使用 `kebab-case.ts(x)`，组件名使用 `PascalCase`，Hook 使用 `useXxx`。
3. 页面负责组织流程，复杂逻辑下沉到 `services`、`hooks`、`store` 或领域模块，不在组件里堆砌业务判断。
4. 所有前端接口层必须统一处理 `401`、`403`、`500`、`601` 等响应，不允许页面各自散写错误分支。
5. SSE 消费必须遵循架构文档中的事件类型：`connected`、`progress`、`provider_switch`、`completed`、`failed`、`heartbeat`、`snapshot`。

### Vue / Soybean 管理前端

1. 当前仓库已有的管理前端基线位于 `packages/ruoyi-plus-soybean-master/`，其内部继续遵循 Vue 3 Composition API、TypeScript、Pinia、Vite 的既有结构。
2. 对该项目的修改优先保持其现有目录模式和脚本习惯，不把小麦业务逻辑直接混入通用基础设施目录。
3. 若需要承载小麦业务管理页面，应新增业务模块或页面，而不是修改其认证与路由底层框架。

## Java 编码规范

1. Java 侧以 `packages/RuoYi-Vue-Plus-5.X/` 为认证、RBAC 与长期业务数据主宿主。
2. 禁止修改 RuoYi 核心认证流程、基础 RBAC 框架和底层公共机制；允许新增小麦业务表、业务模块、菜单和权限标识。
3. 权限标识统一使用 `模块:资源:操作`，例如 `video:task:add`、`learning:record:list`。
4. 所有长期业务数据必须落在 RuoYi 业务表，不能只存 Redis。
5. Controller、Service、Mapper 命名遵循 RuoYi 既有模式，避免引入第二套风格。

## 命名约定

| 类别 | 后端 | 前端 |
|------|------|------|
| 文件 | `snake_case.py` / `PascalCase.java` | `kebab-case.ts(x)` / `PascalCase.vue` |
| 类 / 组件 | `PascalCase` | `PascalCase` |
| 函数 / 变量 | `snake_case` / `camelCase`（Java） | `camelCase` |
| 常量 | `UPPER_SNAKE_CASE` | `UPPER_SNAKE_CASE` |
| 权限标识 | `module:resource:action` | 同后端文档约定 |

## 架构硬约束

1. FastAPI 与 RuoYi 的协作必须经过防腐层，不能直接耦合对方领域模型。
2. 标准业务 CRUD 优先由 RuoYi 承接，FastAPI 不重复建设第二套后台。
3. 日期格式统一为 `yyyy-MM-dd HH:mm:ss`。
4. 分页参数与响应统一使用 `pageNum`、`pageSize`、`rows`、`total`。
5. Redis Key 使用架构约定前缀，例如 `xm_task:{task_id}`、`xm_task_events:{task_id}`、`online_tokens:{tokenValue}`。

## 注释规范

1. 注释只解释“为什么这样做”或“边界在哪里”，不复述显而易见的代码行为。
2. 跨系统边界、降级策略、兼容性处理、第三方能力切换处必须补简短说明。
3. 公共模块和关键入口保留必要文档字符串，说明输入、输出、异常与副作用。
4. TODO 必须写明负责人或触发条件，避免留下无上下文占位。

## 质量门禁

1. 提交前至少完成受影响范围的格式化、Lint、类型检查和测试。
2. 公开函数、 Hook、服务边界变更必须补测试。
3. 任何违反架构边界、接口契约或 Redis TTL 约束的实现，视为阻塞缺陷，不得合并。
