# 长期业务数据边界与表清单基线

> **状态**: 已冻结
> **负责人**: Epic 10 Owner / 后端架构 Owner
> **最后更新**: 2026-03-29

---

## 文档目标

本文档用于冻结 Epic 10 的长期业务数据宿主边界、共享业务实体、最小字段基线与变更门禁。

使用规则：

1. `_bmad-output/planning-artifacts/prd/` 与 `architecture/` 仍是事实来源。
2. 本文档是 Epic 10 在代码、SQL、CRUD、查询与聚合实现前的执行基线。
3. 后续 Story 10.2 ~ 10.8 只能在本基线之上扩展，不允许绕过基线口头补字段。

## 宿主边界矩阵

| 数据域 | 运行态宿主 | 长期宿主 | 文件宿主 | 查询 / 审计边界 | 禁止事项 |
|------|------|------|------|------|------|
| 认证在线态 | Redis | RuoYi 用户 / RBAC | - | 仅认证与会话有效期 | 禁止把长期学习记录塞入在线态 |
| 任务状态 / SSE 事件 | Redis | - | - | 仅断线恢复与运行观测 | 禁止作为 `/learning`、`/history`、后台审计的数据源 |
| 视频任务元数据 | Redis 临时态 | `xm_video_task` | COS 仅保存对象 | 学习中心回看、后台查询、导出、审计 | 禁止只保留 Redis 快照 |
| 课堂会话摘要 | Redis 临时态 | `xm_classroom_session` | COS 仅保存对象 | 学习中心回看、后台查询、导出、审计 | 禁止只依赖 completion signal 缓存 |
| SessionArtifactGraph 索引 | Redis 临时态 | `xm_session_artifact` | COS 产物文件 | Companion 消费、学习中心回看、后台审计 | 禁止让 Companion 直接依赖内容引擎内部对象 |
| Companion turn | Redis 上下文窗口 | `xm_companion_turn` | COS 仅保存白板产物引用 | 学习中心回看、后台查询、审计 | 禁止只靠运行态窗口恢复历史问答 |
| Whiteboard action | Redis 临时渲染态 | `xm_whiteboard_action_log` | COS 渲染产物 | 回看白板动作、审计降级路径 | 禁止把动作细节压成不可恢复的自由文本 |
| Evidence / Retrieval 问答 | Redis 临时检索态 | `xm_knowledge_chat_log` | COS 附件引用 | 学习中心回看、后台审计 | 禁止重新执行检索才能恢复主要内容 |
| Learning Coach 结果 | Redis 临时工作流态 | `xm_quiz_result`、`xm_learning_path` | COS 仅保存附件对象 | 学习中心聚合、历史回看、后台查询 | 禁止把 wrongbook / recommendation 混入普通运行日志 |
| 学习记录 / 收藏 | - | `xm_learning_record`、`xm_learning_favorite` | - | `/learning`、`/history`、`/favorites`、后台查询 | 禁止前端本地状态与长期表分叉 |

## Redis 运行态约束

| Key 前缀 | 用途 | TTL | 长期替代宿主 |
|------|------|------|------|
| `xm_task:{task_id}` | 异步任务状态 | 2h | `xm_video_task` / `xm_classroom_session` |
| `xm_task_events:{task_id}` | SSE 事件缓存 | 1h | 无，事件不作为长期历史 |
| `xm_video_runtime:{task_id}` | 视频运行态 | 2h | `xm_video_task`、`xm_session_artifact` |
| `xm_classroom_runtime:{session_id}` | 课堂运行态 | 会话结束后 1h | `xm_classroom_session`、`xm_session_artifact` |
| `xm_companion_runtime:{session_id}` | Companion 上下文窗口 | 30m | `xm_companion_turn` |
| `xm_context_window:{session_id}:{anchor}` | 锚点窗口 | 30m | `xm_companion_turn` / `xm_knowledge_chat_log` |
| `xm_provider_health:{provider}` | Provider 健康检查 | 60s | 无 |

硬规则：

1. Redis 中不得出现学习记录、收藏、任务元数据、问答历史、quiz 结果、path 结果的唯一长期真值。
2. 任何需要在 `/learning`、`/history`、`/favorites` 或后台查询 / 导出 / 审计中可见的数据，都必须进入 RuoYi / MySQL。
3. 大对象只保存 COS 引用、摘要或结构化索引，不允许直接把大体积结果正文塞进主表。

## 公共字段基线

### 公共审计字段

所有长期业务表至少具备以下公共字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | `bigint` | 主键 |
| `tenant_id` | `varchar(20)` | 租户编号，默认 `000000` |
| `create_dept` | `bigint` | 创建部门 |
| `create_by` | `bigint` | 创建者 |
| `create_time` | `datetime` | 创建时间 |
| `update_by` | `bigint` | 更新者 |
| `update_time` | `datetime` | 更新时间 |
| `del_flag` | `char(1)` | 逻辑删除标志，`0` 为存在，`1` 为删除 |

### 统一语义字段

| 字段 | 说明 |
|------|------|
| `user_id` | 业务归属用户 |
| `status` | 统一状态字段，任务类取值遵循 `pending / processing / completed / failed / cancelled` |
| `summary` / `answer_summary` / `analysis_summary` | 可供学习中心与后台直接展示的摘要 |
| `source_session_id` | 来源视频 / 课堂 / 会话标识 |
| `source_result_id` | 衍生结果回指来源结果 |
| `detail_ref_type` + `detail_ref_id` | 详情定位字段，不允许只靠模糊摘要拼装 |
| `object_key` / `render_object_key` | COS 对象引用 |
| `occurred_at` / `started_at` / `completed_at` / `failed_at` | 关键业务时间字段 |

### 日期与分页规范

1. 日期格式统一为 `yyyy-MM-dd HH:mm:ss`。
2. 分页请求统一使用 `pageNum`、`pageSize`。
3. 分页响应统一使用 `{code, msg, rows, total}`。

|------|------|
| `result_id` | 结果业务 ID，唯一 |
| `user_id` | 用户归属 |
| `result_type` | `checkpoint` / `quiz` / `wrongbook` / `recommendation` |
| `source_session_type` / `source_session_id` | 来源视频 / 课堂 / 学习会话 |

wrongbook / recommendation 规则：

1. 使用 `result_type` 明确区分，不允许混入通用历史。
2. 使用 `source_result_id` 回指来源的 checkpoint / quiz / recommendation 上游结果。
3. 详情通过 `detail_payload_json` 或 `detail_ref_type/detail_ref_id` 结构化挂接，不允许只写自由文本。

## 字段变更门禁

1. 任何新增字段必须同时更新：
   - 对应 SQL 基线；
   - 本文档字段矩阵；
   - 受影响 Story 的 DTO / mapper / VO；
   - 测试样例与 API 文档。
2. 禁止业务 Epic 私自扩表或只在代码里临时拼接新字段。
3. `result_type`、`record_type`、`artifact_type`、`turn_status` 等枚举变更必须同步数据字典。
4. 日期字段不得混用 ISO 8601 与 `yyyy-MM-dd HH:mm:ss`。
5. `detail_ref_type` / `detail_ref_id` 是重开入口的唯一准确信息，不得回退为“靠摘要猜详情”。
6. FastAPI 不得自建第二套长期数据库或平行业务表。

## 执行约束

1. `10.1` 与 `10.2` 完成前，`10.4 ~ 10.8` 不得绕过本基线新建临时持久化结构。
2. `10.3` 的 FastAPI 防腐层必须以本表结构和枚举基线为 DTO 设计出发点。
3. 后续真实 API 联调、后台导出与学习中心聚合，统一以这些长期表为准，不允许回放 Redis 过程数据拼装历史。
