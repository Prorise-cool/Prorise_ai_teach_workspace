# 代码结构与文件拆分规范

> **状态**: 已生效
> **来源**: fastapi-backend + student-web 代码结构审查报告 (2026-04-09)
> **最后更新**: 2026-04-10

---

## 1. 目的

本规范定义文件行数上限、模块职责边界和拆分触发条件，**防止文件无序膨胀导致后续大规模重构**。

所有开发人员在提交代码前必须对照本规范检查新增和修改的文件。

---

## 2. 行数硬限制

### 2.1 后端 (Python / FastAPI)

| 文件类型 | 硬限制 | 建议值 | 说明 |
|----------|--------|--------|------|
| Service / Pipeline 类 | **500 行** | < 300 行 | 超限必须拆分为子模块包 |
| API Route 文件 | **300 行** | < 200 行 | 路由只做参数校验和转发 |
| Schema / Model 文件 | **400 行** | < 300 行 | 数据模型可适当宽松 |
| 工具 / 辅助文件 | **300 行** | < 200 行 | 超过应按功能拆分 |
| `__init__.py` | **150 行** | < 50 行 | 仅做重导出，不包含业务逻辑 |

### 2.2 前端 (TypeScript / React)

| 文件类型 | 硬限制 | 建议值 | 说明 |
|----------|--------|--------|------|
| React 组件 | **300 行** | < 200 行 | 超限必须提取 hook 或子组件 |
| Service / Adapter | **300 行** | < 200 行 | 超限必须拆分为子模块包 |
| Hook 文件 | **200 行** | < 150 行 | 超限按关注点拆分为多个 hook |
| Store 文件 | **200 行** | < 150 行 | 超限按状态域拆分 |
| 工具 / 辅助文件 | **200 行** | < 100 行 | 超限按功能拆分 |
| barrel `index.ts` | **100 行** | < 50 行 | 仅做重导出 + 类型定义 |
| i18n 数据 / Mock 数据 | 豁免 | — | 纯数据文件不适用行数限制 |

---

## 3. 拆分触发条件

**满足以下任一条件时，必须拆分：**

1. 文件行数超过对应硬限制
2. 单个类/组件承担 **3 种以上** 职责
3. 单个文件包含 **10 个以上** 公开函数/方法
4. 新增功能导致文件单次增长超过 **100 行**

---

## 4. 拆分模式

### 4.1 Python — Mixin 继承模式（推荐）

当单个类方法过多时，按职责拆分为 Mixin：

```
# 拆分前: service.py (600行)
class VideoPipelineService:
    def run(self): ...
    def _run_stage_a(self): ...
    def _run_stage_b(self): ...
    def _write_result(self): ...
    def _handle_error(self): ...

# 拆分后: service/ 包
service/
├── __init__.py           # class VideoPipelineService(StageMixin, ResultMixin, ErrorMixin): pass
├── coordinator.py        # 核心编排: run()、__init__
├── stage_runners.py      # 阶段执行: _run_* 方法
├── result_persister.py   # 结果写入: _write_* 方法
└── failure_handler.py    # 错误处理: _handle_* 方法
```

**规则：**
- 每个 Mixin 文件不超过 300 行
- `__init__.py` 仅做类组装和重导出（< 150 行）
- 所有外部 import 路径保持不变（向后兼容）

### 4.2 TypeScript — 子模块包模式

当服务文件过大时，拆分为子目录 + barrel export：

```
# 拆分前: services/sse/index.ts (1100行)

# 拆分后:
services/sse/
├── index.ts          # barrel file (< 100行)
├── parsers.ts        # 解析逻辑
├── mock-stream.ts    # Mock 流
├── real-stream.ts    # 真实流
├── type-guards.ts    # 类型守卫
└── utils.ts          # 工具函数
```

### 4.3 React 组件 — Hook 提取模式

```
# 拆分前: landing-contact.tsx (480行)

# 拆分后:
landing-contact/
├── index.tsx                    # 主组件 (< 100行)
├── use-landing-contact-submit.ts # 表单状态 + 提交 hook
├── contact-form-fields.tsx      # 表单字段子组件
└── landing-contact-helpers.ts   # i18n + 工具函数
```

## 5. Feature 模块标准化

### 5.1 后端 Feature 标准结构

```
features/{feature-name}/
├── __init__.py        # Feature 公共导出
├── routes.py          # API 路由（仅参数校验和转发）
├── schemas.py         # 请求/响应 Schema
├── service.py         # 业务逻辑（或 service/ 包）
├── models.py          # 数据模型（如需要）
└── {sub-module}/      # 子模块（如 pipeline/）
```

**规则：**
- Feature 间**禁止直接导入**，必须通过 `shared/` 层通信
- 新 Feature 必须包含 `routes.py` + `schemas.py` + `service.py`

### 5.2 前端 Feature 标准结构

```
features/{feature-name}/
├── index.ts           # Barrel export（必须）
├── api/               # API 调用层
├── components/        # Feature 组件
├── hooks/             # 自定义 Hooks
├── pages/             # 页面组件
├── schemas/           # 表单 Schema / 验证
├── stores/            # Zustand Store（如需要）
├── styles/            # SCSS 样式
└── shared/            # Feature 内共享（如需要）
```

**规则：**
- 每个 Feature **必须有 `index.ts`** barrel export
- Feature 间**禁止直接导入**，共享逻辑必须提取到 `shared/` 或 `services/`
- 缺失目录用 `.gitkeep` 占位，保持结构一致性

---

## 6. 依赖方向（强制）

```
后端:  routes → service → pipeline/shared → core
前端:  pages → components → hooks → services/api → stores
       ↑ 所有层级都可以引用 shared/、lib/、types/
       ✗ Feature A 禁止直接 import Feature B 的内部模块
```

**违反依赖方向 = 阻塞缺陷，不得合并。**

---

## 7. 质量门禁

### 7.1 PR 检查清单

- [ ] 新增/修改文件未超过行数硬限制
- [ ] 新增文件符合 Feature 标准结构
- [ ] 无跨 Feature 直接导入
- [ ] `__init__.py` / `index.ts` 仅做重导出
- [ ] 拆分后的子模块各自 < 建议行数值

### 7.2 自动化建议

后续可考虑：
- CI 添加 `xargs wc -l` 检查文件行数超限
- ESLint/PyLint 规则检测文件复杂度
- PR template 内嵌本规范检查项
