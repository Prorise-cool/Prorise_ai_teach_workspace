# 前后端测试分层与目录规范

> 状态：生效
> 适用范围：`packages/student-web/`、`packages/fastapi-backend/`
> 最后更新：2026-04-08

---

## 目标

统一 student-web 与 fastapi-backend 的测试目录职责、执行入口与复用方式，避免继续出现：

1. 业务测试集中堆在单一 `src/test/`。
2. pytest 目录只有 `unit / integration` 粗分层，缺少域边界。
3. 同一类测试重复拼装 Provider、Router、MSW 生命周期。

---

## student-web 规范

### 目录职责

- `src/test/`：只保留 `setup.ts` 与跨域复用测试工具。
- `src/test/browser/`：只保留 Browser Mode 专属 setup 与完整应用挂载工具。
- `src/features/**`：页面流、路由流、业务 hook 测试与对应 feature 共置。
- `src/services/**`：adapter、runtime、SSE、mock handler / fixture 测试与基础设施共置。
- `src/stores/**`：状态持久化与 store 测试共置。
- `src/shared/**`：跨域反馈、上下文、共享 hook 测试共置。
- `src/styles/**`：设计令牌与样式契约测试共置。
- `src/**/*.browser.test.tsx`：浏览器级业务流测试，与对应页面或 feature 共置，不混入普通 jsdom 单测入口。

### 统一工具

- `src/test/utils/render-app.tsx`
  - `renderWithApp()`：统一装配 `AppProvider`
  - `renderRouterWithApp()`：统一装配 `AppProvider + MemoryRouter`
- `src/test/browser/render-app.tsx`
  - `renderBrowserApp()`：统一装配 `AppProvider + BrowserRouter + MSW worker`
  - `cleanupBrowserApp()`：统一卸载真实浏览器挂载点
- `src/test/browser/setup.ts`
  - 统一复用 `src/test/setup.ts`
  - 统一清理浏览器挂载内容、storage、theme、auth/profile store
- `src/test/utils/session.ts`
  - `resetAppTestState()`：统一清理 i18n、storage、theme、auth/profile store
  - `seedMockAuthSession()`：统一生成 mock 登录态
  - `seedCompletedUserProfile()`：统一生成已完成 onboarding 的用户资料
- `src/test/utils/msw-server.ts`
  - `registerMswServer()`：统一注册 `listen / resetHandlers / close`

### 编写规则

1. 新增页面测试默认共置到对应 `features/<feature>/pages/*.test.tsx`。
2. 新增 hook 测试默认共置到对应 `features/<feature>/hooks/*.test.ts(x)`。
3. 涉及 `AppProvider`、`RouterProvider` 的测试，优先复用 `renderWithApp` / `renderRouterWithApp`。
4. 涉及登录态、`returnTo`、persist、本地缓存的测试，优先复用 `resetAppTestState` 与 `seedMockAuthSession`。
5. 涉及 MSW handler 的测试，不再手写 `setupServer(...); beforeAll/afterEach/afterAll` 三件套，统一走 `registerMswServer`。
6. 浏览器级测试默认使用 `Vitest Browser Mode + @vitest/browser-playwright`，不再额外引入独立老牌 E2E runner 作为主测试框架。
7. 浏览器级测试必须放在 `*.browser.test.tsx`，并通过独立入口 `vitest.browser.config.ts` 执行，避免与 jsdom 单测互相污染。
8. 浏览器级测试优先覆盖认证守卫、登录回跳、受保护页面导航、输入提交到生成页这类真实路由链路；断言以 `pathname/search` 与可见 UI 结果为主。

## fastapi-backend 规范

### 目录职责

- `tests/api/`
  - 对外 HTTP 路由行为、鉴权入口、SSE / status 查询等 API 面验证。
- `tests/contracts/`
  - OpenAPI、schema、共享契约资产存在性与字段冻结验证。
- `tests/integration/`
  - 按域划分的真实集成链路测试，例如 `video / tasks / learning / companion / ruoyi`。
- `tests/unit/`
  - 按域划分的纯单测，例如 `core / shared / providers / video / learning / task_framework / assets`。

### 包边界

为避免 pytest 因重名模块收集冲突：

- 所有新增二级测试目录必须补 `__init__.py`。
- 若同名测试文件需要同时存在于 `unit` 与 `integration`，必须保证包路径唯一。

### 执行入口

- `pnpm test:fastapi-backend`
- `pnpm test:fastapi-backend:api`
- `pnpm test:fastapi-backend:integration`
- `pnpm test:fastapi-backend:unit`

## 落地说明

本次调整后，student-web 的测试主体已从集中式 `src/test/` 迁出；`src/test/` 退化为测试基础设施层。fastapi-backend 的测试则从“粗粒度平铺”收敛为“分层 + 分域”结构，便于后续按套件执行、并行排查与定向补测。
