> **最后更新**: 2026-04-17  
> **适用范围**: `packages/fastapi-backend/app/features/video/pipeline/`

## 这份文档解决什么问题

当视频管线出现下面这类错误时，后续维护者很容易走错方向：

- `TypeError: Mobject.next_to() got an unexpected keyword argument 'buf'`
- render 明明已经失败，但最终 surfaced 出来的却是 Redis / await 相关异常

这份文档只做一件事：把“应该在哪一层修、绝对不该在哪一层修”说清楚。

## 固定边界

### 1. `buf` 不是运行时兼容层问题

如果模型产出了 `buf`，这属于：

- prompt / codebook 约束不够硬
- patch retry 修复规则没有把正确 API 再强调一遍

不属于 `engine/agent.py` 运行时 AST 改写职责。

### 2. `TeachingVideoAgent` 的职责

`packages/fastapi-backend/app/features/video/pipeline/engine/agent.py`

这里允许做的事：

- 发起 render
- 收集 stderr / stdout / current code
- 调用 patch retry
- 在最终失败时把上下文透传给 orchestrator

这里不允许再做的事：

- 根据某个错误字符串偷偷改生成代码
- 把 `buf -> buff` 之类的关键字替换塞进 render retry 里

### 3. 正确修复层

如果再遇到 `buf` 类错误，先看这几个文件：

- `packages/fastapi-backend/app/features/video/pipeline/prompts/manimcat/api_codebook.py`
- `packages/fastapi-backend/app/features/video/pipeline/engine/agent.py` 里的 `_request_patch_repair()`
- `packages/fastapi-backend/app/features/video/pipeline/prompts/manimcat/code_retry_system.md`

原因很简单：

- 初次 codegen 用的是 `api_codebook + shared specification`
- patch retry 也会再次注入同一套规则

所以正确修法是把规则写对，而不是等 render 崩了再在代理层偷改代码。

## 失败链路注意点

### 1. Failure store 只能“记录失败”，不能“制造失败”

`packages/fastapi-backend/app/features/video/pipeline/engine/render_failure.py`

当前约束：

- 同步 Redis client 不要 `await`
- 异步 Redis-like client 才按 awaitable 处理
- Redis 写失败只能 warning，不能覆盖原始 render error

### 2. Sanitizer 要拿全 stderr/stdout/code

`packages/fastapi-backend/app/features/video/pipeline/orchestration/orchestrator.py`

如果只把 `str(render_error)` 丢给 sanitizer，后续看到的 code snippet 会很弱，排查效率很差。当前必须优先传：

- `render_error.stderr`
- `render_error.stdout`
- `render_error.code`

## 最短排查顺序

1. 看 worker 日志里真正的 Manim 原始报错是不是 `buf`
2. 看 `api_codebook.py` 是否还在输出 `buff` 规则
3. 看 patch retry prompt 是否仍然注入 `apiIndexModule + SHARED_SPECIFICATION`
4. 看 failure store 是否又把主错误遮蔽掉

## 当前结论

2026-04-17 之后，这条链路的约束已经定了：

- `buf` 问题归 prompt / codebook / patch retry
- `agent.py` 不再做运行时关键字硬修
- failure store 不再掩盖主错误
