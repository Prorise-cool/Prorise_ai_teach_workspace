## Story 3.1 验收项：契约与 mock 基线

### 1. 契约资产齐备

操作：

- 检查以下文件存在且能打开：
  - `contracts/video/v1/create-task-request.schema.json`
  - `contracts/video/v1/create-task-response.schema.json`
  - `contracts/video/v1/create-task-errors.md`
  - `contracts/video/v1/preprocess-request.schema.json`
  - `contracts/video/v1/preprocess-response.schema.json`
  - `contracts/video/v1/CHANGELOG.md`

期望：

- `POST /api/v1/video/tasks` 的请求字段明确区分 `text` 与 `image` 两条路径。
- 成功响应码为 `202`，返回字段包含 `taskId`、`taskType`、`status`、`createdAt`。
- 错误响应仍遵循统一 `{ code, msg, data }` 结构。

### 2. mock 样例齐备

操作：

- 检查以下 mock 文件存在：
  - `mocks/video/v1/create-task.text-success.json`
  - `mocks/video/v1/create-task.image-success.json`
  - `mocks/video/v1/create-task.validation-error.json`
  - `mocks/video/v1/create-task.permission-denied.json`
  - `mocks/video/v1/preprocess.success.json`
  - `mocks/video/v1/preprocess.ocr-low-confidence.json`
  - `mocks/video/v1/preprocess.ocr-failed.json`
  - `mocks/video/v1/preprocess.ocr-timeout.json`
  - `mocks/video/v1/preprocess.validation-error.json`

期望：

- 视频创建与图片预处理两条链路的成功、校验失败、权限失败、OCR 降级分支都有可消费样例。
- mock 样例字段口径与对应 schema 一致，不再保留 `200` 成功码的旧视频创建样例。

### 3. mock handler 行为闭环

操作：

- 将前端切到 `mock` 模式后打开 `/video/input`。
- 分别执行文本提交、图片提交、空输入提交。
- 观察 `Network` 中 `POST /api/v1/video/tasks` 与 `POST /api/v1/video/preprocess`。

期望：

- 文本创建与图片创建都能返回成功 payload，并驱动页面跳转到 `/video/:id/generating`。
- 空输入或畸形请求体返回 `422`，而不是 `500`。
- 权限失败分支能返回 `403` 错误包，不会变成未定义异常。

## Story 3.2 验收项：输入页多模态交互

### 1. 单一核心输入区

操作：

- 打开 `http://127.0.0.1:5173/video/input`。

期望：

- 页面只有一个核心输入区，不需要切换独立文本页、图片页或额外弹窗。
- 页面能看到统一主 CTA、建议标签、引导卡片与公开视频瀑布流。

### 2. 文本输入提交

操作：

- 在输入框中输入一段不少于 10 个字符的题目文本。
- 点击 `生成视频`。

期望：

- 提交期间主 CTA 进入 loading 态并禁用。
- 成功后跳转到 `/video/:id/generating`。
- 不会重复提交，不会停留原页无反馈。

### 3. 图片上传、预览与清除

操作：

- 上传一张 PNG / JPG / WebP 图片。
- 观察缩略图、文件名、文件大小。
- 点击移除按钮，再次回到文本模式。

期望：

- 图片上传后页面显示预览区。
- 表单自动切到 `image` 模式。
- 清除图片后可恢复文本模式，旧的文本错误提示不会残留在图片模式里。

### 4. 粘贴图片输入

操作：

- 将剪贴板中准备好的图片直接粘贴到输入卡片内。

期望：

- 页面能识别粘贴图片并进入图片模式。
- 粘贴行为不会导致页面刷新或输入区失焦异常。

### 5. 错误提示与键盘可达

操作：

- 空输入直接提交。
- 上传非法类型文件或大于 10MB 的图片。
- 使用 Tab 在 textarea、上传按钮、语音按钮、主 CTA 之间切换。
- 在 textarea 聚焦状态按 Enter 提交。

期望：

- 输入为空、图片类型错误、图片过大时，页面显示可恢复的 inline 错误提示。
- 上传图片后，旧文本错误不会错误保留。
- 焦点轮廓清晰可见，键盘可完成主流程。

## Story 3.3 验收项：图片预处理接口

### 1. 正常预处理成功

操作：

- 在 `real` 模式下上传一张清晰、合法的图片。
- 观察 `POST /api/v1/video/preprocess` 返回。

期望：

- 返回 `200`。
- `data` 至少包含 `imageRef`、`ocrText`、`confidence`、`width`、`height`、`format`、`suggestions`。
- `imageRef` 为 URI 风格引用，如 `local://...`，前端不直接使用物理存储路径。

### 2. 非法图片拦截

操作：

- 分别上传：
  - 非图片文件，例如 `.pdf`、`.gif`
  - 大于 10MB 的图片
  - 空文件或损坏图片

期望：

- 接口返回 `422`。
- 错误码分别落在视频域约定内，例如 `VIDEO_IMAGE_FORMAT_INVALID`、`VIDEO_IMAGE_TOO_LARGE`、`VIDEO_IMAGE_UNREADABLE`。
- 页面留在当前页，用户可以继续修正输入。

### 3. 低置信度与 OCR 降级

操作：

- 分别准备：
  - 模糊但可识别图片
  - OCR 失败图片
  - OCR 超时场景

期望：

- 低置信度时返回 `confidence < 0.6`，并在 `suggestions` 中提示用户核对结果。
- OCR 失败或超时时，不因 OCR 问题整体阻断图片存储成功。
- 失败与超时分支给出明确错误码或建议，如 `VIDEO_OCR_FAILED`、`VIDEO_OCR_TIMEOUT`。

## Story 3.4 验收项：任务创建、运行态与元数据

### 1. 创建接口返回 202 Accepted

操作：

- 在 `real` 模式下分别发起文本创建与图片创建。
- 观察 `POST /api/v1/video/tasks` 返回。

期望：

- 返回 `202`。
- `data` 包含 `taskId`、`taskType: "video"`、`status: "pending"`、`createdAt`。
- 字段命名与 Story 3.1 契约一致，没有蛇形字段漂移。

### 2. 幂等校验

操作：

- 使用同一 `clientRequestId` 连续提交两次同一请求。

期望：

- 第一次返回 `202`。
- 第二次返回 `409`，并带回已有 `taskId`，不会创建重复任务。

### 3. 权限与输入错误分支

操作：

- 用无 `video:task:add` 权限的账号发起创建。
- 构造缺少 `sourcePayload`、缺少 `imageRef` 或空文本的请求。

期望：

- 权限问题返回 `403`，错误码保持认证域一致，例如 `AUTH_PERMISSION_DENIED`。
- 输入问题返回 `422`，错误码与视频域或统一任务域契约一致。
- 所有失败都遵循统一 `{ code, msg, data }` 包结构。

### 4. RuoYi 元数据承接

操作：

- 创建成功后，用返回的 `taskId` 查询：
  - `GET /api/v1/video/tasks/{taskId}`
  - 或按 `userId`、`sourceSessionId` 查询 `/api/v1/video/tasks`

期望：

- 能查到对应视频任务元数据。
- 元数据中至少能看到 `taskId`、`userId`、`status`、`summary`、`createdAt`、`updatedAt` 等字段。
- 若 RuoYi 暂时不可用，创建链路仍不应被整体阻断。

### 5. Redis 运行态与 Worker 占位

操作：

- 创建成功后，用 `redis-cli` 检查运行态 key。
- 同时观察 Worker 消费后的任务状态变化。

期望：

- 创建后能看到初始化运行态，至少包含 `taskId`、`taskType`、`status`、`progress`、`createdAt`。
- 初始快照应先满足 `pending` 受理语义。
- Worker 占位消费后，状态可进入 `processing`，但不应直接错误跳到 `completed`。

### 6. 页面等待态承接

操作：

- 从 `/video/input` 创建成功后进入 `/video/:id/generating`。

期望：

- 等待页路由参数中的 `taskId` 与创建接口返回值一致。
- 页面至少能展示等待态占位与任务编号，不会白屏或落回输入页。
