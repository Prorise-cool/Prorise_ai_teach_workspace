## 3. 当前架构共识

- 视频链路仍以我方现有 FastAPI 基础设施为主：
  - 任务调度走 `Dramatiq + Redis + TaskFramework`
  - 运行态和 SSE 走现有 `RuntimeStore`
  - 长期业务数据仍通过 RuoYi 防腐层回写
  - Manim 代码生成、修复、渲染、TTS、compose 属于同一条视频 pipeline
- 不采用参考项目把 TTS 直接塞进 Manim `with self.voiceover(...)` 的方式。
- 但参考项目里“语音时长驱动画面时长”的思路是有效的，必须在我方架构里等价实现。

## 4. 参考项目与我方项目的关键差异

### 4.1 参考项目的有效点

- 参考项目模板 [`../../../references/manim-to-video-claw/scenext/code_render/templates/base.j2`](../../../references/manim-to-video-claw/scenext/code_render/templates/base.j2) 通过 `with self.voiceover(...)` 直接把语音时长绑定到每个 scene 的动画时长。
- 这使得“语音很长、画面很短”时，不容易出现最终视频尾部长时间静止的问题。

### 4.2 我方项目的目标架构

- 我方保持 TTS 在外部服务完成，再由 compose 阶段合成。
- 因此必须自己补齐“分镜时长估算 -> Manim 持续时长提示 -> compose 防止超长静止补帧”这套控制链。

## 5. 已经确认并修复的问题

## 5.1 已修复问题：视频有配音，但总结归纳阶段长时间静止

### 现象

- 用户原始反馈：视频里“光有配音，画面完全没动，到了总结归纳就一直停了”。

### 已确认根因

- 旧链路去掉了参考项目的 `voiceover` 时长控制，但没有补上等价的时长机制。
- 旧成功任务 `vtask_20260409113026_d5f0a8e7` 的关键事实：
  - 原始 Manim 渲染视频仅约 `10s`
  - 最终合成视频约 `87s`
  - `compose.py` 通过 `tpad=stop_mode=clone` 把画面硬拉长去匹配语音，导致尾部长时间冻帧

### 已实施修复

- 在 [`../../../packages/fastapi-backend/app/core/config.py`](../../../packages/fastapi-backend/app/core/config.py) 新增视频时长与补帧控制配置：
  - `video_narration_chars_per_second`
  - `video_narration_sentence_pause_seconds`
  - `video_scene_min_duration_seconds`
  - `video_scene_max_duration_seconds`
  - `video_scene_ambient_motion_chunk_seconds`
  - `video_compose_max_pad_seconds`
  - `video_compose_max_pad_ratio`
- 在 [`../../../packages/fastapi-backend/app/features/video/pipeline/_helpers.py`](../../../packages/fastapi-backend/app/features/video/pipeline/_helpers.py) 新增 `estimate_narration_duration_seconds(...)`
- 在 [`../../../packages/fastapi-backend/app/features/video/pipeline/storyboard.py`](../../../packages/fastapi-backend/app/features/video/pipeline/storyboard.py)：
  - 当 LLM 没给可信 `durationHint` 时，按旁白文本估算时长
  - 统一做 storyboard 归一化
- 在 [`../../../packages/fastapi-backend/app/features/video/pipeline/code_render/templates/base.j2`](../../../packages/fastapi-backend/app/features/video/pipeline/code_render/templates/base.j2)：
  - 已存在 `scene_duration_hint`
  - 已有 `hold_scene(...)` 与 `clear_scene(...)`
- 在 [`../../../packages/fastapi-backend/app/features/video/pipeline/manim.py`](../../../packages/fastapi-backend/app/features/video/pipeline/manim.py)：
  - 已把 `scene_duration_hint` 透传到渲染上下文
- 在 [`../../../packages/fastapi-backend/app/features/video/pipeline/compose.py`](../../../packages/fastapi-backend/app/features/video/pipeline/compose.py)：
  - 增加 `-shortest`
  - 使用 `voice_text` 生成字幕
  - 对过大的音视频时长差做拒绝或限制，不再静默制造超长 frozen tail
- 在 [`../../../packages/fastapi-backend/app/features/video/pipeline/artifact_writeback.py`](../../../packages/fastapi-backend/app/features/video/pipeline/artifact_writeback.py)：

## 6. 当前仍未解决的核心问题

## 6.1 问题 A：真实总耗时远超目标，无法压到 2 分钟以内

### 最新基准任务

- 用户提供新 token 后再次实测的任务：`vtask_20260409121947_77679395`
- 该次目标：重新请求真实接口，计算总耗时并评估是否小于 2 分钟

### 真实时间线

- `2026-04-09 20:19:47` 创建任务
- `2026-04-09 20:27:18` 进入第 1 次 `manim_fix`
- `2026-04-09 20:29:00` 进入第 2 次 `manim_fix`
- `2026-04-09 20:31:00` worker 抛出 `TimeLimitExceeded`

### 当前结论

- 该次真实端到端耗时至少 `11 分 13 秒`
- 且任务没有成功产出最终结果
- 因此当前链路距离“2 分钟内完成”差距非常大

### 已观察到的直接耗时来源

- 并非 TTS 独占耗时，主要慢在：
  - 多轮 `gemini-3.1-pro-high` LLM 请求
  - `manim_gen` 后进入 `manim_fix` 自动修复
  - `manim_fix` 连续两轮，进一步拖慢耗时并最终触发超时

## 6.2 问题 B：任务失败后，API 状态没有正确落成失败态

### 已确认事实

- worker 日志在 `2026-04-09 20:31:00` 明确抛出：
  - `dramatiq.middleware.time_limit.TimeLimitExceeded`
- 且随后出现：
  - `Retries exceeded for message ...`
- 但 API 侧仍返回：
  - `/status` 仍是 `processing`
  - `/result` 仍是 `processing`
  - 阶段停在 `manim_fix`

### 当前判断

- 这不是单纯的性能问题。
- 当前还有“worker 超时异常没有被统一映射并回写任务失败状态”的运行态缺口。
- 这会导致前端等待页和外部调用方都误以为任务仍在执行。

## 6.3 问题 C：阶段状态更新粒度不足，容易误判卡死点

### 已确认事实

- 在基准任务运行期间，`updatedAt` 会更新，但 `/status` 中一度长时间停留在：
  - `currentStage = tts`
  - `progress = 84`
- 实际上 worker 里已经继续执行了后续多个 Gemini 调用和 `manim` 生成/修复

### 当前判断

- 当前事件流或状态快照写回不够细，导致 API 的阶段读数滞后。
- 这会让排查者误以为是 `tts` 卡死，实际却是后面的 `manim_gen / manim_fix` 在跑。

## 7. 模型绑定现状

- 没有确认到“代码层硬编码死绑 Gemini”的结论。
- 但从真实运行日志看，视频链路当前有效路由确实是：
  - `provider=openai-gemini-3_1-pro-high`
  - `model=gemini-3.1-pro-high`
- 至少在以下阶段，日志已显示该 provider 被实际调用：
  - `understanding`
  - `storyboard`
  - `manim_gen`
  - `manim_fix`
- 因此“当前运行时为什么慢”的现实答案是：
  - 视频主链路实际上正在跑 `gemini-3.1-pro-high`
  - 而不是更轻量的模型

## 8. 对接手模型最重要的判断

### 不要重复做的事

- 不要把当前主问题继续归因于“总结阶段 frozen tail”，这个问题已经修过并验证过。
- 不要把 TTS 移回 Manim `voiceover` 架构，用户明确要求保持我方现有外置 TTS 架构。
- 不要另起一套脱离现有任务框架和 provider 基建的旁路实现。

### 现在最该打的点

- 第一优先级：为什么 `manim_gen` 经常产出需要二次甚至多次 `manim_fix` 的脚本
- 第二优先级：为什么 worker `TimeLimitExceeded` 后任务仍停留在 `processing`
- 第三优先级：如何把重型阶段从当前 `gemini-3.1-pro-high` 降本提速，至少把总耗时拉回可接受范围

## 9. 建议接手顺序

### 9.1 先解决失败可见性

- 给 `TimeLimitExceeded` 做统一异常捕获、失败态回写、SSE 事件补发
- 保证前端和 API 不再出现“实际上死了，但看起来还在 processing”

### 9.2 再解决 `manim_fix` 命中率过高

- 对本次失败任务的 prompt、生成代码、修复 prompt、修复输出做完整留痕
- 统计是：
  - 模板约束不够
  - 代码生成 prompt 太松
  - 还是某些场景下字幕/布局/对象生命周期特别容易出错

### 9.3 最后解决整体时延

- 给每个阶段加精确耗时埋点：
  - `understanding`
  - `storyboard`
  - `manim_gen`
  - `manim_fix`
  - `render`
  - `tts`
  - `compose`
  - `upload`
- 评估是否允许分阶段切换到更轻量模型
- 评估 `manim_fix` 次数、prompt 大小、并发度和 time limit 配置

## 11. 复现实验建议

- 不要把用户临时 token 写入仓库文档或源码。
