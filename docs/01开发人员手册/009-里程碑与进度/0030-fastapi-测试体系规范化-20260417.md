# FastAPI 测试体系规范化说明

## 背景

- 关联 Issue：`#167 Normalize FastAPI backend test suite structure and CI`
- 背景 Issue：`#166`
- 工作分支：`chore/167-fastapi-test-suite-normalization`
- 范围：`packages/fastapi-backend`

本次工作把 FastAPI 后端测试当作独立治理任务处理，不再沿用“补一个坏一个”的零散修法，而是一次性收敛测试目录边界、pytest 执行契约、公共 helper、本地命令入口和 GitHub Actions，确保后续新增测试能继续沿着统一规范落地。

## 问题基线

治理前的测试资产存在几类明显噪音：

- `pytest --collect-only` 已暴露 marker 未注册问题，执行契约不完整。
- 测试文件重复手写 `create_app + override_auth + TestClient` 装配，路径 helper 也存在硬编码层级。
- `tests/e2e/` 中混入静态 HTML 样例，不属于可执行自动化测试。
- `tests/unit/assets/` 中两组 Epic 10 资产测试依赖仓库外文档和 SQL 文件，既不验证 FastAPI 行为，也让主测试命令出现稳定红灯。

治理前的实际基线是：`313 collected / 6 failed`，6 个失败都集中在上述跨仓库资产测试。

## 本次收敛内容

### 1. 统一 pytest 执行契约

- 新增 `packages/fastapi-backend/pytest.ini`
- 在 `packages/fastapi-backend/pyproject.toml` 中移除旧 pytest 配置，避免配置源分裂
- 在 `packages/fastapi-backend/tests/conftest.py` 中追加目录级自动 marker 规则，让 `unit / api / integration / contract / e2e` 按目录自动归类

这次不要求每个测试文件手写主 marker，统一由收集阶段按目录打标，减少后续维护漂移。

### 2. 抽取公共测试 helper

- 新增 `packages/fastapi-backend/tests/helpers/app.py`
- 新增 `packages/fastapi-backend/tests/helpers/paths.py`
- 新增 `packages/fastapi-backend/tests/helpers/__init__.py`
- 新增 `packages/fastapi-backend/tests/unit/core/test_test_helpers.py`

其中：

- `app.py` 统一承接带认证上下文的 app/client 装配
- `paths.py` 统一承接仓库根路径解析

现有 API / integration 测试里重复的装配逻辑已切到 helper，避免继续散落。

### 3. 清理伪测试与低价值资产测试

以下文件已直接删除：

- `packages/fastapi-backend/test_llm_minimal.py`
- `packages/fastapi-backend/tests/e2e/video-transparent-test.html`
- `packages/fastapi-backend/tests/unit/assets/test_epic10_data_boundary_assets.py`
- `packages/fastapi-backend/tests/unit/assets/test_epic10_ruoyi_xiaomai_module_assets.py`

删除原则很明确：只保留真正验证 FastAPI 代码行为、且能在本仓库边界内稳定执行的测试。跨仓库文档存在性检查、静态演示 HTML 和临时排查脚本不再占用测试语义目录。

### 4. 对齐本地命令与 CI

- 更新根目录 `package.json`，补齐 `test:fastapi-backend:contracts`、`test:fastapi-backend:coverage`、`test:fastapi-backend:ci`
- 更新 `packages/fastapi-backend/README.md`，明确测试分层和 `e2e` 只保留可执行自动化测试
- 新增 `.github/workflows/fastapi-backend-tests.yml`，让 GitHub Actions 直接跑同一套 FastAPI 测试入口

这样本地和 CI 不再各自维护一套命令口径。

## 验证结果

本次治理后的最终结果如下：

- `packages/fastapi-backend/.venv/bin/python -m pytest packages/fastapi-backend/tests --collect-only -q`
  - `306 tests collected`
  - 无 `UnknownMarkWarning`
- `pnpm test:fastapi-backend`
  - `306 passed`
- `pnpm test:fastapi-backend:ci`
  - collect-only、unit、api、integration 全通过
- `pnpm test:fastapi-backend:coverage`
  - `306 passed`
  - 总覆盖率 `80%`

也就是说，本次治理把测试基线从 `313 collected / 6 failed` 收敛到了 `306 passed`，并把噪音资产从主回归路径里彻底移除。

## 当前规范结论

后续在 `packages/fastapi-backend` 新增测试时，统一遵守以下口径：

1. 测试必须落在 `tests/unit`、`tests/api`、`tests/integration`、`tests/contracts`、`tests/e2e` 之一。
2. `tests/e2e` 只允许放真正可执行的自动化测试，不再存放静态样例或人工演示文件。
3. 公共 app/client 装配和路径 helper 优先复用 `tests/helpers/`，不要在业务测试里重复造轮子。
4. 本地执行、CI 和 README 说明统一以 `package.json` 中的标准脚本为准。

## 收口结论

这次 `#167` 的目标不是“多写几个测试”，而是把 FastAPI 测试体系本身变成可维护的工程资产。当前结果已经满足独立分支治理、CI 对齐、文档回写和稳定回归四个收口条件，可以作为后续 FastAPI 测试增量建设的新基线。
