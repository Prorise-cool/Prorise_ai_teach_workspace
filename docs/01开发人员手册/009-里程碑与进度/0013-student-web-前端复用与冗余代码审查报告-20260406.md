# student-web 前端复用与冗余代码审查报告

日期：2026-04-06  
范围：`packages/student-web`  
审查方式：静态代码审查，聚焦组件复用、重复实现、冗余代码与前端基础设施落地一致性。  
说明：当前仓库不存在名为 `docs` 的 Git 分支，本报告按仓库既有约定落在 `docs/01开发人员手册/009-里程碑与进度/`。

## 审查结论摘要

1. `student-web` 已经具备一批可复用资产，尤其是 `components/input-page/*`、`features/auth/components/auth-fields.tsx`、`shared/feedback/*` 这些抽象方向是对的。
2. 但“页面壳层先复制，再局部改造”的趋势已经开始出现，最明显的是 `classroom` / `video` 输入页、`GlobalTopNav` / `LandingTopNav`、`LoginForm` / `RegisterForm`。
3. 当前最需要优先收敛的不是视觉，而是“输入工作区壳层”“顶栏导航能力”“认证验证码流程”这 3 组重复实现；如果继续按现状扩新页面，后续维护成本会明显上升。
4. `components/ui` 原语层已经存在，但落地深度仍然偏浅，尤其 `Card` 的复用不足，说明共享原语与业务场景之间还没有真正接上。

## 审查依据

- 事实源：`_bmad-output/INDEX.md`
- 前端约束：`docs/01开发人员手册/004-开发规范/0010-student-web-前端依赖落地与-scss-分层规范.md`
- 审查代码范围：`packages/student-web/src`

## 核心发现

### 高：`classroom` 与 `video` 输入页已经形成“双轨实现”，复用停留在局部组件，壳层和卡片仍大量重复

涉及文件：

- `packages/student-web/src/features/classroom/pages/classroom-input-page.tsx:41`
- `packages/student-web/src/features/video/pages/video-input-page.tsx:32`
- `packages/student-web/src/features/video/components/video-input-card.tsx:19`
- `packages/student-web/src/features/classroom/styles/partials/_classroom-input-card.scss:5`
- `packages/student-web/src/features/video/styles/partials/_video-input-card.scss:5`

证据：

1. 两个页面都各自声明了 `EntryNavLink`、`WorkspaceRoute`、`GUIDE_CARD_ICONS`，并且都维护了同一套 `motion` 容器动画、顶部 radial glow、`GlobalTopNav` 参数、标题区、建议区、引导卡片和社区瀑布流壳层。
2. `classroom-input-page.tsx` 中的核心输入卡片仍是整块内联实现；`video-input-page.tsx` 虽然把卡片拆成了 `VideoInputCard`，但页面壳层仍几乎与课堂页平行复制。
3. 两份 SCSS 卡片文件的主体结构高度一致：容器、提示栏、正文、工具栏、工具按钮、提交按钮几乎是同一份代码，只在少量交互和视觉值上分叉。

影响：

- 任何输入页公共改动都需要至少改 2 处页面 + 2 处样式。
- 页面已经开始出现“一个场景抽组件，另一个场景继续内联”的不对称状态，后续最容易演变成第三套实现。
- 这类重复直接削弱了已有 `components/input-page/*` 共享层的价值。

建议：

1. 抽出统一的 `WorkspaceInputPageShell`，承接顶部 glow、导航、标题、建议标签、引导卡片、社区区块和动画 variants。
2. 把 `classroom` 当前内联卡片收敛成独立组件，并与 `VideoInputCard` 对齐为同一套基础壳层，例如 `BasePromptCard` + 场景差异 slot。
3. 把卡片主体的共用 SCSS 下沉到共享 partial，页面只覆写差异项，不再维护两份平行 BEM 结构。

### 中：顶栏导航被拆成 `GlobalTopNav` 与 `LandingTopNav` 两套实现，主题、语言、移动菜单逻辑重复

涉及文件：

- `packages/student-web/src/components/navigation/global-top-nav.tsx:82`
- `packages/student-web/src/features/home/components/landing-top-nav.tsx:38`
- `packages/student-web/src/features/home/landing-page.tsx:20`

证据：

1. 两个组件都维护了 `mobileMenuOpen` 状态、品牌按钮、语言切换、主题切换、移动端 `Dialog` 菜单和响应式桌面/移动分支。
2. 两者都消费了同一批依赖：`appI18n`、`useThemeMode`、`Button`、`Dialog`，但没有共享统一导航壳层。
3. `GlobalTopNav` 已经支持 `variant`，但 `LandingTopNav` 仍然另起一套专用实现，只是多了 feature preview popover 和锚点滚动。

影响：

- 顶栏交互的改动会在两套实现中重复发生，容易出现桌面端一致、移动端不一致的漂移。
- 语言切换、主题切换、品牌点击行为的可访问性修复需要双修。

建议：

1. 以 `GlobalTopNav` 为唯一宿主，增加 `landing` 变体或扩展 slot。
2. 让 `LandingTopNav` 只保留落地页特有的 feature preview 内容，而不是重新实现整个导航。

### 中：认证验证码流程在登录与注册表单中成块复制，后续极易漂移

涉及文件：

- `packages/student-web/src/features/auth/components/login-form.tsx:46`
- `packages/student-web/src/features/auth/components/register-form.tsx:45`

证据：

1. 两个表单都维护了相同的 `CaptchaState` 结构。
2. 两个表单都定义了几乎同构的 `refreshCaptcha`，包括：`service.getCaptcha()`、写入 `captchaEnabled/imageBase64/uuid`、清空 `code`、清理错误、失败后回退到 `bootstrapFailed`。
3. 两个表单在提交前都重复校验“验证码必填”和“uuid 缺失”，提交失败后都重复刷新验证码。

影响：

- 验证码接口字段、错误文案、刷新时机一旦变化，需要同步改两份业务逻辑。
- 一旦其中一个表单补了节流、防抖、失败态或 loading 细节，另一个表单很容易漏改。

建议：

1. 抽出 `useAuthCaptcha` 或 `useCaptchaGate`，统一管理获取、刷新、校验与错误回退。
2. 登录与注册表单只保留各自真正不同的提交 payload 和成功回调。

### 中：`react-query` 已经落地，但认证页仍保留手写服务端状态机，基础设施复用不一致

涉及文件：

- `packages/student-web/src/features/auth/pages/login-page.tsx:151`
- `packages/student-web/src/features/profile/hooks/use-user-profile.ts:28`
- `packages/student-web/src/features/video/hooks/use-video-create.ts:27`

证据：

1. `profile` 与 `video` 已经使用 `useQuery` / `useMutation` 承接服务端状态。
2. 但 `login-page.tsx` 仍手写 `useEffect + cancelled flag + useState` 去拉取 `service.getRegisterEnabled(...)`。
3. 这与项目文档里“服务端状态优先进入 `react-query`”的约束不一致。

影响：

- 同一项目里并存两套服务端状态处理风格，增加认知成本。
- `registerEnabled` 的缓存、重试、错误态与失效策略无法复用统一机制。

建议：

1. 新增 `useRegisterEnabledQuery`，把注册开关状态纳入 `react-query`。
2. 认证页只消费 query 结果，不再手写 bootstrap 与取消逻辑。

### 低：`components/ui/card.tsx` 几乎没有进入真实业务复用，说明共享原语层和业务场景之间仍有断层

涉及文件：

- `packages/student-web/src/components/ui/card.tsx:1`
- `packages/student-web/src/features/auth/pages/forbidden-page.tsx:10`

证据：

1. 当前 `Card` 原语只有 `forbidden-page` 明确消费。
2. 与此同时，输入页卡片、社区卡片、落地页 benefit 卡片等仍然各自直接写 `rounded + border + background + shadow` 组合。

影响：

- `Card` 作为原语存在，但没有成为真实的复用基础，容易沦为“存在感很低的半成品抽象”。

建议：

1. 先决定 `Card` 的定位是否真要承接业务卡片。
2. 如果要承接，就让输入页说明卡、状态卡、营销卡片逐步接入；如果不打算承接，就不要继续把它当成已落地能力对外宣称。

## 正向复用资产

以下部分值得保留并继续扩展：

1. `components/input-page/*` 已经把 Header、GuideCards、Suggestions、`useBrowserAsr`、`useFileDropzone` 抽出来，这是正确方向。
2. `features/auth/components/auth-fields.tsx` 对 `Input`、密码显隐、验证码输入的抽象比较扎实。
3. `shared/feedback/*` 已经基于 `sonner` 做了统一封装，说明全局反馈并没有继续散落到页面内。
4. `profile`、`video` 对 `react-query` 的使用方式，已经比认证页更接近项目规范。

## 建议收敛顺序

### 第一优先级

1. 收敛输入工作区壳层与卡片基类。
2. 收敛验证码逻辑。

### 第二优先级

1. 合并顶栏导航宿主。
2. 把认证页注册开关改成 `react-query`。

### 第三优先级

1. 评估 `Card` 是否继续作为共享原语推进。
2. 对 landing 区块做第二轮抽象，只在确认视觉模式稳定后再提炼共享 section / card 模板。

## 本次未覆盖

1. 未做浏览器运行态验证，仅做源码级审查。
2. 未覆盖 `packages/student-web` 之外的后端、契约和 RuoYi 管理端实现。
3. 未对性能、a11y、测试覆盖率做完整专项评估，本报告只聚焦“前端复用与冗余代码”。
