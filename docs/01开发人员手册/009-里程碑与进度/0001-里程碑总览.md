## 当前审批结论

1. 可以继续推进 `Epic 1`，范围以 `1.2 ~ 1.6` 为主。
2. `1.7` 已形成 Story 文档，但仍应与首页边界一起收口后再进入实现。
3. `Epic 3+` 不做“整体放行”，只做逐 Story 审批。
4. `3.6 / 4.10 / 5.9 / 5.10 / 7.7` 等新增后续 Story 已回收到 `backlog`，待前置契约和依赖满足后再放行。

## 审批门禁

1. `working tree` 中尚未冻结的 Story、Epic 或 `sprint-status` 变更，不能仅因文件已存在就视为审批结论。
2. `ready-for-dev` 之前必须同时满足：
   - 依赖 Story / Epic 已满足；
   - Suggested File Targets 与当前真实代码结构一致；
   - canonical source 与开发手册没有相互冲突；
   - mock / contract / runtime 边界已明确。
3. 对后续 Epic 的前台 Story，若当前前端宿主尚无对应 feature 目录或路由承载，默认仍按 `backlog` 处理。

## 阶段目标

### M1 - 技术验证

- 完成 `Epic 0 / 2 / 10` 的底座闭环。
- 让 `Story 1.1` 的认证契约、mock 与 seam 稳定可复用。
- 形成后续 Story 的审批和执行边界，避免“文档先跑、依赖未清”。

### M2 - MVP 开发

- 完成正式 `/login`、首页双入口、视频主链路、课堂主链路的前后台承接。
- 让主链路在 mock / real 下具备一致行为。

### M3 - 测试上线

- 进行主链路联调、端到端回归、部署与监控收口。

### M4 - 赛事提交

- 收口演示视频、答辩 PPT、系统演示环境与参赛材料。
