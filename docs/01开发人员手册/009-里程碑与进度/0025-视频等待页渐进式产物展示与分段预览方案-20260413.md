# 视频等待页渐进式产物展示与分段预览方案

日期：2026-04-13

## 结论

- 技术上可行，但不是“前端单独把最终视频分片加载”这么简单。
- 当前瓶颈主要在 `manim_gen`，它产出的是 Manim 脚本，不是可播放媒体；因此在这个阶段无法兑现“边生成边播放最终视频”的承诺。
- 推荐方案是把等待体验优化为两段式：
  - `storyboard` 完成后，先给用户展示真实分镜与摘要；
  - `render` 每完成一个 section，就把该 section 的片段作为稳定 URL 暴露给等待页预览。

## 当前链路边界

- 等待页目前只消费阶段、进度、失败信息和日志文案，不消费真实中间产物。
- 结果页成功态只认最终 `result.videoUrl`，不会把 section 片段当成最终成功结果。
- 后端内部已有 `storyboard`、`timeline`、`knowledge_points` 等 artifact 结构，也已有 section 级渲染输出，但当前上传层只正式发布最终 `output.mp4` 与 `cover.jpg`。

## 推荐落地

### 1. 保持结果页边界不变

- `/video/:id` 仍然只在最终成片可用后进入成功态。
- 不把 section 片段预览混进结果页成功态，避免“任务未完成但页面像已完成”的语义漂移。

### 2. 新增等待页预览数据源

- 推荐新增 `GET /api/v1/video/tasks/:id/preview`，专门返回等待页渐进预览状态。
- 不建议把完整预览对象塞进统一 `/status` 主体里；可在既有 `task:progress` / `status.context` 中增加 `previewAvailable`、`previewUpdatedAt`、`previewVersion` 这类轻量信号，驱动前端增量拉取。

### 3. 渐进展示真实中间产物

- `storyboard` 完成后：
  - 展示题目摘要
  - 展示知识点摘要
  - 展示 scene 列表与旁白概要
- `render` 阶段每个 section 成功后：
  - 复制片段到稳定资产路径
  - 在 preview 数据里追加 `sectionClipUrl`
  - 等待页即时显示“已完成片段”

### 4. 明确非目标

- 不承诺在 `manim_gen` 阶段流式播放最终视频。
- 不直接向前端暴露 raw TTS 音频。
- 不要求等待页实时依赖长期存储 `xm_session_artifact` 才能恢复。

- 该方案的目标是改善“长时间等待时用户体感空窗”的问题，而不是用 UI 技巧掩盖真实主瓶颈。
- 若后续要进一步缩短总耗时，仍需继续优化 `manim_gen` 本身；渐进展示只能改善感知等待，不会替代性能优化。

按现在代码的真实情况看，有三层能力可以做：

分镜可听
现在 storyboard 生成后，agent.sections 里就已经有 lecture_lines 了，agent.py (line 331) 到 agent.py (line 345) 已经把每个 section 的讲稿拆出来了。
而 TTS 本身也是按 section 生成音频的，orchestrator.py (line 591) 到 orchestrator.py (line 621) 很明确。
所以“用户先听着”是可行的，只是现在编排顺序把 TTS 放在了 render 后面，orchestrator.py (line 346) 到 orchestrator.py (line 403)。要做成你要的效果，核心不是重写 TTS，而是把 TTS 前移到 storyboard 后。

片段可看可听
这个也可行，而且代码里其实已经做了一半。现在最终合成前，系统已经会把每个 section 视频和对应音频先拼成单独的 section_id_with_audio.mp4，然后再 concat 成整片，orchestrator.py (line 640) 到 orchestrator.py (line 666)。
也就是说，技术上完全可以把这些“带声音的 section 成品片段”提前发布给等待页，不必等最终整片出来。

基于内容问答
这条路也成立，而且比“对着最终视频做问答”更稳。因为系统已经有结构化内容资产：

Storyboard / narration / knowledge_points / solution_steps，见 models.py (line 313) 和 models.py (line 341)
这些产物还能进入 artifact graph，同步到长期索引，artifact_service.py (line 27)
所以问答最好不是“基于音频文件本身”，而是“基于分镜讲稿、知识点、解题步骤、section 锚点”的结构化问答。这样能回答“刚才第 3 段为什么这么讲”“这个公式是哪一步来的”，也方便以后接 EvidenceProvider / Learning Coach。

我建议最终产品形态直接做成一条三段式链路，而不是只做预览：

阶段 A：storyboard 完成后，立刻开放“听讲稿”音频预览
阶段 B：每个 section 渲染完成后，追加“可看可听”的片段
阶段 C：整片完成后进入结果页；同时保留“围绕当前视频内容提问”的入口
