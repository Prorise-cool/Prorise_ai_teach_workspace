# 0021 视频管道 429 重试风暴热修复说明

日期：2026-04-10

## 结论

- 已在视频管道热修中处理 `manim_gen` 并行场景生成触发的 429 重试风暴。
- 这次修复同时覆盖了三条链路：Failover 退避打散、429 健康缓存误伤、Manim 单次 fallback 被 cached unhealthy 误杀。
- 默认并发已从 `5` 下调到 `3`，环境变量仍可按上游配额继续调低。

## 关键改动

- `ProviderFailoverService` 重试退避增加 `jitter`，避免多个协程在 `1s / 2s / 4s` 的整点同时重试。
- `429 / rate limit` 归类为可重试但不写坏健康缓存，避免限流把 Provider 长达 `60s` 判成不健康。
- `ManimGenerationService` 在并行 / 逐场景失败后，单次全量 fallback 会显式忽略当前请求内的 cached unhealthy。
- `FASTAPI_VIDEO_MANIM_PARALLEL_SCENE_CONCURRENCY` 默认值改为 `3`。

## 运行建议

- 若上游仍然出现 RPM 限流，可优先通过环境变量把 `FASTAPI_VIDEO_MANIM_PARALLEL_SCENE_CONCURRENCY` 下调到 `2`。
- 重点观察日志中的 `provider_switch`、`status=429` 与同一秒内的重试密度，确认不再出现成片同步重试。
- 若未来仍出现长时间限流，下一步应考虑 provider 级全局 rate limiter，而不是继续单纯堆重试次数。
