# 学生端前端架构调研与骨架建议

> 状态：调研完成，待确认
> 更新时间：2026-03-28
> 适用范围：`packages/student-web/` 学生端前台

## 1. 调研目标

本次调研不是为了挑一个现成模板直接照搬，而是要为小麦学生端确定一套可长期演进、同时满足 Epic 并行开发约束的前端骨架。

本次调研结合了 4 类输入：

1. 小麦现有 Epic / 架构约束；
2. `bulletproof-react` 的应用层分层方式；
3. `vite-component-library-template` 的 Storybook / Vitest / library mode 设施；
4. `slash-admin` 的主题 token、工作台壳层与视觉组织方式。

## 2. 先说结论

小麦最适合采用的不是“单模板继承”，而是“三层拼装”方案：

1. 用 `bulletproof-react` 提供应用层骨架。
2. 用 `vite-component-library-template` 提供组件开发与测试设施，但只吸收 Storybook 9、Vitest、Vite library mode 的方法，不继承其发布与 hooks 流程。
3. 用 `slash-admin` 作为视觉系统与工作台壳层参考，但只借鉴 token 分层、布局方式和学习中心类页面气质，不继承其 Ant Design 兼容层与后台信息架构。

最终落地时，小麦应保持：

- 单仓 `pnpm workspace`；
- 学生端主工程仍然是 `packages/student-web/`；
- 不引入 `husky`、`lefthook`、`lint-staged`、`release-please`、自动发包脚本；
- `lint`、`test`、`typecheck` 只保留手动命令；
- 前端按 `adapter + mock + SSE 状态机 + feature modules` 组织，而不是按页面堆平文件。

## 3. 小麦的硬约束

### 3.1 来自 Epic 与架构文档的约束

- 前端 Starter 已固定：`shadcn/ui CLI v4 + Vite 6.x + React 19 + TypeScript 5.7+ + Tailwind CSS v4 + Radix UI`。
- 必须手动集成：`Zustand`、`Motion`、`KaTeX / Temml`、`Shiki`、HTTP 客户端、`react-router-dom`、`react-i18next`、`Video.js`、SSE 客户端工具。
- 必须先有统一 `adapter` 和 `mock handler`，页面不能直接依赖具体 HTTP 实现。
- 所有任务型页面都必须围绕统一状态枚举与统一 SSE 事件工作：
  - `pending`
  - `processing`
  - `completed`
  - `failed`
  - `cancelled`
- SSE 事件至少支持：
  - `connected`
  - `progress`
  - `provider_switch`
  - `completed`
  - `failed`
  - `heartbeat`
  - `snapshot`
- 路由边界已冻结：
  - `/login`
  - `/learning`
  - `/history`
  - `/favorites`
  - `/profile`
  - `/settings`
  - `/video/...`
  - `/classroom/...`
- i18n 当前只要求“架构预留”，不要求本阶段完成完整双语。

### 3.2 当前仓库现实状态

当前仓库根级脚本、`packages/INDEX.md`、环境文档都假定 `packages/student-web/` 已存在，但实际代码目录中还没有这个包。

这意味着小麦前端当前最真实的问题不是“选哪个模板”，而是：

1. 先把 `packages/student-web/` 的真实工程骨架补出来；
2. 再把 Epic 0 要求的 `adapter/mock/contracts` 基线落地；
3. 再进入 Epic 1 以后各业务域页面开发。

## 4. 三个参考项目的可吸收部分

## 4.1 `bulletproof-react`：适合拿来做“应用层骨架”

推荐吸收：

- `src/app` 作为应用层：
  - `provider.tsx`
  - `router.tsx`
  - `routes/*`
- `src/features/*` 作为业务域模块；
- `src/components/*`、`src/hooks/*`、`src/lib/*`、`src/utils/*` 作为共享层；
- 单一 API client；
- 基于 TanStack Query 的请求封装；
- `ProtectedRoute`、`AuthLoader` 这类认证入口控制思路；
- `testing/mocks` 用 `MSW` 驱动页面状态，而不是写死 JSON。

不直接继承：

- `.husky/`；
- `lint-staged`；
- `prepare` 脚本；
- Playwright 和 Storybook 不是当前 MVP 的必要阻塞项；
- 仓库中示例的 Next.js 多应用结构对小麦没有必要，保留其中的 Vite 版本思路即可。

对小麦的翻译方式：

- 保留它的分层原则，不照搬它的目录名细节；
- 将其 `lib/api-client.ts` 思路，落到小麦约束指定的 `src/services/api/client.ts`；
- 将“feature 独立、app 层组装”的原则用于 Epic 切分，而不是做成纯路由页工程。

## 4.2 `vite-component-library-template`：适合拿来做“组件开发设施”

推荐吸收：

- Storybook 9 基线；
- Vitest + Testing Library；
- `vite-plugin-dts` 类型产物生成思路；
- Vite library mode 的打包方法；
- `src/lib/index.ts` 统一导出入口；
- 组件与 story 同目录维护的习惯。

不直接继承：

- `.husky/`；
- `lint-staged`；
- `prepack`；
- `release-please`；
- 面向 NPM 发布的 `exports` / changelog / 自动 release 流程；
- 其 `Atomic Design` 目录方式只可作参考，不应压过小麦的业务 feature 切分。

对小麦的翻译方式：

- 现阶段不建议把它做成独立可发布 NPM 组件库；
- 更适合把 Storybook / Vitest / 组件测试设施吸收到 `packages/student-web/` 内部；
- 如果未来出现第二个 React 消费端，再考虑把通用 UI 抽出为 `packages/ui/`。

原因很直接：当前小麦只有一个 React 消费端，管理端是 Vue，提前做独立发包式 UI 仓会增加复杂度，但不会立刻带来收益。

## 4.3 `slash-admin`：适合拿来做“视觉与壳层参考”

推荐吸收：

- 主题 token 分层方式：
  - `color`
  - `typography`
  - `base`
  - `shadow`
- CSS Variables + Tailwind token 映射；
- 工作台壳层布局：
  - 顶栏
  - 左侧导航
  - 主内容卡片区
  - 学习中心 / 历史 / 收藏 / 设置页的工作台气质
- 路由分段组织的思想：
  - `auth`
  - `main`
  - `dashboard`

不直接继承：

- `antd` 兼容层；
- `styled-components`；
- `lefthook`；
- 后台模板式信息架构；
- 多品牌预设切换。

对小麦的翻译方式：

- 只借视觉基础盘，不借组件实现；
- 小麦必须坚持单一品牌风格，以暖黄色主色 `#f5c547` 为核心，不做多品牌调色板切换；
- 学习流程页不能完全后台化；
- `/learning`、`/history`、`/favorites`、`/profile`、`/settings` 可以参考它的工作台布局；
- `/video`、`/classroom`、等待页、结果页必须保留更强的学习产品气质。

## 5. 推荐落地架构

## 5.1 工程结构建议

```text
packages/
  student-web/
    src/
      app/
        provider/
        router/
        routes/
        layouts/
      components/
        ui/
        shared/
      features/
        auth/
        home/
        session-task/
        video/
        classroom/
        companion/
        evidence/
        learning-coach/
        learning-center/
        profile-settings/
      services/
        api/
          client.ts
          adapters/
        sse/
        mock/
      stores/
      hooks/
      lib/
      test/
      styles/
      types/
contracts/
  frontend/
  tasks/
  auth/
mocks/
  auth/
  tasks/
  learning/
```

说明：

- `app/` 只负责路由、provider、layout、guard、页面装配；
- `features/` 按 Epic / 业务域拆分；
- `services/api/adapters/*` 作为 mock / real 切换边界；
- `services/sse/*` 负责统一 SSE 解析、重连、snapshot 恢复、`/status` 降级；
- `components/ui/*` 放通用设计系统组件；
- `components/shared/*` 放跨域业务组件，例如统一等待壳层、结果卡片、空态组件；
- `contracts/` 与 `mocks/` 放在仓库根级，服务于前后端并行，而不是只藏在前端包内部。

## 5.2 运行时分层建议

### A. App Shell

职责：

- Router；
- Provider；
- 认证守卫；
- 全局错误边界；
- Toast / Dialog 宿主；
- Query Client；
- i18n Provider；
- Theme Provider。

### B. Feature Modules

每个业务域 feature 自己维护：

- 页面容器；
- domain hook；
- adapter 调用；
- 本域组件；
- 本域类型；
- 本域 mock 样例。

禁止：

- feature 之间互相深度引用；
- 页面直接跨 feature 访问内部状态；
- 页面直接手写 HTTP 请求。

### C. Shared Infrastructure

统一沉淀：

- `api client`
- `request/response unwrap`
- `401/403/500` 统一处理
- 统一任务状态枚举
- SSE 事件类型解析
- mock driver
- 权限失败样例

## 5.3 状态管理建议

推荐组合：

- TanStack Query：服务端数据、列表、详情、分页、提交结果缓存；
- Zustand：会话内 UI 状态、播放器局部状态、等待页降级模式、Companion 侧栏状态；
- `react-hook-form + zod`：登录、输入页、资料编辑表单。

不要做的事：

- 不用 Zustand 承接远程列表缓存；
- 不把 SSE 全量事件流长期塞进 Zustand；
- 不在组件中混写请求、副作用和状态机。

## 5.4 路由组织建议

建议路由分为 4 组：

1. `public`
   - `/`
   - `/login`
2. `task-flows`
   - `/video/input`
   - `/video/:id/generating`
   - `/video/:id`
   - `/classroom/input`
   - `/classroom/:id/generating`
   - `/classroom/:id`
3. `post-session`
   - `/coach/checkpoint/:id`
   - `/coach/quiz/:id`
   - `/coach/path/:id`
4. `workspace`
   - `/learning`
   - `/history`
   - `/favorites`
   - `/profile`
   - `/settings`

其中：

- `task-flows` 更接近学习产品主链路；
- `workspace` 更接近 `slash-admin` 可参考的工作台壳层；
- 不要把所有页面都塞进同一个 dashboard shell。

## 6. 小麦不该继承的内容

以下内容应明确排除：

- `husky`
- `lefthook`
- `lint-staged`
- `release-please`
- `pre-commit` / `pre-push`
- 自动改写 `package.json` 的 `prepack`
- NPM 发包流程
- `Ant Design`
- `styled-components`
- 多品牌调色板切换
- 纯后台菜单式信息架构

保留手动命令即可：

- `pnpm --filter @xiaomai/student-web lint`
- `pnpm --filter @xiaomai/student-web test`
- `pnpm --filter @xiaomai/student-web typecheck`
- `pnpm --filter @xiaomai/student-web build`

## 7. 推荐实施顺序

### Phase 1：补真实骨架

1. 创建真实 `packages/student-web/`。
2. 接入 `shadcn/ui CLI v4 + Vite 6 + React 19 + Tailwind v4`。
3. 建立 `app / features / services / components / test` 目录基线。
4. 建立 `services/api/client.ts` 与 `services/api/adapters/*`。

### Phase 2：补并行开发轨道

1. 建立 `contracts/` 与 `mocks/` 根目录规则。
2. 接入 `MSW` 驱动 mock 模式。
3. 建立统一任务状态机与 SSE 事件解析层。
4. 建立 Storybook 9 与 Vitest，但只作为手动执行设施。

### Phase 3：进入业务 Epic

按以下先后推进最稳妥：

1. Epic 1：认证、首页、双入口；
2. Epic 2：统一任务状态、SSE、恢复与降级；
3. Epic 3-5：视频 / 课堂主链路；
4. Epic 6-9：Companion、Evidence、Learning Coach、Learning Center。

## 8. 当前需要先修正的认知偏差

当前仓库里存在 3 个不一致点，必须先统一：

1. 文档与根级脚本都认为 `packages/student-web/` 已存在，但实际目录缺失；
2. 部分旧文档仍写 `packages/user/`，与当前规划不一致；
3. 参考项目 `ruoyi-plus-soybean` 的脚本体系包含较强的提交与版本管理能力，不能误带到学生端主工程。

## 9. 最终建议

最适合小麦的前端方案不是“选一个模板”，而是：

- 用 `bulletproof-react` 的分层思想搭学生端主工程；
- 用 `vite-component-library-template` 的 Storybook / Vitest / library mode 方法搭组件开发设施；
- 用 `slash-admin` 的 token 与工作台布局方法指导学习中心类页面视觉；
- 同时严格排除所有 Git hooks、自动发布、Ant Design 兼容层和后台模板信息架构。

一句话总结：

> 小麦前端应当是“`bulletproof-react` 的工程骨架 + `vite-component-library-template` 的组件开发设施 + `slash-admin` 的视觉壳层参考”，但最终产物必须服从小麦自己的 Epic 边界、单品牌设计和单仓手动检查策略。
