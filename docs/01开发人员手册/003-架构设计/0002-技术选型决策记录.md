### ADR-001: 双语言后端架构（FastAPI + Spring Boot）

**状态**: 已接受
**日期**: 2026-03-22

#### 背景
项目已有 RuoYi-Vue-Plus (Spring Boot) 作为 ToB 管理后端，提供完善的用户管理和 RBAC 权限体系。同时，核心 AI 功能（课堂生成、视频生成）需要 Python 生态的 LangGraph、Manim 等库支持，这些在 Java 生态中没有等价替代。

#### 决策
采用双语言后端架构：
- **FastAPI (Python, 端口 8090)**：承载 ToC 功能微服务（课堂服务 + 视频服务），复用 OpenMAIC (Python) 和 ManimToVideoClaw (Python) 的核心逻辑
- **RuoYi Spring Boot (Java, 端口 8080)**：保持不动，仅通过 API 对接，提供用户管理和 RBAC
- 通过**共享 MySQL + Redis** 实现数据互通，JWT Token 两端验证，避免 HTTP 互调开销

#### 后果
- **正面**：充分利用 Python AI 生态（LangGraph, Manim, LLM SDK）；保护已有 RuoYi 投资；两端独立开发部署
- **负面**：运维复杂度增加（需维护两套后端）；需确保 JWT Token 格式和验证逻辑两端一致
- **缓解**：共享 Redis 验证 Token，无需 HTTP 互调；Docker 容器化统一部署环境

### ADR-002: ToC 前端选择 React 19 而非 Vue 3

**状态**: 已接受
**日期**: 2026-03-22

#### 背景
ToB 管理前端已使用 Vue 3 + Naive UI (Soybean Admin)。ToC 用户端需要新建前端，面临 React 和 Vue 的选择。参考项目 OpenMAIC 使用 Next.js (React) 构建，其前端组件（课堂渲染、场景展示等）可作为功能参考。

#### 决策
ToC 前端选择 **React 19 + Vite + TypeScript**，UI 组件库选择 **Shadcn/ui**：
- React 19 是最新稳定版，生态成熟
- OpenMAIC 原生使用 React，功能组件可参考复用（风格必须重写）
- Shadcn/ui 内置无障碍支持（符合 WCAG 2.1 AA 要求），高度可定制
- Vite 作为构建工具，保持与 Soybean 一致的开发体验

#### 后果
- **正面**：可参考 OpenMAIC 的 React 组件实现；Shadcn/ui 内置无障碍；React 生态库丰富（Motion、ECharts 等）
- **负面**：团队需同时维护 React 和 Vue 两个前端；技术栈不统一增加学习成本
- **缓解**：ToB 前端保持不动，无需日常开发；ToC 前端是核心交付物，React 生态更适合 AI 交互场景

### ADR-003: LLM Provider 多服务商抽象工厂

**状态**: 已接受
**日期**: 2026-03-22

#### 背景
AI 教学功能依赖 LLM 服务，但单一 Provider 存在可用性风险（限流、宕机、区域限制）。OpenMAIC 的问题审计（#118）也揭示了 Provider 缓存污染问题。

#### 决策
采用**抽象工厂模式**实现 LLM Provider 可插拔设计：
- 支持 Gemini、Claude 等多 Provider
- 自动降级策略：连续 3 次超时（30s）或 5xx 自动切换备用 Provider
- Provider 配置缓存 TTL ≤ 5 分钟，不可用 Provider 立即标记排除
- 每 60s 探测主 Provider 健康状态，恢复后自动切回

#### 后果
- **正面**：提高系统可用性；避免单点故障；可根据成本动态切换 Provider
- **负面**：抽象层增加代码复杂度；不同 Provider 的 API 差异需要适配
- **缓解**：复用 OpenMAIC 的 LLM 抽象层设计经验；统一 Prompt 模板降低适配成本

### ADR-004: TTS 多服务商优先级链 + 本地兜底

**状态**: 已接受
**日期**: 2026-03-22

#### 背景
视频生成和课堂语音需要 TTS 服务，ManimToVideoClaw 已实现豆包、百度、Spark、Kokoro 四种 TTS 的集成。需要确保 TTS 服务的高可用性。

#### 决策
采用**优先级链降级策略**：
- 正常链路：豆包 TTS → 百度 TTS → Spark TTS → Kokoro TTS（本地部署）
- 单次请求超时 15s 或返回错误时自动切换下一个
- Kokoro 作为本地部署的最终兜底方案，确保离线环境下也能生成语音
- 每 120s 探测失败的服务商，恢复后恢复原优先级

#### 后果
- **正面**：四级降级保证 TTS 服务永不中断；Kokoro 本地兜底无需网络依赖
- **负面**：不同 TTS 服务商音色差异；Kokoro 质量可能低于云服务
- **缓解**：抽象工厂模式封装差异；在降级时向用户说明可能的语音质量变化

### ADR-005: 动画引擎选择 Manim Community

**状态**: 已接受
**日期**: 2026-03-22

#### 背景
问答 Agent 的核心竞争力是高质量数学/编程动画视频。ManimToVideoClaw 已基于 Manim Community 构建了完整的视频生成管线（分镜 → 代码生成 → 自修复 → 渲染 → TTS → 合成）。

#### 决策
选择 **Manim Community** 作为动画引擎：
- 达到 3Blue1Brown 级别的数学动画渲染质量
- ManimToVideoClaw 已验证完整管线可行性
- 配合 AI 代码生成 + 自修复机制（AI 修复 + AST 修复 + 渲染错误修复）提高成功率
- 渲染成功率 SLO ≥ 80%，含自修复后成功

#### 后果
- **正面**：动画质量远超传统录屏；ManimToVideoClaw 代码可直接复用
- **负面**：Manim 渲染耗时较长（影响端到端延迟）；代码生成存在失败率
- **缓解**：自修复机制（AI + AST + 渲染）提升成功率；异步任务 + 进度轮询优化用户体验

### ADR-006: 共享 Redis 实现跨后端认证

**状态**: 已接受
**日期**: 2026-03-21

#### 背景
双后端架构下，FastAPI 需要验证用户身份。传统方案是 FastAPI 通过 HTTP 调用 RuoYi 的认证接口，但这引入了网络延迟和单点故障。

#### 决策
采用**共享 Redis 方案**实现跨后端 JWT Token 验证：
- RuoYi 登录时将 JWT Token 存入 Redis
- FastAPI 直接从 Redis 验证 Token，无需 HTTP 调用 RuoYi
- 两端共享同一个 MySQL 数据库，FastAPI 只读用户表
- Nginx 统一入口按路径分发，前端无感知后端差异

#### 后果
- **正面**：消除认证环节的 HTTP 开销；降低系统耦合度；Redis 验证速度极快
- **负面**：需确保 JWT Token 格式两端一致；Redis 成为认证链路的关键依赖
- **缓解**：Redis 本身高可用（可配置哨兵/集群）；Token 格式文档化并集成测试覆盖
