# API 设计规范

> **状态**: 生效
> **负责人**: 架构负责人 / 前后端 Owner
> **最后更新**: 2026-04-04

---

## API 设计原则

1. 机器可读优先：接口一旦进入“契约冻结”，必须同时提供 OpenAPI、schema 与示例 payload，不允许只靠自然语言描述。
2. 契约先行：前后端并行时，以 `_bmad-output/` 中冻结的 Story、`contracts/` 资产和 FastAPI `/openapi.json` 为准，不以聊天记录为准。
3. 单一传输层：前端继续以 `packages/student-web/src/services/api/client.ts` 作为唯一共享 HTTP 传输层，不引入第二套长期并存的 request stack。
4. 生成优先于手抄：前端类型、schema 校验或 adapter scaffold 可以由 OpenAPI / JSON schema 自动生成，但生成产物必须复用共享 client，不得旁路认证、错误归一化和重试逻辑。

## URL 规范

- 功能接口统一挂在 `/api/v1/**`。
- 资源名使用小写短横线或明确业务名词，避免中英混排和语义缩写。
- 长任务相关恢复、事件与状态接口遵循任务域命名，例如 `/api/v1/tasks/{taskId}/events`、`/api/v1/tasks/{taskId}/snapshot`。

## 请求与响应格式

- HTTP 接口默认输出统一响应 envelope。
- 任务域运行时 payload 以 `contracts/tasks/*.schema.json` 为准，字段统一使用 camelCase，例如 `taskId`、`taskType`、`requestId`、`errorCode`。
- 失败响应必须同时提供错误码、示例失败 payload 与追踪字段，不允许只给成功示例。

## 错误码定义

- 错误码属于稳定契约资产，具体规则以 `contracts/_shared/error-code-rule.md` 与各业务域错误码字典为准。
- 任务 / SSE / Provider 相关错误码必须能映射到前端稳定文案与动作，前端禁止解析自由文本猜业务语义。

## 认证与鉴权

- 认证真值来自 RuoYi，学生端消费共享 client 时必须沿用既有鉴权、加密和失败归一化逻辑。
- 自动生成的接口类型或 adapter scaffold 不得绕过共享 client 中的鉴权、加密请求头和统一错误处理。

## API 文档管理

- FastAPI 必须稳定暴露 `/openapi.json` 与 `/docs`。
- 任一 Story 被声明为“契约冻结”时，至少应具备：
  - OpenAPI 路由定义或可追溯到路由的响应模型
  - `contracts/` 下的 schema / 错误码 / 状态枚举资产
  - request / response / error 示例 payload
- 前端自动生成的推荐边界：
  - 生成 TypeScript 类型、schema 校验或 adapter scaffold
  - 继续复用 `packages/student-web/src/services/api/client.ts`
  - 不生成第二套独立的认证、重试、错误处理和埋点实现
- Epic 2 任务域的机器可读资产由两部分组成：
  - `contracts/tasks/*.schema.json`：SSE、snapshot、runtime payload
  - `FastAPI /openapi.json`：HTTP 任务接口与示例
