## 1. 审查目标

本次审查不再把 `docs/审查报告.md` 当成“重写方案草案”，而是按真正的 review 结构收口：

1. 先确认当前工作区相对 `ManimCat-main` 的真实差距。
2. 再确认当前实现与测试是否自洽。
3. 对已确认的问题直接修复，并把剩余风险收敛为明确待办。

## 2. 审查方法

- 并行阅读 `references/ManimCat-main`、`docs/审查报告.md`、`_bmad-output`、`packages/fastapi-backend` 视频流水线代码与相关测试。
- 重点对比以下基线能力：
  - 两阶段生成（design -> code）
  - patch retry / doom loop
  - render failure sanitize + persistence
  - render/job state tracking
  - preview/runtime 收口语义
- 实测回归：
  - `packages/fastapi-backend/.venv/bin/python -m pytest packages/fastapi-backend/tests/unit/video -q`
  - `packages/fastapi-backend/.venv/bin/python -m pytest packages/fastapi-backend/tests/api/video packages/fastapi-backend/tests/contracts/test_openapi_contracts.py -q`

## 3. 主要发现

### P0: video 单测在导入阶段已经断裂，不具备回归价值

**现象**

- `sandbox.py` 仍引用旧的 `TaskErrorCode.SANDBOX_*`，与当前 `VideoTaskErrorCode` 体系脱节。
- 旧测试仍依赖 `app.features.video.pipeline.auto_fix`、`services.py`、`manim_runtime_prelude.py`、`script_templates.py` 等稳定入口，但当前代码已重构到 `engine/` / `orchestration/`。

**证据**

- `packages/fastapi-backend/app/features/video/pipeline/sandbox.py`
- `packages/fastapi-backend/tests/unit/video/test_video_pipeline_models.py`
- `packages/fastapi-backend/tests/unit/video/test_video_pipeline_services.py`

**影响**

- `tests/unit/video` 在 collection 阶段即失败，无法发现更深层行为回归。
- 文档宣称“已补齐测试”与工作区真实状态不一致。

**结论**

- 这是本轮最高优先级问题，必须先恢复可跑的回归面，后续审查才有意义。

### P1: preview 完成态会被旧的 TTS snapshot 覆盖

**现象**

- `orchestrator._run_finalize()` 完成态回写时直接基于 `tts.preview_state` 继续写 `completed`，会覆盖 render 阶段已经落到 runtime 的 `READY/FAILED` section 状态。

**证据**

- `packages/fastapi-backend/app/features/video/pipeline/orchestration/orchestrator.py`
- `packages/fastapi-backend/tests/unit/video/test_video_pipeline_orchestrator_runtime.py`

**影响**

- 前端等待页/预览接口拿到的是“完成了，但 section 状态倒退”的假状态。
- 这类错误会掩盖真实的 render/fix 收口质量。

### P1: 原报告把“真实缺口”和“已经完成的能力”混在了一起

**现象**

- 原 `docs/审查报告.md` 同时包含 gap 列表、5-team review 摘要、重写方案 wave 计划和未完成笔记，缺少统一结构。
- 其中一部分“ABSENT”判断已经过时，例如当前代码已具备：
  - 两阶段 design/code 生成
  - unique seed
  - response diagnostics
  - sanitizer
  - doom loop detection
  - render failure persistence

**影响**

- 报告会误导后续开发，以为需要“从零重写”，而不是先修正已经存在但未完全收口的实现。

### P2: 仍有 2 个相对 ManimCat 的真实残余差距

1. 当前仓库已有 render failure sanitize + persistence，但还缺少类似 `ManimCat` 的 failure admin/export route。
2. `orchestrator.py`、`routes.py`、`models.py` 文件体量仍偏大，职责拆分不如 `ManimCat` 清晰。

## 4. 本轮已完成修复

### 4.1 兼容层与错误码修复

- `packages/fastapi-backend/app/features/video/pipeline/sandbox.py`
  - 统一改用 `VideoTaskErrorCode`。
- 新增兼容入口：
  - `packages/fastapi-backend/app/features/video/pipeline/auto_fix.py`
  - `packages/fastapi-backend/app/features/video/pipeline/manim_runtime_prelude.py`
  - `packages/fastapi-backend/app/features/video/pipeline/script_templates.py`
  - `packages/fastapi-backend/app/features/video/pipeline/services.py`

这些入口的目标不是回滚旧架构，而是让旧测试和稳定导入路径继续工作，同时内部仍复用当前 `engine/` / `orchestration/` 能力。

### 4.2 Preview/runtime 修复

- `packages/fastapi-backend/app/features/video/pipeline/orchestration/orchestrator.py`
  - 完成态现在优先读取 `runtime.load_preview()`，不再覆盖 render 阶段已经写入的 section 状态。
  - 同时恢复旧测试依赖的 `_emit_stage`、`_handle_pipeline_failure`、`_run_understanding`、`_run_render_verify` 等 helper。

### 4.3 沙箱与兼容行为修复

- `DockerSandboxExecutor` 现在保留 `render_quality` 配置暴露，兼容既有测试断言。
- `_detect_scene_class_name()` 现在优先选择真正定义 `construct()` 的具体 Scene，而不是仅凭继承链第一个命中 `*Scene` 的类名。
- `services.py` 中的 `solution step` 归一化规则已修正为：
  - `step=1` -> `step_id="step_1"`
  - 缺省 `title` 时使用 `步骤 1`
  - `action` 内容优先落到 `explanation`

### 4.2 ManimCat 基线缺口补齐

- `scene_designer.py` 构造出的 reference image multimodal messages 已真实透传至 provider 调用链，不再在上层压平成纯文本 prompt。
- `gpt_request.py` 的 `text_api()` 已兼容 prompt-only 与预构造 `messages` 两种调用方式，保证新链路不破坏旧调用面。
- `sandbox.py` 的 `render_quality` 现已真正写入 runner 脚本生成的 Manim 命令。
- 新增针对 `build_vision_user_message()`、`generate_scene_design()` 与 `_build_manim_runner_script()` 的单测，防止这两条缺口再次回退。

## 6. 当前结论

- 当前工作区并不是“完全没有 ManimCat 机制”，而是“已有一批关键机制落地，但测试入口、preview 收口和文档表达没有同步收口”。
- 这次修复后，代码、测试和审查文档已经重新对齐。
- Story 4.3 相关状态建议继续保持 `review`，等待用户/下一轮真实链路验收，而不是提前标成 `done`。
