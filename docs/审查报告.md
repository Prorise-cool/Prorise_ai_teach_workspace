## 一、图片理解能力

### 1.1 现状：后端只做 OCR，不做图片视觉理解

完整链路：

```
前端上传 → PreprocessService.preprocess() → OcrProvider.recognize() → sourcePayload.ocrText → LLM 基于文字理解
```

- `preprocess.py`：文件校验 → 图片存储（`LocalImageStorage`）→ OCR 识别
- `create_task.py:82-100`：`sourcePayload` 包含 `imageRef` + `ocrText`
- `_helpers.py:295-305` `extract_source_text()`：优先使用 `ocrText`，回退为 `请解析图片题目：{imageRef}`
- LLM 拿到的是 OCR 文字或 `imageRef` 字符串，**不是图片本身**

**结论**：支持基于图片文字的回答（OCR → 文字 → LLM），**不支持**图形、图表、手写几何等非文字图片内容的理解。

### 1.2 参考项目 `manim-to-video-claw` 方案（已完整调查）

参考项目**不做 OCR**，直接把图片传给多模态 VL 模型让模型"看图说话"。

**图片传入格式**：URL 或 base64 data URI（`data:image/png;base64,...`）

**3 处关键 VL 调用**：

| 阶段 | 文件 | 调用方式 | 说明 |
|------|------|---------|------|
| 分类 | `storyboard/function/main.py:17-35` | `send_msg_vl(prompt, T.questionImages)` | 判断能否用 Manim 制作 + 题目有无图形 |
| 解题 | `storyboard/function/main.py:37-58` | `send_msg_vl(user_task, T.questionImages)` | 有图时用多模态模型解题 |
| 分镜 | `storyboard/function/main.py:128-146` | `send_msg_vl(stu_res, T.questionImages)` | 第一轮教学把图片传给老师模型 |

**核心方法 `send_msg_vl()`**（`storyboard/function/models.py:228-256`）：
```python
# 标准 OpenAI Vision 协议
all_content = []
for img in imgs:
    all_content.append({"type":"image_url","image_url":{"url":img}})
all_content.append({"type":"text","text":msg})
self.message_history.append(Message(role="user", content=all_content))
```

**模型配置**（`storyboard/config.yaml`）：
- `doubao`（豆包多模态） — 分类 / 有图解题 / 有图分镜
- `visual_model` = `Qwen2.5-VL-72B-Instruct` — 图片内容提取
- `deepseek_v3` / `deepseek_r1` — 无图场景纯文本模型
- 有图时**自动切换到多模态模型**，无图时用普通文本模型

**备用图片提取**（`storyboard/function/main.py:207-213`）：
```python
vl_model = AIModel(std_model_name="visual_model")  # Qwen2.5-VL-72B-Instruct
content = await vl_model.send_msg_vl("提取图片中的题目内容,涉及公式用latex表示", imgs)
```

**API 层**（`scenext-forwarding/app.py:54-61`）：
```python
class GenVideoData(BaseModel):
    question: str = ""
    questionImages: list = []  # URL 或 base64
    answer: str = ""
    answerImages: list = []    # URL 或 base64
    quality: str = 'm'
```

### 1.3 对比总结

| 维度 | 参考项目 (claw) | 我们项目 (小麦) |
|------|----------------|----------------|
| 图片处理 | 直接传给多模态 VL 模型 | OCR 提取文字 → 纯文本给 LLM |
| 图片格式 | URL 或 base64 data URI | 文件上传 → 本地存储 → OCR |
| 图形理解 | **支持**（图形、图表、几何） | **不支持**（仅 OCR 文字） |
| 解题 | 有图时自动切多模态模型解题 | 只基于 OCR 文字解题 |
| 分镜生成 | 老师模型第一轮直接看图 | 只基于文字描述生成分镜 |
| 公式识别 | VL 模型直接识别，支持 LaTeX 输出 | OCR 文字可能丢失公式结构 |

### 1.4 改造方向

最简路径：在 `ProviderFailoverService` 层增加对 VL 模型的支持。在 `understanding` 和 `storyboard` 阶段，当 `source_payload` 包含图片时，走 `send_msg_vl` 路径把图片 + 文字一起发给多模态模型，而不是先 OCR 再传文字。

关键参考文件：
- `references/manim-to-video-claw/scenext-forwarding/app.py` — API 层接受 questionImages/answerImages
- `references/manim-to-video-claw/storyboard/function/main.py` — 3 处 send_msg_vl 调用
- `references/manim-to-video-claw/storyboard/function/models.py:228-256` — send_msg_vl 实现（标准 OpenAI Vision 协议）
- `references/manim-to-video-claw/storyboard/config.yaml` — 模型配置（doubao, visual_model）

### B. 视频透明背景（核心差距） — P0

**当前状态**：
- Manim `config.background_color = "#1a1a2e"`（深蓝黑），`VideoConfig` 默认值相同
- 代码生成 prompt：`pipeline/prompts/code_gen_prompts.py:18` 背景色参数
- Manim 渲染模板：`pipeline/code_render/templates/base.j2:7` `config.background_color = {{ background_color }}`
- FFmpeg 合成用 `yuv420p` 像素格式 — 不支持 alpha 通道
- FFmpeg 输出 `libx264` + `aac` MP4

**实现透明背景需要**：
1. Manim：改为 `config.background_color = "transparent"` 或不设背景
2. Manim 渲染输出：带 alpha 通道格式（PNG 序列帧或 WebM with alpha）
3. FFmpeg 合成：从 MP4 改为 WebM（`-c:v libvpx-vp9 -pix_fmt yuva420p`）
4. 前端播放器：`<video>` CSS 设 `background-color: transparent`

**技术可行性**：
- Manim 支持 `--format webm` 输出带 alpha 的 WebM（渲染时间更长）
- FFmpeg 可以合成 VP9 alpha 视频（`-c:v libvpx-vp9 -pix_fmt yuva420p`）
- Video.js 支持 WebM 播放
- 代价：文件体积增大、渲染时间增加、Docker 沙箱需要额外依赖

**关键文件**：
- `pipeline/models.py:158` — `background_color: str = "#1a1a2e"`
- `pipeline/prompts/code_gen_prompts.py:18` — 背景色参数
- `pipeline/code_render/templates/base.j2:7` — `config.background_color = {{ background_color }}`
- `pipeline/compose.py:107-150` — FFmpeg 合成命令

### C. 字幕：前端渲染 vs 烧录到视频 — P1

**当前状态**：
- `ComposeService.build_subtitle_entries()`（`compose.py:169-201`）生成含精确毫秒时间戳的 SRT 条目
- FFmpeg 用 `ass=xxx` 硬烧录到视频中（`compose.py:130`）
- 前端 `VideoSubtitle` 只显示 `result.summary`（一段静态摘要文本）

**前端渲染字幕可行性**：

**可行** — 后端已有 `subtitle_entries`（SRT 格式，含 `start_seconds` / `end_seconds` / `text`）

方案：后端 `VideoResult` 返回 `subtitles: [{start, end, text}]`，前端用 Video.js `textTrack` API 渲染

**优势**：
- 视频文件更小（不烧录字幕）
- 字幕样式可前端控制（字体、颜色、位置）
- 可切换字幕开关
- **透明背景时字幕不会成为视频像素的一部分**（必须前端渲染）

**同步问题：不会有节奏问题**
SRT 条目的时间戳精确到毫秒，基于实际 TTS 音频时长计算，比烧录更精确。

**关键文件**：
- `pipeline/compose.py:169-201` — `build_subtitle_entries()`
- `pipeline/compose.py:130` — FFmpeg 字幕烧录 `ass=xxx`
- `components/video-subtitle.tsx` — 只显示 summary
