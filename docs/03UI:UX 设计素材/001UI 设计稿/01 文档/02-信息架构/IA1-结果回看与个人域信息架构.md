## 1. 范围说明

### 1.1 本文覆盖范围

- 学习结果聚合域（Learning Result Aggregation Domain）：
  - `LearningCenter.tsx` 页面组件：学习结果聚合入口。
  - 历史视图：历史记录列表与结果重开入口。
  - 收藏视图：收藏列表与结果回看入口。
- 个人域（Personal Domain）：
  - `Profile.tsx` 页面组件（候选路径：`/profile`）：个人资料查看与编辑入口。
  - 设置视图：平台相关选项、账号安全与偏好设置入口。
- 关键入口关系：结果重开、收藏（收藏/取消收藏）、删除确认（历史删除）等动作的入口与一致性约束。
- 数据宿主与访问边界：FastAPI 与 RuoYi 的职责边界、长期数据宿主说明、COS/Redis 的边界提示。

- 历史 / 收藏视图打开后的“结果详情页 / 播放器页 / 课堂结果页”等结果展示路由的内部结构与 UI 规格，属于主链路 IA / 页面规格任务承接；本文只定义从回看域到结果页的跳转关系与动作入口约束。

## 2. 个人域与回看域目标

### 2.1 个人域目标

个人域的目标是提供“用户身份与偏好”相关的稳定入口，支撑：
- 用户基础资料维护（昵称、头像、学校等），并满足基本校验与可用性要求（见 `epics.md` 中 FR3）。
- 个人设置与偏好入口的可发现性（设置本身可分期交付，但入口与路由需稳定）。
- 与学习沉淀闭环一致：个人域只承接“个人资料与平台设置”，不承接历史、收藏等学习结果聚合，也不承接任何“生成中的任务状态展示”。

### 2.2 学习结果聚合域目标

学习结果聚合域的目标是支撑 P0 的“结果可回看闭环”和后续扩展：
- 历史记录：任务完成后写入历史，按时间倒序展示，支持重新打开对应结果（`epics.md` FR24）。
- 收藏：结果页支持收藏/取消收藏，并在学习中心聚合域可查看收藏状态（`epics.md` FR25）。
- 删除：支持删除单条历史记录，删除前必须确认，删除后列表即时更新（`epics.md` FR26）。

## 3. 路由与导航关系（Route + Nav）

### 3.1 页面组件与候选承载（必须覆盖）

| 域 | 页面 / 视图 | 候选路径 | 核心职责 | 与主链路关系 |
|---|---|---|---|---|
| 学习结果聚合域 | `LearningCenter.tsx` 页面组件 | `/learning` | 作为学习结果聚合入口，承接历史、收藏、学习沉淀与后续推荐 | 不直接参与长任务执行，但承接结果回看与学习沉淀 |
| 学习结果聚合域 | 历史视图 | `/history` | 查看历史条目、打开结果、重开、删除 | 通过“打开结果”跳转到结果页；通过“重开”进入新任务创建 |
| 学习结果聚合域 | 收藏视图 | `/favorites` | 查看收藏条目、打开结果、取消收藏 | 通过“打开结果”跳转到结果页 |

### 3.2 全局导航（入口建议）

基于事实源中“正式页面后置、但 IA 先行”的原则，本文只冻结“入口位置”的约束，不冻结具体 UI 组件形态：
- 全局右上角用户入口（Avatar/用户名区域）应提供到个人域的快速入口：
  - 个人资料（`/profile`）
  - 平台设置（`/settings`）
- 学习中心聚合页入口见第 4 节；历史 / 收藏作为学习中心聚合内容承接，不再归到个人中心。

## 4. 学习中心聚合页目标位置

### 4.1 角色定位（与回看域的区别）

学习中心聚合页（Learning Hub）的定位是“学习沉淀的聚合视图”，聚合来源至少包含：
- 历史记录（结果回看）
- 收藏结果
- 学习辅助产生的测验/错题/路径建议等沉淀

历史 / 收藏视图定位为“学习中心中的结果管理视图”；学习中心定位为“学习结果与学习沉淀的聚合入口”。两者是包含关系，不再与个人中心混写。

### 4.2 目标位置（路由与入口约束）

- 目标位置：学习中心统一使用 `/learning` 作为学习结果聚合入口。
- 入口约束：
  - 必须从全局导航可达（例如：主导航项或顶部入口中的一级项）。
  - 允许从结果页在“完成后下一步”区域提供跳转（例如“去学习中心查看沉淀”），但不将结果页视为学习中心的唯一入口。

> 提醒：本轮已冻结“学习中心承担结果聚合、个人域承担资料与设置”的职责边界；后续仅在页面层级与导航样式上迭代，不再改动职责归属。

## 5. 关键入口关系（重开 / 收藏 / 删除确认）

### 5.1 结果重开（Re-open / Re-run）

目标：在“结果可回看闭环”中提供清晰的“再次生成”入口，但不把它混同为“打开历史结果”。

- 在历史视图的每条历史记录条目上，必须同时支持两类动作（入口可以是按钮或更多菜单，但语义必须区分）：
  - **打开结果**：跳转到对应结果页（同一条历史记录的结果引用）。
  - **重开（重新生成）**：以该条记录的输入参数为预填，创建一条新的任务（进入主链路的创建/提交步骤，而不是进入等待页的内部状态展示）。
- 重开动作的确认策略：
  - 默认不需要二次确认（因为它是“创建新任务”，不是破坏性操作）。
  - 若产品需要控制成本或避免误触，再引入轻量确认（记录在“冲突 / 待确认点”）。

### 5.2 收藏（Favorite / Unfavorite）

目标：收藏是一种“结果管理行为”，入口应在结果页与回看域保持一致，并且收藏状态可回写到收藏视图。

- 收藏动作的入口关系（必须）：
  - 结果页提供“收藏/取消收藏”入口。
  - 历史视图列表条目提供“收藏/取消收藏”快捷入口（可选，但推荐；若后续决定只在结果页提供，也需要保持一致性并记录在冲突点）。
  - 收藏视图列表条目必须提供“取消收藏”入口。
- 收藏动作不需要二次确认（非破坏性，且可逆）。

### 5.3 删除确认（Delete with Confirmation）

目标：删除属于破坏性操作，仅针对“历史记录条目”进行删除，并且删除前必须确认（`epics.md` FR26）。

- 删除入口关系（必须）：
  - 历史视图条目提供“删除”入口。
  - 删除前必须弹出确认（Modal/Popover 等均可），确认文案必须面向用户且不暴露技术细节。
  - 删除后列表即时更新。
- 删除的影响范围（需要明确的默认结论）：
  - 默认结论（建议）：删除仅删除“用户侧历史记录引用”，不承诺同步删除 COS 文件产物；如需要“彻底删除”策略，属于数据合规与存储策略议题，必须单独评审。

> 关键约束：删除确认属于“非路由交互”，但它不属于 FastAPI 长任务等待流；它是回看域的管理交互，避免把确认弹窗复用成等待流的错误态弹窗，从而造成语义混淆。

## 6. 数据宿主与访问边界（RuoYi / FastAPI / Redis / COS）

### 6.1 双后端职责边界（必须一致）

依据冻结输入与架构主文档，双后端职责边界结论如下：
- FastAPI：负责 AI 功能执行与编排、异步任务协调、视频与课堂生成、Provider 调度、SSE 进度推送、运行时状态管理。
- RuoYi：负责用户/角色/权限，以及小麦长期业务数据持久化（学习记录、收藏、任务元数据、问答日志等），并承接查询、审计、导出与管理宿主能力。

### 6.2 长期数据宿主（RuoYi）说明（必须写入）

个人域与回看域所依赖的“长期业务数据”必须以 RuoYi 为宿主：
- 用户资料（profile）相关字段（昵称、头像、学校等）。
- 历史记录（history）条目、与结果引用的元数据（任务类型、创建时间、输入摘要、结果引用等）。
- 收藏（favorites）条目与收藏状态。
- 删除行为的最终一致性落点（删除操作应以 RuoYi 数据为准）。

### 6.3 Redis 与 COS 的边界提示（避免混写）

- Redis：仅用于运行时状态、SSE 事件缓存、短期上下文、Provider 健康状态等短期数据，必须设置 TTL；不作为个人域/回看域的权威数据源。
- COS：文件产物（例如视频）统一进入 COS；个人域/回看域展示时应通过“结果引用”访问，而不是把 COS 当作业务列表数据源。

## 7. 优先级分层（结构稿 vs 高保真）

> 依据事实源中“后端能力与契约先行、正式前端页面后置”的节奏，本文把 IA 产出拆成两个层级：先补齐路由与入口的结构定义，再在成品图冻结后进入高保真页面规格。

### 7.1 需要优先出结构稿（优先级高）

以下页面建议先出“结构稿 / 信息结构稿”（可对应后续线框与交互说明），用于支撑后端契约与数据模型闭环：
- `LearningCenter.tsx` 页面组件：聚合结构、入口分区、历史 / 收藏 / 路径建议模块骨架。
- 历史视图：列表信息结构、条目动作（打开结果 / 重开 / 删除确认 / 收藏）与空态 / 错态骨架。
- `Profile.tsx` 页面组件：资料字段结构、编辑入口与校验边界骨架。
- 收藏视图：列表信息结构、打开结果与取消收藏骨架。
- 设置视图：设置聚合信息结构（先定义分组与入口，不强求全部子项落地）。

### 7.2 可以延后高保真（依赖成品图冻结）

以下内容不阻断 P0 的“可回看闭环”，建议在成品图冻结与接口契约稳定后再进入高保真规格：
- 设置视图的细分子页（若需要），以及与账号安全相关的完整高保真交互（属于 P1/P2 迭代承接）。
- 收藏视图的高级管理能力（例如文件夹 / 批量操作等），依据 PRD 的 P2 后置范围可延后。
- 学习中心聚合页的复杂推荐、路径规划与增强模块的高保真视觉（属于 P1/P2 扩展，按产品节奏推进）。

## 8. 冲突 / 待确认点

1. 历史 / 收藏 / 设置视图在交互实现上采用一级路由、二级路由还是内部 Tab，仍可继续迭代；但职责归属已经固定，不再改变。
2. “重开（重新生成）”是否需要二次确认：本文默认不需要确认；若后续成本或误触风险评估要求确认，需要统一到交互规范中。
3. 删除的语义边界：本文建议“删除仅删除历史记录引用，不承诺同步删除 COS 文件产物”；若产品或合规要求“彻底删除”，需要数据合规与存储策略评审并明确实现边界。
4. 历史视图条目是否需要提供“收藏 / 取消收藏”快捷入口：本文建议提供以提升效率，但若后续决定仅在结果页提供，也可接受；需在交互一致性层面定稿。
