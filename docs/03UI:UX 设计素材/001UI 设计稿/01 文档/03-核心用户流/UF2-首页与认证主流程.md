# UF2-首页与认证主流程

- 文档编号：UF2
- 范围：`/` 首页双入口 + 独立登录页（建议：`/login`）+ 受保护路由的未登录拦截 + 401/403 处理 + 登出清理与回退
- 冻结事实源（仅用于“事实”表述）：`prd.md`、`ux-design-specification.md`、`architecture.md`、`epics.md`
- 结构对齐参考（非事实源）：`IN0-设计输入包冻结版.md`、`IA1-全局导航与路由骨架.md`、`IA1-学习主链路信息架构.md`、`IA1-结果回看与个人域信息架构.md`

---

## 1. 角色与前置条件

### 1.1 角色

- 访客（未登录）。
- 已登录用户（持有有效 Token，且后端 Redis 在线态有效）。
- 无权限用户（已登录，但访问受保护资源被判定为无权限，返回 403）。

### 1.2 前置条件与约束

- 公共路由：`/` 与独立登录页（建议：`/login`），可访客访问。
- 受保护路由：除公共路由外，业务路由默认受保护（例如 `/classroom/*`、`/video/*`、`/knowledge`，以及 LearningCenter / Profile 页面组件与其内部视图等）。
- `architecture.md` §10.3.1 / §10.3.3 已冻结鉴权分流：`401 -> 清除 Token，跳转登录`；`403 -> 提示无权限 / 登录失效`。
- 当前轮冻结：主动登录入口与未登录拦截统一进入独立登录页，不再以 Dialog 作为主承载。

## 2. 主流程（至少 1 条闭环）

### PF-1：访客从首页进入双入口并完成登录后回到原上下文（闭环）

**目标**：访客进入 `/`，选择入口（课堂 / 视频），进入统一登录流程后回到原目标页面（含参数），继续主链路。

**步骤**：

1. 访客访问 `/`，首页完成首屏渲染，展示“双入口”：
   - 入口 A：初学场景（建议跳转 `/classroom/input`）。
   - 入口 B：问答场景（建议跳转 `/video/input`）。
2. 访客点击入口 A 或入口 B。
3. 系统判定目标路由为受保护路由，且当前未登录：
   - 记录 `returnTo = 原目标路由`。
   - 导航到独立登录页（建议：`/login`）。
4. 访客完成登录或注册提交。
5. 提交成功：
   - 导航到步骤 2 中的原目标路由（例如 `/classroom/input` 或 `/video/input`）。
6. 用户在入口页开始后续学习链路（本 UF 只覆盖到“成功进入入口页”的认证收口）。

**主流程收口条件**：

- 用户已处于“已登录”状态，且已回到原目标上下文（入口页路由）。

## 3. 异常流程（必须覆盖）

### EF-1：未登录访问受保护路由（含深链）

**触发**：访客直接访问任一受保护路由（例如直接打开 `/video/input` 或 `/classroom/:id` 链接）。

**处理**：

1. 保留当前路由意图（`returnTo = 当前 URL`）。
2. 进入独立登录页；若已触发 401，则遵循 `architecture.md` 的固定处理：清除 Token 并跳转登录页。
3. 登录成功后返回 `returnTo`。

**结果**：

- 若认证成功：执行“恢复流程 RF-1”。
- 若用户离开登录流：留在当前页面外壳（或回退到 `/`），并明确当前内容不可用（具体策略见“冲突 / 待确认点”）。

### EF-2：登录失败（账号密码错误、校验失败、网络失败）

**触发**：用户在认证对话框提交登录/注册，但接口返回失败或网络异常。

**处理**：

1. 保持登录页当前表单状态，不丢失已输入字段（除密码可按安全策略决定是否清空）。
2. 明确展示错误信息（用户可理解的文案，不暴露敏感信息与调试细节）。
3. 提供明确动作：
   - 继续重试（主按钮）。
   - 切换登录/注册（次动作）。
   - 关闭对话框返回原页面（如果允许）。

**结果**：

- 用户可再次提交；成功则进入“恢复流程 RF-1”。

### EF-3：Token 失效或非法导致 401（登录失效）

**触发**：已登录用户在受保护页面发起请求，或页面初始化请求时收到 401。

**处理（原则）**：

1. 立即将当前会话标记为“未认证态”（至少：本地 Token 视为无效）。
2. 保留用户当前上下文（`returnTo = 当前 URL`）。
3. 清除 Token，并跳转独立登录页。

**结果**：

- 重新登录成功：执行“恢复流程 RF-2”。
- 用户放弃登录或离开登录流：执行“恢复流程 RF-3”（回退到可访问范围并展示清理后的状态）。

### EF-4：403 无权限（已登录但无权限）

**触发**：用户访问受保护资源返回 403。

**处理**：

1. 进入 403 对应的“无权限 / 登录失效”提示通道（与 `VITE_SERVICE_MODAL_LOGOUT_CODES=403` 对齐）。
2. 在当前页面给出“无权限”提示与可行动出口：
   - 返回首页 `/`。
   - 返回上一级（浏览器回退）。
  - 重新认证 / 切换账号（走独立登录页）。

## 4. 恢复流程（Recovery）

### RF-1：认证成功后返回原上下文（未登录拦截场景）

**输入**：`returnTo`（用户原本要去的受保护 URL）。

**步骤**：

1. 认证成功后恢复前端认证态。
2. 导航到 `returnTo`（含 path、query、hash）。
3. 触发页面必要的数据刷新（例如用户信息、导航上“登录/注册”区域切换为用户入口）。

### RF-2：401 重新登录后的上下文恢复

**目标**：用户在任务/输入/结果页面中途 Token 失效后，重新登录可回到原页面并继续。

**步骤**：

1. 记录 `returnTo`，跳转统一登录流。
2. 用户登录成功后回到 `returnTo`。
3. 页面按当前路由语义进行恢复：
   - 输入页：恢复输入框内容（若本地已暂存）。
   - generating 页：重连 SSE 或走 snapshot 恢复。
   - 结果页：重新拉取结果详情或从本地缓存恢复展示。

### RF-3：用户取消重新登录后的收口（401 场景）

**目标**：避免“无限弹窗”或“死循环跳转”。

**步骤（建议）**：

1. 用户离开登录流程或明确选择“暂不登录”。
2. 清理本地认证态（确保不会继续以旧 Token 请求）。
3. 将用户带回可访问范围（例如 `/`），并用可恢复提示解释“需要登录后继续访问刚才内容”。

## 5. 状态与反馈（UI Feedback）

### 5.1 首页（`/`）相关

- 登录前：顶部入口显示 `登录` / `注册`，点击后进入独立登录页。
- 登录后：顶部入口显示用户入口（头像/昵称等），包含登出入口（具体承载在个人域或菜单内，见 IA 对齐）。

### 5.2 独立登录页（Login Page）

- 路由承载：建议使用 `/login`。
- 主动触发：首页的“登录 / 注册”按钮跳转该页。
- 提交中：按钮 loading，防重复提交。
- 成功：恢复用户态，并按 `returnTo` 回跳。
- 失败：错误提示 + 可行动按钮（重试 / 切换登录注册 / 返回首页）。

### 5.3 401 / 403 通用反馈

- 401：提示“登录已失效 / 需要重新登录”，清除 Token 并跳转独立登录页；同时保证不会出现无限跳转或死循环。
- 403：提示“无权限 / 登录失效”，通过统一提示通道提供“返回首页 / 返回上一级 / 重新认证”出口。

### 5.4 登出（Logout）

- 入口：建议位于用户菜单（个人域入口）中。
- 动作结果：
  - 本地认证态立即清理（至少包括 Token）。
  - 当前页面若为受保护路由，应回退到 `/` 或进入独立登录页；不再把“弹出认证对话框”写成唯一固定语义。
  - 清理后全局导航更新为未登录态。

## 6. 冲突 / 待确认点（必须列出）

> 规则：仅记录冲突与待确认，不在本文件中改写事实源。

1. **独立登录页的最终路径与注册承载方式**：
   - 当前轮按“建议 `/login`，注册作为同页状态切换”执行。
   - 待确认：是否固定为 `/login`；是否需要额外拆出 `/register`、`/forgot-password`。

2. **未登录拦截时，用户离开登录页后的落点**：
   - 选项：留在当前页面但置灰/遮罩；或强制回到 `/`；或保持当前 URL 但展示不可用状态。
   - 待确认：一致性策略需避免死循环与不可恢复状态。

3. **`returnTo` 的持久化时机与作用域**：
   - 是仅存于内存 / 路由状态，还是可跨刷新短暂保留。
   - 待确认：实现层怎样持久化 `returnTo`，才能兼顾回跳体验与安全边界。
