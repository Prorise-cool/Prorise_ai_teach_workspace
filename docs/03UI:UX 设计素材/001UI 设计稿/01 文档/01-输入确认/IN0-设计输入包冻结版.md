## 1. 产品目标（冻结）

面向中国高职教育场景，小麦是一款 AIGC 原生教学平台，通过“双入口学习体验 + 学习辅助 + 结果回看闭环”，实现可稳定演示、可持续扩展的 MVP / 公测版。

本版本交付必须满足的目标要点：
- 用户可登录并进入双入口主链路。
- 视频生成主链路稳定打通，课堂生成主链路稳定打通。
- 知识问答与学习辅助具备可演示闭环。
- 长任务过程可见、断线可恢复、结果可回看。
- 长期业务数据有明确持久化归属，外部能力具备 Failover 与降级策略。

## 2. 版本范围（P0 / P1 / P2 冻结）

说明：以下是 UI/UX 设计输入侧的“范围冻结”，用于指导后续 IA、Flow、视觉与组件系统任务拆解；实现顺序仍需遵循“后端能力、接口契约与测试先行，正式前端页面后置”的节奏。

### 2.1 P0（必须）

- 认证与进入：登录态打通，未登录访问受保护页面有一致的引导与拦截语义。
- 首页双入口清晰：
  - 入口 1：初学场景（主题输入 → 课堂生成 → 幻灯片/测验/讨论片段）。
  - 入口 2：问答场景（题目文本或图片 → OCR → 视频生成 → 播放复习）。
- 长任务体验闭环：统一等待体验，阶段与进度可见；断线后可恢复到任务快照；完成后可回访。
- 回看与聚合：历史记录或学习中心能重新打开结果。
- 知识问答与学习辅助：至少达到“可演示、可闭环”的体验标准（不要求一次性覆盖所有后置能力）。

### 2.2 P1（应该）

- 个人中心基础能力：资料查看与基础入口。
- 历史记录展示与跳转体验优化。
- 老师风格切换：风格差异对生成结果可辨识（风格是数据人格预设，不是页面主题）。
- 学习中心内可查看学习路径建议与推荐内容，并可跳转到相关学习结果。

### 2.3 P2（可后置）

- 收藏夹增强管理。
- 中英文切换（i18n）。
- 多 Agent 深度讨论（更强交互形态）。
- 白板高级布局（复杂布局优化不阻断 P0 主链路）。

## 3. 目标用户与核心场景（冻结）

### 3.1 目标用户

- 高职学生：快速理解知识点与难题，典型场景为课后复习、作业难题、考试冲刺。
- 高职教师：快速生成教学视频与课堂内容，典型场景为备课、课堂资源制作、翻转课堂。
- 自学者：定向突破知识盲点，典型场景为补基础、专项练习、技能提升。
- 管理角色（预留）：查看记录、审计与运营数据，典型场景为后台查询、导出、统计。

### 3.2 核心场景（以“触发意图”组织）

- 初学：用户表达“我想学 XX” → 快速获得结构化课堂内容与可完成的小测 → 记录沉淀到学习中心。
- 解题：用户输入/拍照题目 → 生成单题讲解视频 → 播放与复习 → 记录沉淀到历史/学习中心。
- 追问与辅助：在结果或知识问答页继续提问、做小测、看路径建议与推荐，实现连续学习。

## 4. 双入口产品定位（冻结）

- 入口 1（初学场景）：面向“主题学习”，强调课堂式结构化内容（幻灯片、测验、讨论片段等）。
- 入口 2（问答场景）：面向“单题解决”，强调输入低门槛（打字/粘贴/拍照）与视频可播放复习。

UX 目标约束：入口选择必须“直觉化、低焦虑”，尽量通过智能推荐与轻量确认降低认知负担；不得要求用户理解 LLM/Manim/TTS 等技术概念。

## 5. 双后端边界与数据宿主原则（冻结）

### 5.1 双后端职责边界

- FastAPI：AI 功能执行与编排、异步任务协调、视频与课堂生成、Provider 调度、SSE 进度推送、运行时状态管理。
- RuoYi：用户/角色/权限，以及小麦长期业务数据持久化（学习记录、收藏、任务元数据、问答日志等），并承接查询、审计、导出与管理宿主能力。

### 5.2 数据宿主（Data Host）原则

- Redis：仅用于运行时状态、SSE 事件缓存、短期上下文、Provider 健康状态等短期数据，必须设置 TTL。
- 长期业务数据：必须进入 RuoYi/MySQL 或 COS（文件产物统一进入 COS）。
- FastAPI：不承担长期业务主库存储职责。

## 6. 统一任务状态、SSE 事件与关键错误态（冻结）

### 6.1 统一任务状态（前端一致性语义）

任务状态统一使用 5 态：
- `pending`：已受理未开始。
- `processing`：执行中，有阶段与进度。
- `completed`：成功完成，可进入结果回看。
- `failed`：失败完成，需要展示错误态与恢复动作。
- `cancelled`：任务被主动取消或系统中止，需要明确收口反馈。

> 设计侧要求：所有“视频/课堂/学习辅助”等长任务的等待体验必须复用同一套状态语义与组件语言，避免每条链路各自发明状态。

### 6.2 SSE 事件（前端消费契约）

SSE 事件类型冻结如下（用于统一等待页与断线恢复）：
- `connected`：连接建立。
- `progress`：任务进度更新（包含阶段、进度、提示信息）。
- `provider_switch`：Provider 切换（包含 from/to/reason）。
- `completed`：任务完成（包含结果引用）。
- `failed`：任务失败（包含错误信息）。
- `heartbeat`：心跳。
- `snapshot`：断线恢复快照（包含 status/progress/stage）。

### 6.3 关键错误态（用户可理解、可恢复）

统一错误处理类别（UI 必须有明确动作与文案，不暴露调试信息）：
- 网络错误：提示重试。
- 超时：提示检查网络与重试。
- 服务不可用：提示降级与稍后重试；必要时由系统触发 Provider 自动切换。
- 渲染/生成失败：展示失败原因的“可理解摘要”，提供重试、返回入口、查看历史等替代动作。

错误码集合用于工程实现与埋点，但设计文案必须面向用户；代表性错误码包含：
- `TIMEOUT`、`CANCELLED`、`EXTERNAL_SERVICE_UNAVAILABLE`、`EXTERNAL_SERVICE_TIMEOUT`
- `LLM_UNAVAILABLE`、`LLM_RATE_LIMITED`、`LLM_RESPONSE_INVALID`
- `MANIM_CODE_GENERATION_FAILED`、`MANIM_RENDER_FAILED`、`MANIM_SANDBOX_ERROR`
- `TTS_SYNTHESIS_FAILED`、`TTS_ALL_PROVIDERS_FAILED`
- `COS_UPLOAD_FAILED`

## 7. 页面与交互边界（冻结）

### 7.1 正式页面（Route 级交付）

“正式页面”指可对外验收的路由页面，必须以前置条件为门槛：
- 成品图已冻结。
- 关键状态（空态、错态、加载态、禁用态）与核心交互说明齐备。
- 对应后端接口契约稳定，不处于高频变更期。

同时冻结一个节奏原则：线框图可支撑后端 Story 与临时联调壳层启动，但不构成“正式页面已交付”的充分条件。

### 7.2 非路由交互（Non-Route Interactions）

非路由交互指不切换主路由、以弹窗/面板/浮层等方式完成的关键动作，本项目至少包含：
- 认证入口的登录/注册对话框。
- 输入区附近的老师风格触发器与风格面板（局部点缀，不做全局主题切换）。
- 上传/粘贴/拍照触发的输入辅助浮层与权限提示。
- 等待页中的错误提示浮层、重试确认、取消确认（如有）。

> 设计侧冻结要求：非路由交互必须服务于“零负担输入与等待透明”，避免把用户拉离主输入任务。

### 7.3 后置能力（Deferred Capabilities）

以下能力明确允许后置，不应阻断 P0 设计交付：
- 更完整的移动端体验（当前 UX 规格以桌面端 ≥1024px 为 MVP 目标）。
- 国际化（i18n）与无障碍增强（可在后续迭代细化）。
- 更复杂的白板布局策略与更深的多 Agent 讨论形态。

## 9. 已知缺口与待评审冲突点

### 9.1 已知缺口（不在 IN0 补齐，只做冻结声明）

- “正式页面”级别的成品图尚未在本任务中提供，IN0 仅冻结输入边界与契约语义；后续 Gate/评审任务需以成品图冻结作为页面交付前置。
- Manim 沙箱安全策略细节、监控告警指标定义、数据库迁移策略等属于架构侧建议补充项，不在本任务展开。

### 9.2 冲突 / 待确认点

- 老师风格选择的交互形态表述存在轻微差异：
  - 事实源一处强调“输入框中以下拉形式展开”（更偏控件形态）。
  - UX 规格强调“头像/胶囊触发器展开轻量面板，不打断输入”。
  - 结论：两者可兼容为“触发器 + 下拉面板”的统一形态；如需锁定具体组件形式，待成品图评审定稿。
