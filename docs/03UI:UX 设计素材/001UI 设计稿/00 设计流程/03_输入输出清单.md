# 输入输出清单（调研后设计阶段）

> 用法：按节点推进，做到“输入够用就开工，输出可被下游直接使用”。
> 起点默认：调研结论已完成（业务目标/用户结论/竞品结论已具备）。

## 16节点输入/输出对照（一步一行）

| 节点 | 最小输入（够用即可） | 最小输出（可执行即可） |
|---|---|---|
| 0. 输入确认（调研结论） | 调研摘要、业务目标、核心用户、范围边界 | 设计输入包v1（目标/范围/约束/里程碑） |
| 1. 定义信息架构（IA） | 设计输入包v1、功能清单、内容清单 | IA结构图（一级/二级结构+命名） |
| 2. 绘制核心用户流 | IA结构图、关键任务场景、业务规则 | 主流程+异常流程图（含关键状态） |
| Gate1 结构评审 | IA结构图、用户流程图 | Gate1结论（通过/退回+修改点） |
| 3. 产出关键页面清单 | Gate1通过结论、用户流程图 | 页面清单（核心页/支持页/状态页） |
| 4. 绘制低保真线框 | 页面清单、流程图、平台约束 | 低保真线框（核心页面全覆盖） |
| 5. 校验线框可用性 | 低保真线框、核心任务脚本 | 线框问题清单（阻塞/重要/建议） |
| 6. 制作风格板（视觉方向） | 线框v1、品牌基础、参考样式 | 风格板2-3套+推荐方向 |
| 7. 落地设计令牌（Design Tokens） | 确认风格方向、平台规范 | 令牌v1（颜色/字体/间距/圆角/阴影） |
| Gate2 视觉基线评审 | 风格板、设计令牌v1 | Gate2结论（通过/退回+统一修正项） |
| 8. 建立组件系统 | Gate2通过结论、令牌v1、线框v2 | 组件库v1（基础组件+状态+命名） |
| 9. 产出高保真页面 | 组件库v1、页面清单、文案素材 | 高保真页面集v1（覆盖主流程） |
| 10. 串联完整原型并走查 | 高保真页面集v1、交互规则 | 可点击原型v1+走查记录 |
| 11. 开发标注与资源导出 | 原型v1、组件库、切图规范 | 标注稿v1+资源包v1 |
| Gate3 交付评审 | 高保真稿、原型、标注、资源包 | Gate3结论（可开发/补充后开发） |
| 12. 版本归档与进入开发 | Gate3通过结论、命名规范 | 归档交付包v1（可追溯） |

## 常见缺失输入的替代方案（轻量）

| 缺失输入 | 轻量替代方案（当天可执行） | 风险提示 |
|---|---|---|
| 没有完整PRD | 用“1页需求卡”替代：目标、用户、核心任务、边界、上线时间 | 后续易出现范围漂移，需每周补齐 |
| 没有用户访谈材料 | 先用客服高频问题+运营反馈做“临时用户假设” | 假设必须在Gate1后尽快验证 |
| 没有品牌规范 | 先用风格板做“临时视觉基线”（主色/字体/圆角） | Gate2前必须得到品牌确认 |
| 没有技术约束说明 | 与前端开30分钟对齐会，先形成“可做/暂缓清单” | 高保真阶段可能返工 |
| 没有统一文案 | 用占位文案规范（短句、同语气、长度标注）先出稿 | 交付前需统一替换 |
| 没有组件库历史资产 | 从页面中反向抽取Top20高频组件先建v0.1 | 初期复用率不高属正常 |
