# Story 4.2: 题目理解与分镜生成服务

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 等待生成讲解视频的用户，
I want 系统先正确理解题目并生成可执行分镜，
so that 后续动画与讲解能够围绕清晰结构展开。

## Acceptance Criteria

1. 视频任务进入 `understanding` 阶段时，后端调用理解服务产出结构化理解结果，至少包含 `topicSummary`（题目摘要）、`knowledgePoints`（核心知识点列表）、`solutionSteps`（解题步骤结构），后续分镜服务能直接消费该结果而不是重新解析原始输入。
2. 分镜生成阶段基于理解结果构建视频叙事，产出可供 Manim 与 TTS 消费的 `Storyboard` 结构，包含有序 `scenes` 列表，每个 scene 至少有 `sceneId`、`title`、`narration`（旁白文本）、`visualDescription`（动画指令描述）、`durationHint`（建议时长秒数）。
3. 分镜结果总时长受目标时长约束（默认 120s，范围 90–180s），若 LLM 输出超出范围则截断或压缩到合理边界，而不是任意长度自由输出。
4. 理解或分镜阶段失败时，任务返回明确错误码（`VIDEO_UNDERSTANDING_FAILED` 或 `VIDEO_STORYBOARD_FAILED`）和 `task:failed` 事件，不会跳过关键中间结果直接暴露不可解释的最终错误。
5. 理解与分镜阶段的中间结果以结构化形式写入 Redis 运行态（TTL 与任务 TTL 一致），后续 stage 可按 `taskId` 检索，不依赖进程内存传递。
6. 理解与分镜阶段各自通过 SSE `task:progress` 事件推送进度更新（含 `stage`、`stageLabel`、`progress`），前端可实时感知阶段推进。

### Business Context

- 理解与分镜是视频生成主链路的前两个阶段，决定了后续动画和旁白的质量与结构。
- `knowledgePoints` 不仅用于分镜，还会出现在最终视频结果的摘要区和公开卡片元数据中。
- `userProfile` 透传允许 LLM 根据用户的学习阶段和偏好调整解题深度和叙事风格。
- 分镜结构是 Manim 代码生成（Story 4.3）和 TTS 合成（Story 4.5）的直接输入，Scene 的 `narration` 给 TTS，`visualDescription` 给 Manim。

### Technical Guardrails

- LLM 调用必须通过 Story 2.7 Provider Protocol 进行，不得直接 `import openai` 或硬编码 API key。
- 理解结果与分镜结果必须写入 Redis 而非进程内存，因为 Worker 可能重启或 stage 分布在不同进程中。
- Redis key 命名遵循 Story 2.4 定义的 `video:task:{taskId}:*` 前缀规则。
- 分镜时长约束是业务规则，不是"建议"；超出 180s 的分镜不能被传递到下游 stage。
- 理解与分镜的 LLM prompt 设计不在本 Story 范围内（属于 prompt 工程迭代），但必须预留 prompt template 可替换的扩展点。
- 中间结果的序列化格式必须稳定，后续 stage 按约定字段消费；若需修改字段必须走 schema 变更流程。
