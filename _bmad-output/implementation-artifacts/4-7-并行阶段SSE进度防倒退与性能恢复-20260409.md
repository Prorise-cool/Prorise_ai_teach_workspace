# Story 4.7: 并行阶段 SSE 进度防倒退与性能恢复

Status: review

日期: 2026-04-09

## 背景

- `orchestrator.py` 已把 `manim_gen` 与 `tts` 调整为并行执行（`asyncio.gather`）。
- 旧的公开 stage 区间仍按串行假设设计，导致 `tts` 先推高进度后，`manim_gen` 再把进度拉回旧区间。
- 同一轮修复还需要恢复 `stash@{0}` 中的性能优化：`understanding+storyboard` 合并、TTS 并行、Docker 镜像预热。

## 本次修复

### 1. 恢复性能优化改动

- 恢复 `understanding+storyboard` 合并路径。
- 恢复 TTS scene 并行与 Docker 镜像预热。
- 补齐恢复后暴露出的兼容问题：逐场景 Manim 代码无效时回退全量生成；默认分镜目标时长恢复为 `120s`。
- LLM / TTS provider 改为复用单实例 `httpx.AsyncClient`，避免并行场景生成与多段旁白时重复建连。

### 2. 重新冻结并行阶段区间

- `understanding`: `0–12`
- `storyboard`: `13–25`
- `manim_gen`: `26–42`
- `tts`: `43–58`
- `manim_fix`: `59–65`
- `render`: `66–78`
- `compose`: `79–94`
- `upload`: `95–100`

### 3. 增加后端 SSE 防倒退守卫

- 普通阶段若 `progress < max_emitted_progress`，直接丢弃。
- `manim_fix` 为 render 失败后的补救阶段：保留 `stage=manim_fix` 与 `stageProgress`，但把全局 `progress` 钳制到当前最大值，避免等待页倒退。

### 4. 同步前端与契约

- `packages/student-web/src/features/video/config/video-stages.ts` 已同步新区间。
- `contracts/video/v1/pipeline-stages.md` 与 `mocks/video/v1/pipeline-stages.*.json` 已同步并行阶段语义。

  - `elapsed_seconds=470.179`
  - `monotonic=True`
  - 阶段序列：`understanding -> storyboard -> manim_gen -> tts -> render -> manim_fix -> render -> manim_fix -> render -> compose -> upload`
  - 题目理解阶段产出 `6` 个场景，超过当前 `parallel_scene_concurrency=5`，进入第二波并行请求，并触发两轮 `manim_fix`。
  - 完整修复链路中未出现 `58 -> 42`、`66 -> 59` 这类全局进度回退，说明普通阶段、修复阶段和最终完成态的 monotonic guard 在真实 SSE 流中生效。

### 当前结论

- 并行阶段 SSE 进度倒退问题已修复，且已通过成功样本与重载样本的真实链路观察验证。
- 本地可控的性能恢复项已全部落地：合并 understanding+storyboard、TTS 并行、Docker 预热、Provider 连接复用、failover 重试次数修正。
- 当前轻量成功样本仍略高于 `180s`（约 `181.26s`），重载样本则因 `6` 场景与双次修复链路扩展到 `470.179s`；剩余瓶颈主要来自上游 LLM 延迟与分镜数量/修复链路波动，而非本地串行等待或 SSE 回退。
