# Story 5.2: 主题输入与课堂任务创建

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 想系统学习某个主题的用户，
I want 通过输入主题发起课堂生成任务，
so that 我可以快速获得结构化讲解内容。

## Acceptance Criteria

1. 当用户位于 `/classroom/input` 并输入主题提交时，系统必须创建课堂任务并返回可查询的任务 ID 与基础任务信息，同时把 `userProfile` 与当前课堂配置一起稳定透传。
2. 当主题输入为空、过短或不满足最小约束时，页面必须返回明确、可恢复的输入校验提示，用户可以在当前页面修正后再次提交。
3. 当用户在视频输入页与课堂输入页之间切换时，必须感知相似的输入结构与配置位置，但课堂页文案与交互仍明确偏向“主题学习”而不是“单题讲解”。

## Tasks / Subtasks

- [ ] 收敛课堂输入页的页面职责与主叙事（AC: 1, 3）
  - [ ] 保持标题区、核心输入卡片、建议标签与引导卡片的页面结构，不让扩展配置挤占主 CTA。
  - [ ] 明确课堂输入页只负责主题输入、最小配置与任务创建，不顺手承接证据面板或 Companion 逻辑。
  - [ ] 保持与视频输入页一致的输入心智，但用课堂语义文案替换视频术语。
- [ ] 建立课堂任务创建表单 schema 与 payload 适配层（AC: 1, 2）
  - [ ] 补齐主题输入最小长度、去空格、重复提交保护与禁用态。
  - [ ] 从 profile store / API 读取 `userProfile` 并作为课堂创建请求的一部分透传。
  - [ ] 通过 adapter 隔离 mock / real 模式，禁止在页面组件里直接拼接后端 payload。
- [ ] 完成创建成功、失败与跳转闭环（AC: 1, 2）
  - [ ] 创建成功后跳转到 `/classroom/:id/generating`。
  - [ ] 401 / 403 / 4xx 输入错误 / 5xx 服务异常都有明确提示，且不丢失当前主题输入。
  - [ ] 对真实模式与 mock 模式统一返回任务摘要，不让页面感知后端差异。
- [ ] 补齐页面测试与运行态校验（AC: 1, 2, 3）
  - [ ] 覆盖成功创建、输入校验失败、会话失效与服务失败。
  - [ ] 覆盖跳转路由结果与 `userProfile` 透传。

## Dev Notes

### Business Context

- 当前课堂输入页已经有高保真视觉壳层，但还没有形成真实的“创建任务 → 等待页”业务闭环。
- `userProfile` 是课堂创建的重要输入，不允许在本 Story 中因为后端尚未 fully ready 就省略透传。
- 课堂输入页与视频输入页共享体验心智，但两者职责不同；课堂页不能被实现成“视频输入页换个标题”。

### Technical Guardrails

- 页面容器只负责业务编排、表单状态、服务调用与导航，不要把复杂视觉交互继续堆进单文件页面逻辑。
- `userProfile` 读取必须复用 profile 域既有 store / API，不再写第二套本地常量或 fallback 规则。
- 课堂创建请求必须通过 adapter 隔离 mock / real 模式，与任务统一 envelope 保持一致。
- 错误态必须在页面内可见且可恢复，禁止“点击提交后无反馈”或“失败后清空输入”。

- 当前 student-web 已有通用任务 adapter 与统一 SSE 入口，课堂创建不应再手写一套分散请求逻辑。
- 课堂 feature 当前尚未形成 `hooks / schemas / api` 完整分层，本 Story 可顺手补齐最小骨架，但不得越界实现等待页和结果页逻辑。

### Testing Requirements

- 覆盖输入校验、成功创建、失败提示与会话失效四类核心路径。
- 覆盖创建成功后的 `pathname` 断言，确保真正跳转到 `/classroom/:id/generating`。
- 覆盖 `userProfile` 缺省 / 已完成配置两类透传场景。
- 覆盖 mock 与 real adapter 共享同一字段语义，不出现双轨 payload。
