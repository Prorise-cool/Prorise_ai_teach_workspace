# Story 5.5: 白板布局与基础可读性规则

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 正在观看课堂的用户，
I want 白板内容即使复杂也保持基本可读，
so that 我不会因为公式、步骤和注释重叠而完全看不懂课堂结果。

## Acceptance Criteria

1. 当课堂结果包含白板内容时，系统生成的白板布局数据至少包含分步结构、主内容区与基础布局信息，前端不需要通过猜测坐标或自由布局才能勉强渲染结果。
2. 当白板内容较长、对象较多或结构复杂时，系统必须优先保证主要步骤可读与不严重重叠；即使达不到理想视觉效果，也要走降级布局而不是输出不可阅读的重叠内容。
3. 当某些白板片段不适合复杂空间布局时，系统可以输出更简化的步骤式结构，课堂主链路仍然可用，不因局部白板美观性不足而整体失败。

## Tasks / Subtasks

- [ ] 冻结白板布局 schema 与可读性检查规则（AC: 1, 2）
  - [ ] 定义 whiteboard step、element group、layout hint、fallback mode 等字段。
  - [ ] 明确哪些字段是渲染提示，哪些字段是业务步骤语义。
  - [ ] 产出最小白板示例，覆盖正常布局与降级布局。
- [ ] 实现服务端基础布局规则（AC: 1, 2）
  - [ ] 为主要步骤、公式、标注和辅助说明提供最小空间分配策略。
  - [ ] 在布局计算时优先保障主内容不严重重叠。
  - [ ] 对超长内容进行分步切片或分屏提示，而不是强塞同一帧。
- [ ] 实现结构化降级（AC: 2, 3）
  - [ ] 当空间布局失败时，退化为步骤式、列表式或分段式白板结构。
  - [ ] 通过明确状态字段标记当前结果为白板降级，而不是静默返回残缺坐标。
- [ ] 补齐白板可读性测试与异常样例（AC: 1, 2, 3）
  - [ ] 覆盖复杂内容、长公式、多对象、坐标冲突与不支持元素。
  - [ ] 覆盖降级结果仍可被结果页稳定渲染。

## Dev Notes

### Business Context

- 白板是课堂结果的关键辅助表达层，但 MVP 目标是“基本可读”，不是一步到位达到复杂演示编辑器水准。
- 本 Story 的成功标准不是白板绝对美观，而是确保主链路不会因为白板局部复杂而整体不可用。
- 课堂结果页将直接消费本 Story 产出的结构，所以字段语义必须比视觉实现更稳定。

### Technical Guardrails

- 白板输出首先是业务语义结构，其次才是坐标与布局提示；不能把前端渲染细节直接当成唯一真值。
- 严重重叠必须显式进入降级模式，禁止继续输出“看似成功、实际不可读”的坐标数据。
- 结构化降级结果必须可被前端结果页统一渲染，不要求依赖特定第三方白板引擎。
- 不得直接照抄 OpenMAIC 的白板实现；只能借鉴其“白板作为独立结构层”的思路。

### Testing Requirements

- 覆盖复杂白板、长公式、多对象与超长步骤的可读性边界。
- 覆盖降级模式输出的合法性与可消费性。
- 覆盖不支持布局类型时的明确状态，而不是 silent failure。
- 覆盖结果结构校验，防止前端收到缺字段或非法坐标。
