## Story

As a 等待讲解视频生成的用户，
I want 系统逐段生成并展示可观看的片段视频，而不是等全部生成完才给我看，
so that 我能边看边等，在最终整片完成前就开始消费部分结果。

---

## 核心改动：从 Batch 模式重构为 Per-Section Streaming 模式

### 问题根因

当前 pipeline 是 **batch 模式**：所有 section 的代码先生成完，再统一渲染，再统一 TTS，最后合成。用户在 77% 的时间里（manim_gen，459s）什么可消费内容都看不到。

```
当前（batch）：
understanding → storyboard → [等 459s manim_gen 全部完成] → render → TTS → compose → upload
                              ↑ 用户在这 7.5 分钟里只能盯着进度条
```

### 目标架构：Per-Section Streaming

把 manim_gen → render → 片段发布 合并为 per-section 的流式管线。每个 section 独立走完"生成代码 → 渲染 → 合成音频 → 发布给用户"的全链路，用户看到一段就多一段。

```
目标（streaming）：
understanding → storyboard → TTS(全部) → 
  [section 1: manim_gen → render → 合成音频 → 发布 ✓ 用户可看] →
  [section 2: manim_gen → render → 合成音频 → 发布 ✓ 用户可看] →
  [section 3: manim_gen → render → 合成音频 → 发布 ✓ 用户可看] →
  ... →
  final_compose(拼接所有 section) → upload → 跳转结果页
```

### 为什么要 TTS 前移

TTS 只需 ~5s（5 个 section），放在 storyboard 之后、manim_gen 之前：
- 每个 section 渲染完时，TTS 音频已经就绪，可以立即合成"带音频的 section 片段"
- 用户看到的每个片段都是**有声音的成品**，而不是无声动画

### 实测基准（vtask_20260412143844_92ee9db0，5 sections，45s 视频）

| 阶段 | 实测耗时 | 占比 | 说明 |
|------|----------|------|------|
| understanding + storyboard | ~80s | 13% | 不变 |
| TTS | ~5s | 1% | **前移到 storyboard 之后** |
| per-section pipeline（核心改动） | ~510s | 85% | manim_gen 459s + render 49s，但用户**每完成一个 section 就能看到** |
| final compose + upload | ~5s | 1% | 拼接所有 section 片段 |

## 新 Pipeline 执行模型

### 新 orchestrator 执行流程

```python
# === 阶段 A：准备（~85s）===
agent.generate_outline()                           # understanding (~39s)
agent.generate_storyboard()                        # storyboard (~42s)
tts_audio_map = _run_tts_all_sections()            # TTS 前移 (~5s)
emit("storyboard_complete", preview_data)          # 用户看到分镜 + 可听旁白

# === 阶段 B：Per-Section Streaming Pipeline ===
for i, section in enumerate(agent.sections):
    # B1: 生成该 section 的 Manim 代码
    emit("section_generating", {section_index: i, total: N})
    agent.generate_section_code(section)
    
    # B2: 渲染该 section（含内部 fix 循环）
    success = agent.render_section(section)         # 含最多 3次重生成 + 5次修复
    
    if success:
        # B3: 合成该 section 的视频 + TTS 音频
        section_clip = _compose_section(section, tts_audio_map[section.id])
        
        # B4: 发布片段 URL
        clip_url = _publish_section_clip(task_id, section.id, section_clip)
        
        # B5: 通知前端
        emit("section_ready", {
            section_index: i, section_id: section.id,
            clip_url: clip_url
        })
    else:
        emit("section_failed", {section_index: i, section_id: section.id})

# === 阶段 C：最终合成 ===
final_video = _concatenate_all_sections(rendered_section_clips)
upload_result = upload_svc.execute(final_video)
emit("completed", {video_url: upload_result.video_url})
```

### Pipeline 并行优化（可选，Phase 2）

基础版是严格串行的 per-section loop。如果性能不够，可以改为流水线并行：

```
时间轴（流水线并行）：
  manim_gen:  [s1 生成]            [s2 生成]            [s3 生成]      ...
  render:              [s1 渲染]            [s2 渲染]            [s3 渲染]  ...
  publish:                    [s1 发布 ✓]          [s2 发布 ✓]          ...
  用户看到:                   ↑ ~170s 可看 s1      ↑ ~260s 可看 s2
```

这样 section N 渲染的同时，section N+1 已经在生成代码，吞吐量更高。但实现复杂度也更高。**建议 Phase 1 先做串行，验证体验后再考虑并行。**

### 旧状态机（阶段维度）

```
understanding → storyboard → manim_gen → render → compose → upload
     0-8%        19-28%      29-42%     61-70%   81-93%   94-100%
```

### 新状态机（混合维度）

```
阶段 A（准备态）：
  understanding(0-10%) → storyboard(10-20%) → tts(20-25%)

阶段 B（流式态，循环 N 次）：
  section_i_generating(25% + i*(70%/N))  →  section_i_rendering → section_i_ready

阶段 C（收尾态）：
  final_compose(95-98%) → upload(98-100%) → completed
```

## 用户体验三段式

### 阶段 A：分镜可听（~85s）

storyboard + TTS 完成后：

- 展示题目摘要 + 知识点
- 展示分镜卡片列表（scene title、旁白文本、时长预估）
- 每个分镜卡片有"试听旁白"按钮（TTS 音频已就绪）
- 用户可以**边听旁白边等视频**

### 阶段 B：片段可看可听（流式产出）

per-section pipeline 逐个完成：

- 每个 section 完成后，卡片从"排队中/生成中"变为"可播放"
- 卡片内嵌入 mini 播放器，播放该 section 的**带音频片段**
- 修复状态 per-section 展示（"第 2 段正在自动修复"，不影响第 1 段观看）
- 进度用 "N/M sections completed" 表达，比百分比更直观

### 阶段 C：跳转结果页（~5s）

final compose + upload：

- 所有 section 拼接为最终完整视频
- 跳转到结果页，播放完整视频
- 结果页不受影响，仍只消费 `VideoResult.videoUrl`

## Acceptance Criteria

1. **TTS 前移**：TTS 阶段在 storyboard 完成后立即执行（不再等 render），storyboard 完成后用户可试听各 section 旁白。

2. **Per-section streaming pipeline**：后端将 `generate_codes(ALL) → render_all_sections(ALL)` 的 batch 模式重构为 per-section 的 `generate_section_code(i) → render_section(i) → compose_section_with_audio(i) → publish(i)` 循环。

3. **Section 级 SSE 事件**：每个 section 的状态变化（generating → rendering → fixing → ready/failed）通过独立 SSE 事件推送；前端收到 `section_ready` 事件后立即展示可播放的片段。

4. **Section 片段发布**：每个 section 渲染 + 合成音频完成后，立即将片段发布为稳定可访问 URL（`/api/v1/video/assets/video/{taskId}/sections/{sectionId}.mp4`）；刷新页面后通过 preview endpoint 恢复已发布片段。

5. **结果页边界不变**：`/video/:id` 结果页仍只消费最终 `VideoResult.videoUrl`；section 片段只是等待页的渐进预览，不是任务完成的替代品。

6. **Per-section 失败隔离**：某个 section 修复失败时，该 section 标记为 failed/omitted，但不影响其他已完成 section 的展示；最终合成使用所有成功 section 的片段。

### Technical Guardrails

- **结果页语义不变**：`/video/:id` 仍只消费 `VideoResult.videoUrl`
- **Per-section 失败隔离**：单个 section 失败不阻塞其他 section
- **TTS 前移不影响质量**：TTS 只需要 `voice_text`（来自 storyboard），不需要 render 产物
- **Section 片段是预览不是最终产物**：片段用于等待页消费，最终合成仍需重新拼接
- **保持 Dramatiq 预算控制**：per-section 循环仍需在总 time limit 内完成

### 关键技术决策

| 决策点 | 方案 | 理由 |
|--------|------|------|
| Section 执行顺序 | Phase 1 串行，Phase 2 可选流水线并行 | 串行实现简单，先验证体验 |
| TTS 位置 | storyboard 之后、manim_gen 之前 | TTS 只需 ~5s，提前准备好音频让 section 片段可带声音 |
| Section 片段合成 | 单独 FFmpeg merge（video + audio） | 每个 section 独立合成，不等全部完成 |
| Final compose | 拼接所有 section 片段（已有音频） | 最终合成只做 concat，不再做音频合并 |
| Preview 数据源 | Redis 运行态 + 独立 endpoint | 不把完整预览塞进 SSE，轻量信号驱动前端拉取 |
