# Story 1.2: 独立认证页中的真实账密登录、受控注册、i18n 基线与回跳

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 访客，
I want 在独立认证页中完成真实账密登录，并在 RuoYi 开启注册时切换到注册态，同时让认证页从一开始就具备 i18n 文案资源化基础，
so that 我可以先稳定进入小麦，也不会在后续因为注册开关、国际化和认证页视觉返工。

## Acceptance Criteria

1. 访客进入 `/login` 后，可通过账号密码走真实 RuoYi 认证链路完成登录；成功后建立已认证会话并回跳原页面或原意图上下文。
2. 登录页根据 RuoYi 用户注册开关动态决定是否展示“注册” tab；当注册关闭时，页面只展示登录态，不出现不可用注册流程。
3. 当注册开启时，访客可在同页切换到注册态并提交账密注册；注册字段、校验与提交流程遵循 RuoYi 默认注册语义，注册成功后给出明确反馈并回到登录态或按后端默认流程继续，不伪造前端自动登录。
4. 登录 / 注册页的关键静态文案进入 i18n 资源管理，MVP 默认中文；当前阶段不要求完整双语切换，但后续新增语言资源时不需要重写页面结构。
5. 页面在无效输入、凭证错误、注册失败或认证异常时，展示明确的字段级或表单级错误提示，用户可在当前页继续修正。
6. 登录页的视觉密度与设计稿收口，优先调整左右面板比例、内边距和信息密度等“显大”问题；同时保留键盘操作、可见焦点与 1024px 桌面最小体验。

## Dev Notes

### Business Context

- `/login` 是整个产品唯一认证入口，首页与受保护页面只负责把用户带到这里。
- Story 1.2 必须消费 Story 1.1 的认证契约、mock handler 和 `returnTo` 约定。
- 当前版本重点是“先打通真实登录注册 + 尽早埋下 i18n 基线”，不是继续堆三方登录、邮箱验证、发件能力或复杂身份分流页面。
- 注册是否可用以 RuoYi 后台开关为准，不由前端写死。

### Technical Guardrails

- 认证页必须是独立页面，不允许首页弹窗完成登录。
- 成功后必须回跳原上下文，未授权重定向不能丢失用户原始意图。
- 登录必须对接真实 RuoYi 认证链路，不能长期停留在 mock 登录。
- 注册字段、图形验证码显示与租户 / client 语义必须遵循 RuoYi 默认注册接口，不自行扩展手机号、邮箱验证码或发件流程。
- i18n 范围只要求认证页静态文案资源化与 provider 基线，不要求本阶段完成完整双语切换。
- 视觉修正优先聚焦内部留白、左右面板比例与信息密度，不把 `1080 x 680` 外层容器尺寸误判为唯一问题。
- 关键交互必须支持键盘操作，焦点样式不可被动画或自定义样式吞掉。
- 认证失败文案不能泄露账户是否存在、密码策略等敏感信息。
- 继续沿用现有前端栈：React 19、React Router 7、React Hook Form、Zod、Testing Library。
