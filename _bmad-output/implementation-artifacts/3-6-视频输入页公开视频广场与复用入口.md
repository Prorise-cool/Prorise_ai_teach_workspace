# Story 3.6: 视频输入页公开视频广场与复用入口

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 想快速判断是否已有相近讲解的用户，
I want 在视频输入页浏览公开视频卡片并把合适内容复用到当前输入流程，
so that 我可以先看已有结果或少走重复创建路径。

## Acceptance Criteria

1. `/video/input` 页面在核心输入区下方展示公开视频卡片列表或推荐集合，主输入区与主 CTA 仍保持页面主叙事位置，公开内容区不抢占核心输入区的视觉权重。
2. 公开视频卡片至少展示缩略图、题目摘要、时长、观看次数（或热度指标），用户可快速扫读判断是否有需要的内容。
3. 用户点击卡片上的"查看讲解"可进入对应公开视频结果页（`/video/:id`）；点击"复用题目"可将该视频的题目文本或摘要带回当前输入区，自动填充到 textarea 中。
4. 公开内容为空时展示空态插画与提示文案（"暂无公开视频，快来创建第一个"）；加载失败时展示降级提示但不阻断主输入流程；mock 模式下使用 mock 卡片数据。
5. 公开视频数据通过 `GET /api/v1/video/public` 获取，使用 `useQuery` 管理缓存与刷新；接口返回分页列表，首次加载展示一屏（6-12 张卡片）。
6. 公开视频发现区的加载失败、空态、骨架屏均不影响核心输入区的可用性——用户始终可以直接输入题目并创建新视频任务。
7. 卡片布局使用响应式网格，桌面端 3-4 列，配合已有的 `CommunityFeed` 组件或升级其数据源。

### Business Context

- 公开视频发现区是"视频输入页的辅助层"——帮助用户快速判断"是否已有人生成过类似题目的讲解"。这是降低用户重复创建成本的核心体验。
- 但公开视频区的优先级低于核心输入区。Epic 3 Frontend Design Reference 明确规定"公开视频发现区属于辅助理解与复用层，不得让其抢占核心输入区与主 CTA 的主叙事位置"。
- 当前输入页已有 `CommunityFeed` 组件渲染 mock 卡片（`VIDEO_FEED_MOCK_CARDS`），本 Story 需将其升级为真实数据源驱动，或在需要时替换为更适合视频域的卡片组件。
- "复用题目"是一个低频但高价值的操作——当用户发现已有类似讲解时，可以直接复用其题目文本，节省输入成本。

### Technical Guardrails

- 公开视频列表使用 `@tanstack/react-query` 的 `useQuery` 管理，不手写请求状态机。
- 公开视频数据的加载失败必须被隔离——不能让 API 失败导致整个输入页白屏或不可用。使用 React Error Boundary 或 `useQuery` 的 `isError` 状态在局部区域降级。
- "复用题目"通过 `react-hook-form` `setValue` 设置 textarea 值，不绕过表单管理直接操作 DOM。
- 卡片布局如果复用 `CommunityFeed`，需确保其数据接口可泛化（不仅限于 mock 数据）。若 `CommunityFeed` 的接口设计过于刚性，可新建 `VideoPublicFeed` 组件。
- 公开视频的后端接口（`GET /api/v1/video/public`）在本 Story 中只定义契约和 mock，真实后端实现由 Story 4.10 负责。
- 所有 `className` 通过 `cn()` 合并，卡片变体如需定制使用 CVA。
