# Story 3.4: 视频任务创建接口与初始化运行态

Status: backlog

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 想发起视频任务的用户，
I want 系统在接收输入后创建一个规范的异步任务，
so that 我可以立即进入等待态并跟踪后续执行。

## Acceptance Criteria

1. `POST /api/v1/video/tasks` 接口接收通过校验的输入，返回 `202 Accepted` 并携带 `{ taskId, taskType: "video", status: "pending", createdAt }`，字段口径与 Story 2.1/3.1 契约一致。
2. 后端创建任务时在 RuoYi 侧写入视频任务元数据记录（含 `taskId`、`userId`、`inputType`、`sourcePayload` 摘要、`status: pending`、`createdAt`），元数据字段与 Epic 10 长期承接定义对齐。
3. 后端在 Redis 中写入任务运行时状态 key（`task:{taskId}:status`），初始快照包含 `{ taskId, taskType, status: "pending", progress: 0, createdAt }`，与 SSE 事件和 `/status` 查询接口共享同一份运行态。
4. 创建成功后向 Dramatiq + Redis broker 分发异步视频任务消息，消息体包含 `taskId`、`inputType`、`sourcePayload`、`userProfile`，Worker 可独立消费。
5. 创建接口遇到输入错误返回 `422`、权限问题返回 `403`、幂等键冲突返回 `409`、内部错误返回 `500`，均遵循统一 `{ code, msg, data }` 结构。
6. `clientRequestId` 幂等处理：同一 `clientRequestId` 重复提交时，返回已有 `taskId` 而非创建新任务，前端可安全重试。
7. 创建接口响应时间 P95 < 500ms（不含 Worker 执行时间），不阻塞用户等待。

### Business Context

- Story 3.4 是视频主链路的"起点服务"——用户点击"开始生成"后，后端需要在毫秒级完成任务受理（202 Accepted），然后把真正的计算工作交给 Worker 异步执行。
- 创建接口不执行视频生成的任何 stage（理解、分镜、Manim、TTS、合成），只负责"受理 + 初始化 + 分发"。
- 幂等处理是关键——前端在网络不稳定时可能重试提交，后端必须保证同一 `clientRequestId` 不会创建重复任务。
- RuoYi 元数据写入是为了长期持久化（Epic 10），但 MVP 阶段不应让 RuoYi 不可用阻断任务创建。

### Technical Guardrails

- 创建接口必须是异步受理模式（202），不能同步等待 Worker 执行结果。
- Redis 运行态 key 必须使用 Story 2.4 约定的命名规则和 TTL 策略。
- Dramatiq actor 注册必须通过 Story 2.2 的 task 基类完成，不能绕过统一调度骨架。
- RuoYi 调用通过防腐层客户端（Story 10.3）完成，不能直接调用 RuoYi REST API。
- 所有数据库/Redis/消息队列操作必须在 `request_id` 追踪上下文中完成。
- 创建接口的 Pydantic model 必须与 `contracts/video/v1/create-task-request.schema.json` 保持同步，不能出现字段漂移。
