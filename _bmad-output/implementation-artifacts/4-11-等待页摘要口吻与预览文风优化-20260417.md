# 4-11 等待页摘要口吻与预览文风优化

- 日期：2026-04-17
- Story：`4-11-视频等待页渐进式产物展示与分段预览`
- 状态：`review`

## 问题现象

等待页“先看这段快速解释”虽然已经不再回退成题面原文，但生成出来的内容仍然偏像：

- 证明提纲
- 论文摘要
- 竞赛题标准解题步骤

对学生来说，这类内容的问题不是“信息少”，而是“第一眼不亲近”，读起来像在看答案模板，不像老师先把题意讲明白。

## 根因

### 1. understanding prompt 仍把模型往结构化学术文风上推

- 文件：`packages/fastapi-backend/app/features/video/pipeline/services.py`
- 旧 prompt 虽然已经要求 JSON-only，但 `topicSummary` 仍只要求“2-4 句快速解释”，`solutionSteps` 也只强调“学生能直接看懂”。
- 这会让模型天然选择最省 token 的“提纲式表达”，例如“构造辅助条件 → 应用定理 → 完成证明”。

### 2. preview 摘要拼装继续强化“步骤列表感”

- 文件：`packages/fastapi-backend/app/features/video/pipeline/orchestration/orchestrator.py`
- 旧 `_build_preview_summary()` 会把 `solutionSteps` 渲染成 `1. **标题**：解释`。
- 即使 explanation 已经口语化，这种编号粗体结构也会把成品重新拉回“证明步骤卡片”的观感。

## 本次修复

### 1. prompt 改为老师讲题口吻

- 明确要求模型：
  - 不是写学术摘要，而是像老师在学生旁边讲题
  - 先说“这题在问什么 / 先抓什么”
  - 解释“为什么这样做”
  - 至少提醒一个常见易错点
  - 可以使用 Markdown / LaTeX，但必须顺手解释公式
  - 避免“本题探讨……”“构造辅助条件”“应用某定理即可”这类技术腔

### 2. `topicSummary` 拉长为老师式 4-6 句

- 目标不再是最短摘要，而是让学生在等待视频时，先把题意和主线吃进去。
- 这样即使视频还没生成完，用户也已经拿到了第一轮真实帮助。

### 3. preview 步骤列表改成轻量 bullet

- 从：
  - `1. **标题**：解释`
- 调整为：
  - `- 标题：解释`

这会弱化“标准答案提纲感”，保留结构，又不把用户推到过强的技术阅读模式里。

## 结果预期

修复后，等待页摘要应更接近：

- 先讲题意
- 再讲主线
- 顺手提醒误区
- 最后再用轻量步骤帮学生抓住顺序

而不是只给一段“形式上正确，但读起来像标准答案目录”的内容。
