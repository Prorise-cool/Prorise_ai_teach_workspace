# Story 10.4: 视频与课堂任务元数据长期承接

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 平台与回访用户，
I want 视频和课堂的任务元数据被长期保存，
so that 结果页回看、学习中心聚合与后台查询有稳定数据来源。

## Acceptance Criteria

1. 视频任务或课堂任务创建、执行、完成或失败时，系统至少保存任务 ID、用户归属、任务类型、状态、摘要、结果资源标识与时间信息，这些数据进入长期业务表而不是只依赖 Redis 运行态。
2. 任务失败时，后台或学习中心查询任务能够识别失败状态、失败时间和最小错误摘要，不会在长期记录中只留下成功结果。
3. 用户从学习中心再次打开视频或课堂结果时，页面能通过长期元数据定位到正确结果详情，不需要重新扫描运行态缓存恢复入口。

## Dev Notes

### Business Context

- 该 Story 是视频与课堂结果可回看、可聚合、可审计的长期数据入口，直接支撑 Epic 4、Epic 5 与 Epic 9。
- `10.4` 不负责视频 / 课堂执行逻辑本身，而是负责把执行结果稳定沉淀为长期业务元数据。
- 一旦元数据没有长期承接，学习中心和后台就会被迫依赖 Redis 运行态或重新扫描结果产物，无法形成稳定产品能力。

### Technical Guardrails

- 视频 / 课堂任务元数据必须进入 RuoYi 业务表，不得只停留在 Redis 运行态。
- 任务状态应统一使用 `pending`、`processing`、`completed`、`failed`、`cancelled`。
- 失败记录必须长期可见，不能只保留成功记录。
- 结果详情应通过稳定对象引用、结果摘要或 SessionArtifactGraph 索引定位，不依赖 Redis 快照回放。
- 时间字段与分页 / 查询语义必须对齐 RuoYi 全局规范。
