# Story 4.3 Hotfix: Manim 429 重试风暴热修复

Status: review

日期: 2026-04-10

## 背景

- 视频流水线在 `manim_gen` 并行场景生成时，`6` 个场景中的前 `5` 个会按 `video_manim_parallel_scene_concurrency=5` 同时触发 LLM 请求。
- 上游返回 `429` 后，Failover 仍按固定退避 `1s -> 2s -> 4s` 同步重试，短时间内形成对同一 Provider 的重试波浪。
- 并行路径失败后，`_generate_single_pass()` 会立刻接管，但此前 429/连接失败写入的 `xm_provider_health:{provider}` 会把同一 Provider 直接判成 cached unhealthy，导致保底路径也被跳过。

## 根因

1. `ProviderFailoverService` 的退避时间是确定性的，多个协程会在同一秒一起重试。
2. `429` 被 `ConnectionError` 分支先吞掉，误判为需要写坏健康缓存的不可用状态。
3. `ManimGenerationService` 的单次全量 fallback 没有绕过当前请求周期里刚写入的 cached unhealthy。
4. 默认并发数 `5` 对当前上游 RPM / 并发预算偏高，放大了重试风暴的窗口。

## 本次热修

### 1. Failover 重试增加 Jitter

- 文件：`packages/fastapi-backend/app/providers/failover.py`
- 改动：退避从固定 `min(2 ** attempt_index, 8)` 改为 `base_backoff + random.uniform(0, 1)`。
- 目的：打散并发协程的重试时点，避免同步重试波浪。

### 2. 429 / rate limit 不再污染健康缓存

- 文件：`packages/fastapi-backend/app/providers/failover.py`
- 改动：新增 rate-limit 分类标记，`429` / `rate limit` / `too many requests` 保持 `retryable=True`，但 `mark_unhealthy=False`。
- 目的：限流属于容量信号，不应把 Provider 直接缓存成“不健康”。

### 3. Manim 单次 fallback 可忽略 cached unhealthy

- 文件：`packages/fastapi-backend/app/features/video/pipeline/manim.py`
- 改动：并行生成或逐场景生成失败后，进入 `_generate_single_pass(..., ignore_cached_unhealthy=True)`。
- 文件：`packages/fastapi-backend/app/providers/factory.py`
- 改动：为 failover 入口补充 `ignore_cached_unhealthy` 透传。
- 目的：允许同一请求中的保底路径绕过刚写入的短期缓存，至少再尝试一次全量生成。

### 4. 默认并发从 5 下调到 3

- 文件：`packages/fastapi-backend/app/core/config.py`
- 改动：`FASTAPI_VIDEO_MANIM_PARALLEL_SCENE_CONCURRENCY` 默认值 `5 -> 3`。
- 目的：在不完全牺牲吞吐的前提下先压低上游瞬时请求峰值。

## 结论

- 这次热修优先解决“429 同步重试风暴 + 健康缓存投毒 + fallback 被误杀”的组合问题。
- 真实 `DeepSeek` 运行时绑定下的并发 `manim_gen` smoke test 已通过，说明修复后的 failover 入口不会阻断正常 provider 调用。
- 新 token 下的真实视频任务已在 `101s` 内完成出片，满足“3 分钟内出视频且不降低质量”的当前验收目标。
- 当前仍未把 Semaphore 持有范围从“整个 failover 调用”细化到“单次尝试”，如果后续仍遇到上游强限流，建议继续评估全局 rate limiter 或 provider 级令牌桶。
