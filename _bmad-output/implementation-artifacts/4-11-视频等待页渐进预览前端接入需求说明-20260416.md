# Story 4.11 视频等待页渐进预览前端接入需求说明

> 本文是 `Story 4.11` 的前端接入需求文档，不是代码教程。唯一事实源仍以 `_bmad-output/` 为准；本文件用于明确 student-web 在接入新视频等待页 API 时必须实现的能力边界、消费契约与代码归属点。

Status: review

> 2026-04-16 更新：student-web 已完成本次前端接入并进入 review。当前实现保持 `/video/:id/generating` 入口不变，同时把输入页收敛成“干净输入框 + 底部 icon 工具栏”，三档质量预设与高级参数都改为图标入口并全量接入 i18n / 亮暗色；等待页已新增 `/preview` adapter + hook、section 级 store/SSE 汇合、三段式渐进预览布局，以及 mock / Vitest 覆盖。定向回归已通过：`pnpm --filter @xiaomai/student-web typecheck`、`pnpm vitest run src/features/video/pages/video-input-page.test.tsx src/features/video/pages/video-generating-page.test.tsx`、`pnpm build`。

## 文档目标

本次文档只回答 4 个问题：

1. 前端要向用户提供什么新能力。
2. 新增的 preview / section 级事件如何与现有等待页、结果页职责拼合。
3. 哪些代码位置是本次接入的明确归属点。
4. 哪些边界不能被改坏。

本次文档不承担以下职责：

- 不给出 React hook、zustand、组件拆分的代码写法教程。
- 不把 section 预览误写成“结果页提前完成”。
- 不把开发态本地资产路由误写成正式产品契约。

## 业务目标

当前等待页已经具备“创建成功后进入 `/video/:id/generating`、通过 `/status` 恢复、通过 SSE 刷新进度、完成后跳结果页”的基础闭环；但对 `Story 4.11` 来说，这还不够。

等待页需要从“只显示总进度”升级为“分三段展示真实中间产物”：

1. `阶段 A`：先展示理解摘要、知识点、分镜与旁白试听。
2. `阶段 B`：随着 section 产出，逐段展示可播放的 preview clip。
3. `阶段 C`：最终整片合成完成后，仍旧跳转 `/video/:id` 结果页播放完整视频。

用户在等待页看到的 section clip 只是渐进预览，不代表任务已经成功完成。

## 前端必须实现的能力

### 1. 创建后入口不变

- 继续以 `POST /api/v1/video/tasks` 返回的 `taskId` 为唯一任务标识。
- 继续以 `/video/:id/generating` 作为等待页唯一入口。
- 继续以 `/video/:id` 作为最终结果页唯一入口。

也就是说，`Story 3.5` 的“创建成功即跳等待页”链路保留不变，只是在等待页内部增加渐进预览能力。

### 2. 等待页要具备三段式内容能力

#### 阶段 A：分镜与旁白可消费

等待页在 preview 可用后，必须能够展示：

- `summary`
- `knowledgePoints`
- `sections[]` 的标题、顺序、旁白文本或试听入口
- `audioUrl` 对应的试听能力

这里的展示来源必须是后端真实 preview 数据，不能靠解析日志 message 自行拼装。

#### 阶段 B：section 预览卡片流

等待页必须按 section 维度维护状态，而不是只维护全局 `progress`：

- `pending`
- `generating`
- `rendering`
- `fixing`
- `ready`
- `failed`

当 section 进入 `ready` 时，前端必须能把对应 `clipUrl` 挂到该 section 卡片上，并允许用户即时预览。

#### 阶段 C：最终结果交接

当任务进入最终完成态时：

- 等待页不再以 section clip 充当成功态。
- 仍按既有机制跳转 `/video/:id`。
- 结果页仍只消费最终 `result.videoUrl` 与结果详情，不消费 section clip 列表。

### 3. 刷新恢复能力必须成立

用户刷新 `/video/:id/generating` 时，前端不能退回成“只剩一个总进度条”。

刷新恢复至少要同时覆盖：

- `/status` 提供的任务快照恢复
- `/preview` 提供的 summary / knowledgePoints / sections / audioUrl / clipUrl 恢复
- SSE 重连后 section 状态继续推进

### 4. SSE 与轮询降级必须继续成立

现有等待页已经基于统一 SSE parser 消费任务事件，并具备断线恢复/降级语义。本次接入后这些能力不能退化。

新增能力要求：

- 继续消费既有 `progress` / `completed` / `failed` / `snapshot` 语义。
- 新增消费 `section_progress` 与 `section_ready`。
- 当 SSE 不可用时，页面不能失去 preview 已有内容的恢复能力。

### 5. 失败态要区分任务失败与 section 失败

- 单个 section `failed` 不等于整任务失败。
- 只有任务进入 `failed` / `cancelled` 终态时，等待页才进入全页失败态。
- section 级失败应该体现在 section 卡片，而不是直接终止整个等待页体验。

## API 接入边界

### 必须消费的接口

#### 1. 创建任务

- 路径：`POST /api/v1/video/tasks`
- 作用：生成 `taskId`，进入等待页
- 前端边界：只负责拿到任务 ID 并进入等待页，不在创建接口阶段承载 preview 数据

#### 2. 等待页事件流

- 路径：`GET /api/v1/video/tasks/{task_id}/events`
- 真实委托：后端视频路由转发到统一 tasks SSE 能力
- 作用：主通道，持续驱动等待页状态推进
- 恢复要求：继续遵守 `Last-Event-ID` 恢复语义

#### 3. 等待页 preview 数据

- 路径：`GET /api/v1/video/tasks/{task_id}/preview`
- 作用：提供真实中间产物与 section 恢复数据
- 最关键字段：
  - `previewAvailable`
  - `previewVersion`
  - `summary`
  - `knowledgePoints`
  - `sections[].status`
  - `sections[].audioUrl`
  - `sections[].clipUrl`

#### 4. 等待页状态快照

- 路径：`GET /api/v1/video/tasks/{task_id}/status`
- 作用：恢复任务生命周期、stage、progress 与终态
- 边界：它不是 preview 替代品；它解决“任务状态恢复”，不解决“渐进内容恢复”

#### 5. 最终结果页详情

- 路径：`GET /api/v1/video/tasks/{task_id}/result`
- 作用：最终结果页唯一权威数据源
- 边界：只有最终整片可用后才由结果页消费

### 只能视为开发态能力的接口

- 路径：`GET /api/v1/video/assets/{asset_key:path}`
- 用途：本地开发环境读取落盘资产
- 产品边界：这是开发态服务路由，不应被前端需求文档写成正式线上契约

## 代码归属点

本次接入要明确“谁拥有什么职责”，避免把 preview 逻辑散落在页面里。

### student-web 前端归属

#### 1. 创建链路归属

- 文件：`packages/student-web/src/services/api/adapters/video-task-adapter.ts`
- 当前职责：只负责 `createTask()`
- 本次要求：保持“创建任务”职责清晰，不把 preview / result / SSE 解析硬塞回创建 adapter

#### 2. 等待页页面编排归属

- 文件：`packages/student-web/src/features/video/pages/video-generating-page.tsx`
- 当前职责：等待页壳层、快照恢复、SSE 启动、终态跳转
- 本次要求：这里负责“等待页编排与三段式展示入口”，但不应成为 preview 数据结构的唯一存储点

#### 3. SSE 消费归属

- 文件：`packages/student-web/src/features/video/hooks/use-video-task-sse.ts`
- 当前职责：消费通用任务事件并驱动 `video-generating-store`
- 本次要求：扩展为识别 `section_progress` / `section_ready`，但继续复用统一 SSE parser，不重写协议层

#### 4. 状态恢复归属

- 文件：`packages/student-web/src/features/video/hooks/use-video-task-status.ts`
- 当前职责：消费 `/status` 快照并恢复任务上下文
- 本次要求：保留任务状态恢复职责，不承担 preview 详情结构的长期归属

#### 5. 阶段映射归属

- 文件：`packages/student-web/src/features/video/config/video-stages.ts`
- 当前职责：全局 stage → label/progress/ETA 映射
- 本次要求：继续只维护“全局流水线阶段”语义，不把 section 卡片状态写进这里

#### 6. 等待页状态机归属

- 文件：`packages/student-web/src/features/video/stores/video-generating-store.ts`
- 当前职责：保存全局任务状态、stage、progress、error、fix 状态
- 本次要求：这里是 section 维度状态机的自然归属点，需要从“单全局任务状态”扩展为“全局任务状态 + sections[] 状态”

### FastAPI 后端归属

#### 1. 视频路由归属

- 文件：`packages/fastapi-backend/app/features/video/routes.py`
- 相关接口：
  - `create_video_task_endpoint`
  - `get_video_task_preview`
  - `get_video_task_status`
  - `get_video_task_events`
  - `get_video_task_result`
- 职责：对外暴露 video 域接口边界

#### 2. 统一任务事件流归属

- 文件：`packages/fastapi-backend/app/api/routes/tasks.py`
- 相关能力：
  - `get_task_status`
  - `get_task_events`
  - `stream_task_events`
- 职责：统一任务快照 / SSE 恢复 / heartbeat / replay 语义

## 不允许被改坏的边界

1. `/video/:id` 结果页仍只认最终整片成功态。
2. section clip 只属于等待页渐进预览，不属于结果页成功态。
3. 等待页仍必须支持刷新恢复、断线恢复和失败解释。
4. 前端不能通过猜测 message 文本拼接 section 内容。
5. preview 内容必须来源于后端真实 preview 结构。
6. 开发态本地资产路由不能被当成线上产品契约。

## 手工验收口径

本次前端接入完成后，至少应满足以下人工验收口径：

1. 创建任务后，页面仍立即进入 `/video/:id/generating`。
2. 等待页在 preview 可用时能看到 summary、knowledgePoints 和 section 列表。
3. 某个 section ready 后，该 section 卡片能展示可播放片段，其他 section 仍可继续推进。
4. 刷新页面后，已 ready 的 section 片段不会丢失。
5. 单 section 失败不会把整页直接打成任务失败。
6. 最终任务 completed 后，页面仍跳 `/video/:id`，且结果页播放的是完整视频而不是 section clip。

## 前端点击验收清单

1. 打开 `/video/input`，确认主输入框内部不再被预设控件占位；底部工具栏新增“质量预设”和“高级参数”两个 icon 入口。
2. 点击“质量预设”图标，确认默认方案为“均衡讲解”；切换三档预设后，时长/分段/并发/渲染质量/布局摘要 chip 会同步变化。
3. 点击“高级参数”图标，确认 `目标时长 / 分段数量 / 并发生成数 / 渲染质量 / 布局偏好` 都可调，并且关闭弹层后仍保留刚才设置。
4. 在亮色与暗色模式下分别检查 `/video/input` 和 `/video/:id/generating`，确认输入卡片、弹层、摘要 chip、等待页顶部状态岛与 section 预览区都可正常阅读，没有遮罩空白或颜色失真。
5. 从 `/video/input` 创建一条新视频任务，确认页面仍立即进入 `/video/:id/generating`，顶部灵动岛区域能显示总进度与阶段状态。
6. 等待 preview 首次可用，确认页面左侧能看到“摘要 / 分段讲解 / 分段试看”三段式内容，而不是只剩总进度条。
7. 当某个 section 进入 `ready` 后，点击顶部 track pill 或右侧 section 卡片，确认片段预览区出现该 section clip，且旁白试听入口可点击播放。
8. 在仍有 section 未完成时刷新 `/video/:id/generating`，确认已 ready 的 section 标记、摘要与 clip 预览不会丢失。
9. 若某个 section 显示 `failed` 或 `fixing`，确认只有该 section 卡片出现对应状态，整页不会立即切成任务失败页。
10. 等待任务最终完成，确认页面自动跳转 `/video/:id`，结果页播放的是完整成片，不再展示 section clip 作为成功态主播放器。
