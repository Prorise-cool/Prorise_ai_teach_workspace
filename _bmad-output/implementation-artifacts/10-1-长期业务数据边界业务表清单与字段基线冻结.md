# Story 10.1: 长期业务数据边界、业务表清单与字段基线冻结

## Story

As a 架构与开发协作团队，
I want 先冻结长期业务数据边界与业务表清单，
so that 各业务域在回写和查询时不会再争论“这个数据到底该放 Redis 还是 RuoYi”。

## Acceptance Criteria

1. 团队查看数据边界清单时，能明确区分哪些数据进入 Redis、哪些进入 RuoYi / MySQL、哪些进入 COS，学习记录、收藏、任务元数据与问答历史不再被误放为 Redis 长期数据。
2. 团队查看业务表清单时，至少能看到视频任务、课堂会话、Companion turn、artifact、whiteboard action、Evidence chat、quiz 结果、learning path、learning record、favorite 等表或等效实体，上层业务不再各自新建临时表结构。
3. 任一需要供学习中心回看或后台审计的数据在设计时都被分配到长期宿主，不允许以“后面再补表”为理由长期滞留在运行态缓存。

## Dev Notes

### Business Context

- Epic 10 是长期业务数据宿主层，`10.1` 与 `10.2` 是其余长期数据类 Story 的共同前置。
- 该 Story 不是学生端页面开发，也不是 Redis 运行态设计；它要先把“哪些数据必须长期落位”说清楚。
- Epic 4、5、6、7、8、9 的回写、学习中心聚合与后台查询都会消费这里冻结的数据边界。

### Technical Guardrails

- 严格遵循“三层数据策略”：运行态入 Redis、长期态入 RuoYi / MySQL、文件入 COS。
- Redis 仅用于运行态、认证态和事件缓存，且所有 Key 必须设置 TTL；长期业务数据不得仅保存在 Redis。
- 对学习中心可见、个人中心 / 设置可见、后台可查询 / 可导出 / 可审计的数据，必须定义长期宿主。
- 字段基线必须对齐统一状态枚举、日期格式和分页语义，避免后续聚合查询出现跨模块字段漂移。
- `SessionArtifactGraph` 相关索引需要被纳入长期宿主规划，避免 Companion 后期反向依赖引擎内部实现。
