# Story 0.2: 契约资产目录、命名规则与版本规则冻结

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 前后端协作团队，
I want 为契约、mock、示例 payload、错误码与状态枚举建立统一存放和命名规范，
so that 后续所有 Epic 的“契约冻结”都有可执行、可查找、可复用的落地点。

## Acceptance Criteria

1. 当任一业务域冻结契约时，团队能明确知道接口 schema、示例 payload、错误码字典、状态枚举和 mock 样例应该存放在哪里，不再把契约信息散落在聊天记录、页面注释或个人笔记中。
2. 当某个契约升级时，契约变更具备版本标识、变更说明和影响范围说明，前端不会在联调阶段被动发现后端字段暗改。
3. `contracts/` 与 `mocks/` 形成统一命名与层级规则，后续 Story 只能在既定规则下扩展，而不能各自发明目录结构。

## Tasks / Subtasks

- [ ] 冻结 `contracts/` 与 `mocks/` 目录规范（AC: 1, 3）
  - [ ] 明确业务域子目录命名规则，如 `auth/`、`task/`、`video/`、`classroom/`、`companion/`、`evidence/`、`learning/`、`center/`。
  - [ ] 约束 schema、错误码、状态枚举、示例 payload、mock fixtures 的建议文件类型与命名模式。
  - [ ] 明确 README、索引和版本说明的最小要求。
- [ ] 冻结版本与变更说明规范（AC: 2）
  - [ ] 定义契约版本号、破坏性变更标记和兼容性说明格式。
  - [ ] 明确每次变更必须记录影响 Story / 页面 / API 范围。
  - [ ] 禁止“直接改 schema 不留痕”。
- [ ] 冻结错误码与状态枚举资产规范（AC: 1, 2, 3）
  - [ ] 明确公共错误码、业务错误码与状态枚举的存放方式。
  - [ ] 规定示例成功 / 失败 payload 必须与错误码、状态枚举一并发布。
  - [ ] 明确前端不得通过自由文本 `message` 猜测语义。
- [ ] 补齐目录级模板与最小示例（AC: 1, 3）
  - [ ] 基于已有 `auth/` 资产提炼通用模板。
  - [ ] 为至少一个新业务域预留空模板，验证规则可扩展。

### Business Context

- `Story 0.2` 是整个项目“契约先行”的真正起点，没有它，后续所有“契约冻结”都无法被工程化消费。
- 这张卡的产出不仅服务前后端协作，也直接决定测试夹具、mock 数据、接口文档和 Story 文件如何互相引用。
- `Story 0.3` 与 `0.4` 都依赖这里定义的规则；前者消费 mock 资产，后者输出 schema 与示例 payload。

### Technical Guardrails

- 必须使用仓库根目录现有的 `contracts/` 与 `mocks/` 作为唯一资产落点，不另建并行目录。
- 契约资产至少覆盖：schema、示例 request/response、错误码、状态枚举、mock fixtures；只写自然语言说明不算完成。
- 契约变更必须保留版本与影响范围，禁止“直接覆盖旧文件但不说明破坏性变化”。
- mock 资产不是随手写的假数据，它必须与契约文件同源、可追踪、可被前端状态机消费。
