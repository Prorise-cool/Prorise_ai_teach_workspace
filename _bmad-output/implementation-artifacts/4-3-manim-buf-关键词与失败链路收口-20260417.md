# Story 4.3 Manim `buf` 关键词与失败链路收口

> **日期**: 2026-04-17  
> **范围**: `packages/fastapi-backend/app/features/video/pipeline/`  
> **状态**: review

## 背景

真实任务日志暴露了两层问题叠在一起：

1. Manim 渲染阶段生成了 `Mobject.next_to(..., buf=...)`，当前运行时 API 实际要求 `buff`，导致 section render 失败。
2. 失败落库链路又在 `RenderFailureStore` 里对同步 `redis.Redis` 客户端做了 `await`，把真正的渲染错误二次遮蔽成：
   - `TypeError: object int can't be used in 'await' expression`

这会直接误导排查方向，让人以为核心问题出在 Dramatiq / Redis，而不是 Manim 代码生成质量。

## 本次收口原则

### 1. `buf` 问题只在源头修，不在渲染代理层硬改

本次明确拒绝把 `buf -> buff` 的 AST 改写放进 `engine/agent.py`。原因是：

- `TeachingVideoAgent` 的职责是组织 render / patch retry / error reporting，不应该在运行时偷偷改生成代码。
- patch retry 本身已经会携带最新的 `api_codebook + shared specification`，具备足够的 prompt 上下文修正能力。
- 把语法兼容偷塞进代理层，会让后续排查失真，也会继续积累“模型出错靠运行时补锅”的坏味道。

### 2. 真实修复点回到 prompt / codebook

已在 `prompts/manimcat/api_codebook.py` 中完成以下收口：

- `method_param_codebook` 中不再出现 `buff: buf`
- `Mobject_methods / unique_args_index` 中相关调用全部改成 `buff`
- `SHARED_SPECIFICATION` 新增硬约束：`use the real keyword buff; never invent or abbreviate it as buf`

这意味着：

- 初次 codegen 会吃到正确约束
- patch retry 也会继续使用同一套约束

### 3. 失败链路必须只“记录错误”，不能“制造新错误”

`engine/render_failure.py` 现已兼容 sync / async 两类 Redis client：

- 同步 `redis.Redis`：直接取返回值
- 异步 Redis-like client：仅在返回 awaitable 时执行 `await`

同时，失败落库改成 best-effort：

- Redis 写入异常只记 warning
- 不再覆盖原始 render failure

### 4. Sanitizer 必须拿到完整上下文

`orchestrator._handle_section_failure()` 现会优先读取：

- `render_error.stderr`
- `render_error.stdout`
- `render_error.code`

这样 `sanitize_render_error()` 才能基于真实 stderr/code 生成更可靠的错误摘要和 snippet。

## 最终边界

后续如果再遇到 `unexpected keyword argument 'buf'`，优先顺序固定为：

1. 先看 `api_codebook.py` / prompt 规则是否退化
2. 再看 patch retry prompt 是否正确注入 `apiIndexModule + SHARED_SPECIFICATION`
3. 最后看模型输出是否仍然违背规则

不要再把这类语义修复塞回 `agent.py` 做运行时 AST 改写。

## 结论

这次收口后，视频管线对外暴露的失败语义重新变干净：

- 代码生成规则错误由 prompt / patch retry 承担修正
- render 代理只负责编排与透传错误
- failure store 不再把主错误掩盖成 Redis await 异常
