# Story 5.6: 课堂结果页中的幻灯片、讨论与白板浏览

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 正在学习主题内容的用户，
I want 在课堂结果页稳定浏览幻灯片、讨论与白板内容，
so that 我可以获得连续、可读的课堂体验。

## Acceptance Criteria

1. 当课堂生成已完成时，用户进入 `/classroom/:id` 必须能看到结构化幻灯片、讨论片段和白板内容，且主内容区与侧边 Companion 区域职责边界清晰。
2. 当课堂包含多角色讨论或复杂白板内容时，页面至少可以稳定展示多角色讨论片段和主要白板步骤；同一时刻白板主要内容不会严重破坏可读性。
3. 当页面运行在 mock 模式时，必须能分别查看内容完整态、白板降级态、讨论为空态、加载失败态与权限失败态，无需等待真实课堂生成引擎才开始验收布局与交互。
4. 当页面存在后续动作入口时，`Companion`、`Evidence / Retrieval` 与 `Learning Coach` 只能作为侧栏或后续入口出现，正式 quiz 不得插入课堂主内容流打断主叙事。

## Dev Notes

### Business Context

- 课堂结果页是 Epic 5 的最终消费页，决定用户是否真正获得“像课堂一样”的完整浏览体验。
- 本 Story 只拥有课堂主结果区与入口挂载位，不拥有 Companion、Evidence 或 Learning Coach 的完整实现权。
- 成品图是视觉基线，但成品图未展开的业务点仍必须按 Epic 5 与相关 Epic 的边界补足。

### Technical Guardrails

- 页面必须消费统一 adapter 与 mock 资产，禁止结果页单独解析原始接口返回。
- 结果页主叙事以课堂内容为中心，侧栏能力只能作为辅助入口或次区域。
- 白板 viewer 必须接受结构化降级结果，不能默认所有 whiteboard payload 都是高保真坐标流。
- 正式 quiz 入口只能以课后出口出现，不能嵌进课堂内容主区域。

- 结果页应优先在 `features/classroom/` 内形成 `page + components + styles + shared` 结构，而不是在 `app/` 中堆逻辑。
- `references/OpenMAIC/app/classroom/[id]/page.tsx` 可以帮助理解“课堂结果 / 回放页”作为独立页面容器的组织方式，但不等于我方页面职责。

### Testing Requirements

- 覆盖从等待页完成跳转与直接访问结果页的路由行为。
- 覆盖后续动作入口的挂载位可见，但不误触发正式 quiz 主流程。
