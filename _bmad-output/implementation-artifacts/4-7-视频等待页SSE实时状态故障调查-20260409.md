# Story 4.7: 视频等待页 SSE 实时状态故障调查

Status: investigated

日期: 2026-04-09

## 调查目标

- 现象：用户从 `/video/input` 进入 `/video/:id/generating` 后，后端 SSE 持续产生日志和事件，但前端等待页拿不到实时状态推进。
- 边界：本次只定位根因，不修复实现。
- 事实源优先级：以 `_bmad-output/`、前后端源码、现有测试结果为准。

## 结论摘要

本问题的主因不在后端“没有推”，而在前端 `student-web` 的 SSE 消费实现把 `text/event-stream` 当成普通 HTTP 文本响应一次性读完再解析，导致：

1. 后端只要保持长连接持续推送，前端的 `await response.text()` 就不会返回。
2. `useVideoTaskSse()` 中的 `for await` 因为上游异步迭代器一直拿不到事件，页面不会收到任何实时进度。
3. 同一条链路上的降级 polling 也不会触发，因为控制流被卡在“等待整段 SSE body 结束”这一步。

此外还存在一个次级契约问题：

1. 后端把 `connected` / `heartbeat` 事件以 `ensure_identity=False` 输出。
2. 这两类事件缺少 `id` / `sequence`。
3. 前端 parser 明确要求每个 SSE 事件必须有合法 `id` / `sequence`，因此会直接丢弃 `connected` / `heartbeat`。

## 根因一：前端没有按流式方式消费 SSE

### 关键代码链路

- `packages/student-web/src/features/video/hooks/use-video-task-sse.ts`
  - `startStream()` 调用 `resolveTaskEventStream(...).streamTaskEvents(...)`
- `packages/student-web/src/services/sse/index.ts`
  - `createRealTaskEventStream()` 在 `streamTaskEvents()` 内调用 `streamSseAttempt()`
  - `streamSseAttempt()` 调用 `fetch(...)` 后立刻执行 `parseTaskEventResponse(response, logger)`
  - `parseTaskEventResponse()` 对 `text/event-stream` 分支执行 `await response.text()`

### 为什么这会卡住

SSE 的设计是“连接保持打开，边到达边消费”。但当前实现的行为是：

1. 等待整个响应体结束。
2. 结束后再把完整文本交给 `eventsource-parser`。
3. 再把解析后的事件数组一次性返回给上层。

只要后端持续保持连接：

- `response.text()` 不会 resolve。
- `streamSseAttempt()` 不会返回。
- `for await (const event of iterable)` 永远拿不到第一条事件。

因此页面表面现象就是“后端一直在推，前端却完全没更新”。

### 运行时验证

使用 Node 构造了一个 `text/event-stream` 响应：

- 第 1 条事件立即写入
- 第 2 条事件在 100ms 后写入
- 连接在约 2000ms 后才关闭

验证结果表明 `response.text()` 在流关闭前不会返回，输出耗时约 `1991ms`，而不是在第 1 条事件到达时立即产出。这与当前前端实现的阻塞行为一致。

## 根因二：`connected` / `heartbeat` 被前端 parser 丢弃

### 后端输出行为

- `packages/fastapi-backend/app/api/routes/tasks.py`
  - `stream_task_events()` 发送 `connected` 时调用 `encode_sse_event(..., ensure_identity=False)`
  - 发送 `heartbeat` 时同样使用 `ensure_identity=False`
- `packages/fastapi-backend/app/core/sse.py`
  - `encode_sse_event()` 在 `ensure_identity=False` 时不会补 `id` / `sequence`

### 前端解析行为

- `packages/student-web/src/services/sse/index.ts`
  - `normalizeTaskEventPayload()` 在 `eventId` 不是字符串或 `sequence === null` 时直接丢弃事件

### 影响

- `connected` 不会进入 `useVideoTaskSse()`，`setSseConnected(true)` 实际不会被触发
- `heartbeat` 不会进入前端恢复语义，心跳保活与事件连续性判断会失真
- 这不是“完全收不到实时进度”的主因，但说明前后端对 Story 2.5 的 SSE 契约没有完全对齐

## 为什么现有测试没有发现

### 前端测试模型偏差

- `packages/student-web/src/services/sse/task-event-stream.test.ts`
  - 真实流测试使用 `new Response("完整 SSE 文本", ...)`
  - 这类响应在创建时已经是完整 body，`response.text()` 会立即结束
  - 它验证了“离线文本解析”，没有验证“长连接流式到达”

### 后端测试模型偏差

- `packages/fastapi-backend/tests/api/tasks/test_task_recovery_routes.py`
  - 验证了 `StreamingResponse` 会持续产出 frame
  - 也验证了补发 / 心跳 / 终态
  - 但没有和前端当前的 fetch + `response.text()` 消费方式联调

### 次级契约偏差未被挡住

- 前端 parser 测试里的 `connected` mock 事件带了合法 `id`
- 后端真实实现里的 `connected` 恰恰没有 `id`
- 因此前端单测和后端单测各自通过，但集成语义仍然错位

## 建议的后续修复方向（仅记录，不在本次执行）

1. 前端 `services/sse/index.ts` 改为基于 `response.body.getReader()` 的增量读取，并按 chunk 喂给 `eventsource-parser`。
2. 后端 `connected` / `heartbeat` 事件补齐契约要求的 `id` / `sequence`，避免被前端 parser 丢弃。
3. 增加一条真正的集成测试：后端保持连接打开、分两批推送事件，断言前端能在连接关闭前看到第一批进度。
