## 1. Epic Review

### 1.1 结论

Epic 0 已达到规划目标，且其价值已经被后续 Epic 真实消费，不是停留在文档层的“纸面底座”。

已确认的完成事实：

- `sprint-status.yaml` 中 `epic-0` 为 `done`，`0.1` 至 `0.6` 六张 Story 全部为 `done`
- `13-epic-0.md` 定义的六类交付物均已有落地证据：目录骨架、契约 / mock 规范、adapter / mock 基线、schema / OpenAPI 基线、追踪骨架、DoR / DoD 与并行门禁
- `Epic 1` 的 `1.1`、`1.2`、`1.3` 已明确复用 `Epic 0` 产物，说明并行开发轨道已经开始工作

### 1.2 本 Epic 做对了什么

#### 1. 先冻结边界，再推进实现

`Epic 0` 明确把目标限定为“开发轨道”，而不是偷带业务实现。这一点非常关键。

证据：

- `13-epic-0.md` 明确把“具体业务接口、页面实现、业务表设计”排除在范围外
- `0-1-monorepo-基础目录与工程骨架冻结.md` 明确要求只冻结结构和最小运行面
- `0.1` 的完成记录里专门把过早冻结的 `task_framework` 状态枚举与错误码回收为中性占位，避免越界承接 `Epic 2`

这说明团队在最容易“顺手多做一点”的地方，成功守住了边界。

#### 2. 真正建立了“契约先行、双端并行”的工程底座

`Epic 0` 最有价值的产出，不是某个单点文件，而是把“契约、mock、adapter、schema”变成了一套可复用工作流。

证据：

- `0.2` 冻结了 `contracts/`、`mocks/`、版本规则、错误码规则、状态枚举规则
- `0.3` 冻结了前端 `adapter + mock handler + fixture + SSE mock + 环境切换` 边界
- `0.4` 冻结了后端 `{code, msg, data}` / `{code, msg, rows, total}`、schema、OpenAPI、示例 payload 输出
- `1.1` 明确依赖 `0.2`、`0.3`、`0.4`
- `1.2` 已在真实登录链路上复用契约、adapter、i18n / theme 基线推进正式页面

复盘判断：`真实并行，而不是口头并行` 这一设计目标已经被证明成立。

#### 3. 把运行态可观测性放到了足够早的位置

很多团队会把 `request_id`、`task_id`、统一日志字段拖到业务出问题后再补。`Epic 0` 没有犯这个错。

证据：

- `0.5` 已冻结 `request_id / task_id / 日志追踪骨架`
- 对应测试要求覆盖普通请求、任务请求、失败请求与 SSE 事件追踪
- `Epic 2` 的故事定义已经开始依赖 `0.5` 的追踪骨架
- `1.3` 也已经把认证一致性、Redis 在线态探针与真实联调证据纳入收口

这意味着后续长任务链路不是“先生成、再补排障”，而是从第一层基础设施开始具备可追踪性。

#### 4. 把 Done 的判断从口头共识升级为工程规则

`0.6` 是 `Epic 0` 的收口点，也是整个项目后续是否会回到“伪完成”状态的分水岭。

证据：

- `docs/01开发人员手册/004-开发规范/0008-story-交付门禁与-dor-dod.md`
- `docs/01开发人员手册/004-开发规范/0009-并行开发联调门禁.md`
- `_bmad-output/implementation-artifacts/story-template-checklist.md`

这些产物把以下问题制度化了：

- 什么叫 Ready
- 什么叫 Done
- 什么叫 mock 可开发
- 什么叫真实接口可联调
- 什么叫可以合并
- 什么叫可以发布

这是 `Epic 0` 最重要的长期资产之一。

### 1.3 本 Epic 暴露出的系统性问题

#### 1. Infrastructure Story 很容易越界成“偷跑实现”

`0.1` 已经出现过一次苗头：为了让骨架更“像未来”，差点提前冻结后续任务框架语义。虽然最后纠正了，但这说明基础 Story 天然有越界风险。

系统性结论：

- 基础 Story 必须显式写出非目标
- Reviewer 需要把“有没有提前替后续 Epic 做决定”作为固定审查项

#### 2. 底座型 Epic 的价值，不能只靠“文件存在”判定

如果只看 `0.2` 到 `0.6` 是否有文件，很容易得到一个形式上完成、实质上没人用的假结论。幸运的是，当前不是这种情况，因为 `Epic 1` 已开始消费。

但这也暴露出一个规则：

- 以后基础 Story 判定 Done 时，除了文件落地，还应检查是否已有明确消费者路径

#### 3. Retrospective 收口动作偏晚

当前 `Epic 0` 已经完成，`Epic 1` 也已进入 `in-progress`，而 `epic-0-retrospective` 在本次复盘前仍是 `optional`。

这不是产品风险，但它是流程风险：

- 复盘晚做，问题会被带入下一 Epic
- Action Items 容易失去执行窗口
- Sprint 状态会出现“Epic done 但复盘未闭环”的管理空洞

### 1.4 本 Epic 的关键经验

1. 底座类 Epic 必须用“边界纪律”衡量成功，而不只是看文件数量。
2. `contracts + mocks + adapter + schema` 必须作为一个完整闭环交付，拆开交付就会重新回到猜字段开发。
3. `request_id / task_id / 日志字段` 越早进入链路，后续异步任务域越稳定。
4. DoR / DoD 与联调门禁必须写进规范和清单，否则团队会自然滑回 happy path 优先。
5. 基础 Epic 的真正完成标志，是下游 Epic 能直接复用，而不是回头重造一套平行机制。

### 1.5 上一轮 Retrospective 连续性

无上一 Epic retrospective 可追溯。`Epic 0` 是当前上半区设计里的首个复盘基线。

- 需要继续严格执行 `Epic 0` 留下的门禁，否则 `Epic 1` 会退化成局部成功、全局失配

### 2.2 对 Epic 1 的准备建议

#### 已经准备好的部分

- 认证契约与 mock 基线已经存在
- 正式认证页已验证真实登录 / 注册 / 回跳链路
- 认证一致性 Story 已进入 `review`
- DoR / DoD、并行联调门禁已经有正式文档承接

#### 仍需持续盯住的部分

1. `401 / 403 / returnTo / session persist` 必须被视为回归基线，而不是某个单 Story 的局部实现。
2. 后续受保护页面 Story 不得绕过现有 `RequireAuthRoute`、adapter、统一错误处理机制另起一套。

4. 所有影响认证、路由守卫、持久化与回跳的变更，必须继续做真实浏览器验证，且以 `http://127.0.0.1:5173` 为准。

#### Process

   - 完成标准：`epic-X-retrospective` 不再滞后于下一个 Epic 的正式推进

2. 所有 Infrastructure Story 在准备阶段显式写出“非目标 / 不承接内容”。
   - Owner: `Winston (Architect)` + `Bob (Scrum Master)`
   - 完成标准：Story 文档内存在可审查的 `Out of Scope` 或等效约束

3. 所有 Contract / Infrastructure Story 的 Done 判断，增加“已识别下游消费者路径”检查。
   - Owner: `Bob (Scrum Master)` + `Amelia (Developer Agent)`
   - 完成标准：Story 文档能明确说明谁会直接消费该资产

#### Technical / Quality

4. 将认证相关真实运行态验证作为固定回归包。
   - Owner: `Quinn (QA Engineer)` + 前端 Owner
   - 触发点：任意涉及 `/login`、受保护路由、`401/403`、`returnTo`、本地会话持久化的合并前检查
   - 完成标准：至少覆盖登录成功、未登录重定向、已登录访问 `/login`、登出回退、权限不足分流

5. 新业务域首次冻结契约时，同时交付 contract、mock、adapter parity 证据。
   - Owner: `Amelia (Developer Agent)` + `Quinn (QA Engineer)`
   - 触发点：`task`、`video`、`classroom` 等域的首张 Contract Story
   - 完成标准：contract、mock、adapter、测试四者口径一致

6. 若 Story 在开发中发生范围纠偏，先改 Story 文档与验收口径，再继续实现。
   - Owner: `John (Product Manager)` + `Bob (Scrum Master)`
   - 完成标准：不再出现代码已转向、Story 仍写旧目标的状态漂移

### 2.4 Critical Path

3. 后续入口级 Story 必须遵守 `0008-story-交付门禁与-dor-dod.md` 与 `0009-并行开发联调门禁.md`，不得跳过。

### 2.5 是否需要更新下一 Epic 计划

当前判断：`否`。

原因：

- `Epic 0` 的设计目标与落地产物一致
- `Epic 1` 已经开始真实消费这些底座资产
- 尚未发现会推翻 `Epic 1` 目标或依赖模型的结构性问题

但有一个明确提醒：

- `Epic 1` 的范围必须继续保持 Story 级纪律，不能因为底座已具备就默认可以一张 Story 吞下更多目标

## 3. Readiness Assessment

### Epic 0 是否真正完成

结论：`是`，但属于“基础设施完成”，不是“用户功能完成”。

已满足：

- Story 维度：`6 / 6` 完成
- 规划维度：满足 `13-epic-0.md` 的 Exit Criteria
- 复用维度：已被 `Epic 1` 实际消费
- 规范维度：DoR / DoD、并行门禁、模板清单均已沉淀

### 剩余风险

1. 若团队后续不按 `0008` / `0009` 执行，`Epic 0` 的价值会被人为绕开。
2. 若基础 Story 不继续写清非目标，仍可能出现“提前替后续 Epic 做决定”的范围污染。
3. 若 retrospective 继续滞后，流程 learnings 会晚于实际开发节奏。

### 最终判断

- `Epic 0`：完成，可作为后续 Epic 的稳定底座
- `Epic 1`：可继续推进，无需因 `Epic 0` 结果重开规划
- 管理要求：从本次开始，把 retrospective 状态回写视为 Epic 收口的一部分，而不是可选装饰动作
