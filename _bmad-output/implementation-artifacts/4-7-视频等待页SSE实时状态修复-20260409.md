# Story 4.7: 视频等待页 SSE 实时状态修复

Status: review

日期: 2026-04-09

## 修复背景

- 故障调查记录：[`4-7-视频等待页SSE实时状态故障调查-20260409.md`](./4-7-视频等待页SSE实时状态故障调查-20260409.md)
- 现象：用户从 `/video/input` 进入 `/video/:id/generating` 后，后端 SSE 持续推送，但前端等待页长时间停留在初始态，无法实时推进阶段与进度。

## 本次修复范围

- 仅修改 `student-web` 侧 SSE transport 与测试，不调整 FastAPI 行为。
- 本次目标是让等待页恢复“连接不断开时也能实时消费事件”的能力，并兼容当前后端 `connected` / `heartbeat` 事件缺少契约身份字段的现实输出。

## 修复实现

### 1. 改为增量消费 `text/event-stream`

- 文件：`packages/student-web/src/services/sse/index.ts`
- 改动：新增 `streamTaskEventResponse(...)`，基于 `response.body.getReader()` + `TextDecoder` + `eventsource-parser` 按 chunk 解析 SSE。
- 效果：前端不再等待 `response.text()` 直到整条长连接结束，而是在首个 chunk 到达后即可产出 `connected` / `progress` / `completed` 等事件。

### 2. 拆分“打开连接”和“消费事件”职责

- 文件：`packages/student-web/src/services/sse/index.ts`
- 改动：把原来的 `streamSseAttempt(...)` 重构为 `openSseAttempt(...)`，只负责拿到 `Response` 与恢复头信息，再由上层异步迭代消费。
- 效果：`createRealTaskEventStream()` 能在连接存续期间逐条 `yield` 事件，不再被整段响应体阻塞。

### 3. 为 `connected` / `heartbeat` 增加 transient identity 兼容

- 文件：`packages/student-web/src/services/sse/index.ts`
- 改动：当后端发送 `connected` / `heartbeat` 但未提供合法 `id` / `sequence` 时，前端会生成形如 `transient:{taskId}:{event}:{n}` 的临时 id，仅用于本地标准化与状态流推进。
- 效果：等待页可以接收到连接建立与保活事件，不会因这两类消息被 parser 丢弃而一直停在“未连接”状态。

### 4. 严格隔离 transient id 与恢复游标

- 文件：`packages/student-web/src/services/sse/index.ts`
- 改动：`pickLatestEventId(...)` 仅接受标准格式 `{taskId}:evt:{sequence}` 的事件 id；transient id 不会写入 `Last-Event-ID`，也不会驱动重连序号。
- 效果：兼容当前后端输出的同时，不破坏 Story 2.5 约定的恢复语义。

## 残留风险与后续建议

- FastAPI 当前仍然会为 `connected` / `heartbeat` 输出无 `id` / `sequence` 的事件，这与 [`contracts/tasks/sse-events.md`](../../contracts/tasks/sse-events.md) 仍不完全一致。
- 当前前端兼容层已经足以恢复页面实时反馈，但后续若要完全回到契约基线，仍建议后端补齐这两类事件的标准身份字段。
