## 背景

`b8ff177339d` 已经完成两件关键能力：

1. 等待页的真实取消链路已经闭环。
2. 输入页顶部已经有 bell + dropdown 的“进行中任务中心”。

但用户从 `/video/:id/generating` 返回 `/video/input` 后，输入页主体里仍缺少一个显式的“当前任务状态”承接区。结果就是：

- 任务虽然还在运行，但只能点右上角 bell 才能看到。
- “返回工作区后还能继续查看/取消”的关系没有在输入页显式落下来。
- 设计稿里等待页通过 `focusTask` / `toast` 回传输入页的承接感没有接住。

## 本次补齐内容

1. 输入页主体新增“当前任务”状态卡。
   - 直接展示当前任务标题、状态、进度条、阶段文案。
   - 提供“继续查看”和“取消任务”两个主操作。
   - 视觉沿用现有输入页 / 任务中心的圆角玻璃卡、secondary surface、brand progress bar。
2. 输入页支持读取等待页回传的 query 参数。
   - `focusTask`：优先聚焦对应任务。
   - `toast=returned`：提示用户任务仍在继续，可从输入页继续查看或取消。
   - `toast=cancelled`：提示用户任务已取消，可重新编辑后发起新任务。
3. 等待页“返回工作区 / 取消任务”改为携带输入页承接参数。
   - 返回工作区：`/video/input?focusTask=<taskId>&toast=returned`
   - 取消任务成功：`/video/input?toast=cancelled&taskId=<taskId>`
4. 输入页右上角 bell 任务中心改为始终可见。
   - 无活跃任务时也保留入口，和设计稿空态一致。
   - 红点仍只在存在活跃任务时显示。

## 与 `b8ff177339d` 的关系

- `b8ff177339d` 解决的是“任务中心存在”和“等待页能取消”。
- 本次补的是“输入页主体里显式显示当前任务状态，并承接等待页返回后的任务继续/取消操作”。
- 也就是说，这不是推翻 `b8ff177339d`，而是把它遗漏的最后一段输入页承接体验补齐。

## 实施细节

### 输入页

- `VideoInputPage` 继续复用既有 `useVideoWorkspaceTasks` 查询结果，不新增第二套状态来源。
- 当前任务优先级：
  1. `focusTask`
  2. 活跃任务列表第一项
- 当前任务卡的取消动作继续走现有 `cancelTaskMutation`，避免重复实现。
- 顶栏任务中心与当前任务卡共享同一批数据，因此：
  - bell 里看的是“全部活跃任务”
  - 主体卡看的是“当前聚焦任务”

### 等待页

- `useCancelVideoTask` 仍负责真实取消 mutation。
- 本次只调整 `returnTo` 与 `handleReturn` 的输入页落点，不改等待页原有状态机、preview、SSE、结果页 CTA 逻辑。

## 后续约束

- 输入页当前任务卡与 bell 任务中心必须继续共用同一批活跃任务数据，不能分叉成两套状态源。
- 若后续实现“查看全部任务”页面，本次 `focusTask` / `toast` 承接参数仍应保持兼容，避免等待页返回体验再次断裂。
