# Story 4.7: 视频等待页前端状态机、恢复与降级

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 等待结果的用户，
I want 在等待页实时看到视频生成进度并在断线后恢复状态，
so that 我知道任务正在推进且不会因为刷新或网络抖动失去上下文。

## Acceptance Criteria

1. 视频任务进入执行中状态后，用户进入 `/video/:id/generating` 时，页面展示统一等待壳层，包含当前阶段标签、阶段说明文案、总体进度条与失败/重试提示区域，至少覆盖 `understanding`、`storyboard`、`manim_gen`、`manim_fix`、`render`、`tts`、`compose`、`upload` 等关键阶段。
2. 页面在 mock 模式下可稳定消费 mock SSE 或 mock status 状态流，所有关键状态（处理中、修复中、失败、恢复中、降级查询中）都可被演示与验收。
3. SSE 中断、页面刷新或浏览器恢复会话时，客户端优先通过 SSE `snapshot` 事件恢复状态，若 SSE 不可用则自动降级到 `/api/v1/video/tasks/:id/status` 轮询查询（间隔 3s），页面不会失去状态。
4. 任务失败时，页面展示可解释失败信息（基于 `failedStage` 和 `errorCode`）、可用的下一步动作（重新生成按钮或返回输入页按钮），不会继续伪装成处理中或自动跳到结果页。
5. 任务成功完成时，页面接收到 `task:completed` 事件后自动跳转到 `/video/:id` 结果页，跳转前展示短暂成功提示（≤2s）。
6. 等待页状态机使用 `zustand` store 管理，状态包含 `currentStage`、`stageLabel`、`progress`、`status`、`error`、`sseConnected`、`degradedToPolling`，store 支持序列化以配合 SSE 恢复。
7. `manim_fix` 阶段在等待页展示特殊 UI（如"正在尝试修复…第 1/2 次"），用户能感知系统正在积极修复而非卡死。

### Business Context

- 等待页是用户对视频生成过程的唯一可视窗口，stage 级别的进度展示直接影响用户等待耐受度。
- `manim_fix` 的修复尝试展示让用户感知系统在积极工作，而非卡死不动——这对降低用户流失率至关重要。
- SSE 断线恢复和 status 降级是网络不稳定环境下的保底策略，确保用户刷新页面后不会看到空白。
- 等待页与 Story 3.5 存在交集：Story 3.5 建立了等待页的基础壳层和 SSE 消费骨架，本 Story 扩展为完整的多 stage 状态机和修复态 UI。

### Technical Guardrails

- 状态机 store 使用 `zustand`，不使用 `persist` 中间件——状态恢复依赖 SSE snapshot 或 status 查询，不依赖 localStorage。
- SSE 事件消费必须通过 Story 2.6 建立的统一 SSE client，不得重新实现 EventSource 连接管理。
- 降级轮询使用 `@tanstack/react-query` 的 `useQuery` + `refetchInterval`，不使用 `setInterval` 手写轮询。
- 进度条动画使用 Motion（`animate` prop），不使用 CSS `transition` 避免与 Motion 冲突。
- stage 标签到中文显示名的映射必须由 Story 4.1 冻结的 `displayLabel` 驱动，不得在前端硬编码映射表。
- 失败态的 `failedStage` → 中文说明映射应集中在一个 mapping 文件中，便于后续扩展。
- 页面级样式使用 feature SCSS，不在页面组件中内联大段 Tailwind 类。
- 等待页路由 `/video/:id/generating` 必须为受保护路由（需登录），且任务 ID 不属于当前用户时展示 403。
