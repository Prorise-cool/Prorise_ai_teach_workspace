# Story 1.1: 统一认证入口中的注册与登录

Status: ready-for-dev
Story Key: 1-1-统一认证入口中的注册与登录

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 学生或教师，
I want 通过统一认证入口完成注册和登录，
so that 我可以快速进入小麦平台开始学习。

## Acceptance Criteria

1. **Given** 访客位于首页并打开认证入口，**When** 访客提交有效注册信息，**Then** 系统通过 RuoYi 承接的认证流程完成账户创建，**And** 在不强制跳转到无关页面的前提下返回清晰成功反馈。
2. **Given** 已注册用户打开认证入口，**When** 用户提交有效登录凭证，**Then** 系统完成登录并返回已认证的首页上下文，**And** 为后续受保护访问建立可复用的认证会话。

## Tasks / Subtasks

- [ ] 搭建 ToC 认证入口壳层（AC: 1, 2）
  - [ ] 在 `packages/landing-page/` 创建或补齐用户端 React 19 SPA 的认证入口，不要把用户端登录直接做进 `packages/ruoyi-plus-soybean-master`。
  - [ ] 在首页 `/` 提供登录、注册 CTA，并以对话框方式打开，不使用整页跳转作为主流程。
  - [ ] 支持登录态下关闭对话框并返回首页上下文，未登录态保持双入口首页可用。

- [ ] 实现注册表单与成功回流（AC: 1）
  - [ ] 表单至少覆盖 `tenantId`、`username`、`password`、`confirmPassword`、`code`、`uuid`、`userType`，其中默认 `tenantId='000000'`、`userType='sys_user'`。
  - [ ] 如果租户开关开启，则先调用 `/auth/tenant/list` 加载租户选项；否则使用默认租户值。
  - [ ] 如果验证码开关开启，则调用 `/auth/code` 获取验证码和 `uuid`，注册失败后刷新验证码。
  - [ ] 注册成功后展示明确成功反馈，并切回登录态；不要默认自动登录，除非接口契约与产品决定一起调整。

- [ ] 实现登录流程、Token 持久化与用户上下文初始化（AC: 2）
  - [ ] 登录请求必须带上 `clientId`、`grantType='password'`、`tenantId`，并调用 RuoYi `/auth/login`。
  - [ ] 登录成功后保存 `access_token`、`refresh_token`，随后调用 `/system/user/getInfo` 初始化当前用户上下文。
  - [ ] 前端统一通过 `Authorization: Bearer <token>` 访问受保护资源，并为后续受保护页面守卫预留状态读取入口。
  - [ ] 登录失败时保留表单上下文并刷新必要验证码，不清空用户刚输入的非敏感字段。

- [ ] 对齐异常、401 与登出清理语义（AC: 2）
  - [ ] 与规划文档一致，未授权场景统一视为 401 语义：清理 Token、回到未认证首页状态，并提示重新登录。
  - [ ] 登出调用 `/auth/logout`，随后清理本地认证存储；不要保留失效会话残留。
  - [ ] 不在用户端自建第二套鉴权协议，不绕开 RuoYi 现有 JWT + Redis 在线态方案。

- [ ] 补齐测试与联调证明（AC: 1, 2）
  - [ ] 为认证表单校验、登录请求参数拼装、注册成功回流、401 清理、Token 持久化编写测试。
  - [ ] Mock RuoYi 认证接口，覆盖租户开启/关闭、验证码开启/关闭、注册失败、登录失败、Token 失效等边界条件。
  - [ ] 增加最小联调清单，证明首页对话框、注册、登录、登出与 `getInfo` 能串成闭环。

## Dev Notes

### Story Context Summary

- 本 Story 是整个产品链路的起点。它不只是“做一个登录页”，而是为后续首页双入口、FastAPI 受保护接口访问、历史记录、个人中心等能力建立统一认证基线。
- 当前仓库没有 `sprint-status.yaml`，也没有任何既有 implementation story 文件；本文件按默认顺序落地为第一个 story。
- 最近 5 个提交全部集中在规划文档与仓库整理，说明认证主链路仍应被视为“未实现”，不要把文档存在误读成代码已完成。

### Reuse First, Do Not Reinvent

- 用户管理与认证源头在 `packages/RuoYi-Vue-Plus-5.X/`，不要在 FastAPI 或用户端重新发明用户表、密码校验、RBAC 规则或长期认证状态模型。
- `packages/ruoyi-plus-soybean-master` 是现成的 RuoYi + Soybean 前端参考实现，当前最有价值的是复用其接口契约、参数字段、加密方式、Token 存取与 401 清理语义，而不是直接复用其整页 `/login/:module` 路由体验。
- `packages/fastapi-backend/` 当前仍是占位状态，没有现成的用户管理实现。本 Story 不应该把注册/登录主流程落到 FastAPI。

### Architecture Guardrails

- 规划架构已明确规定：ToC 用户前端放在 `packages/landing-page/`，技术栈为 `React 19 + Vite 6.x + shadcn/ui CLI v4 + Tailwind CSS v4`。当前仓库尚未存在该包，如需实现本 Story，可在该路径初始化最小可运行骨架。  
  [Source: `_bmad-output/planning-artifacts/architecture.md`，章节 “FR-UI 前端 UI” / “Monorepo 目录结构”]
- 认证职责分层必须保持：前端持有 Token，FastAPI 只校验共享认证态，RuoYi 负责用户、RBAC 与长期业务数据。  
  [Source: `_bmad-output/planning-artifacts/architecture.md`，章节 “认证鉴权” / “双后端分层”]
- FastAPI 与 RuoYi 的协作必须通过防腐层完成；不要让 FastAPI 直接依赖 RuoYi 领域模型。  
  [Source: `_bmad-output/planning-artifacts/architecture.md`，章节 “FastAPI 与 RuoYi 的协作关系”]
- Redis 只承担运行态与在线态，不承担长期业务数据；所有相关 Key 必须设置 TTL。  
  [Source: `_bmad-output/planning-artifacts/architecture.md`，章节 “Redis 使用边界”]
- 前端请求与响应语义要与 RuoYi 保持一致，统一消费 `{code, msg, data}`。  
  [Source: `_bmad-output/planning-artifacts/architecture.md`，章节 “API 响应格式”]

### UX Guardrails

- 首页 `/` 是双入口页面，登录/注册必须以对话框方式打开，而不是强制跳转独立认证页。  
  [Source: `_bmad-output/planning-artifacts/ux-design-specification.md`，章节 “首页/双入口页面” / 行为表 “登录/注册：点击 → 弹出对话框，不跳转页面”]
- 用户端体验应保持“极简首页 + 非阻塞认证”，认证失败不能破坏首页双入口可用性。  
  [Source: `_bmad-output/planning-artifacts/ux-design-specification.md`，章节 “Effortless Interactions” / “首页/双入口页面”]
- 本 Story 只需要“统一认证入口中的注册与登录”闭环，不包含社交登录、短信登录、找回密码、完整个人资料设置等增强功能。

  [Source: `packages/ruoyi-plus-soybean-master/src/router/routes/index.ts`]

- 参考前端本地清理逻辑只移除 `token` 与 `refreshToken`，并在未认证时跳回登录态。  
  [Source: `packages/ruoyi-plus-soybean-master/src/store/modules/auth/shared.ts`]
- 参考请求层用 `Authorization: Bearer <token>` 访问受保护资源，并支持基于环境变量的 401 / token 失效清理。  
  [Source: `packages/ruoyi-plus-soybean-master/src/service/request/shared.ts`；`packages/ruoyi-plus-soybean-master/src/service/request/index.ts`]

### API Contract Requirements

- 登录接口：`POST /auth/login`
  - 请求头：`isToken: false`、`isEncrypt: true`、`repeatSubmit: false`
  - 关键字段：`username`、`password`、`tenantId`、`clientId`、`grantType`
  - 后端会校验 `clientId` 与 `grantType` 是否匹配可用客户端，并校验租户。  
  [Source: `packages/ruoyi-plus-soybean-master/src/service/api/auth.ts`；`packages/ruoyi-plus-soybean-master/src/typings/api/auth.d.ts`；`packages/RuoYi-Vue-Plus-5.X/ruoyi-admin/src/main/java/org/dromara/web/controller/AuthController.java`]
- 注册接口：`POST /auth/register`
  - 请求头：`isToken: false`、`isEncrypt: true`、`repeatSubmit: false`
  - 关键字段：`tenantId`、`username`、`password`、`confirmPassword`、`code`、`uuid`、`userType`
  - 后端会校验注册功能开关、验证码与用户名重复，并使用 BCrypt 持久化密码。  
  [Source: `packages/ruoyi-plus-soybean-master/src/service/api/auth.ts`；`packages/ruoyi-plus-soybean-master/src/views/_builtin/login/modules/register.vue`；`packages/RuoYi-Vue-Plus-5.X/ruoyi-admin/src/main/java/org/dromara/web/service/SysRegisterService.java`]
- 用户信息接口：`GET /system/user/getInfo`
  - 登录成功后必须调用，用于初始化 `user`、`roles`、`permissions`。  
  [Source: `packages/ruoyi-plus-soybean-master/src/service/api/auth.ts`；`packages/ruoyi-plus-soybean-master/src/typings/api/auth.d.ts`]
- 登出接口：`POST /auth/logout`
  - 前端先走接口，再清本地认证存储。  
  [Source: `packages/ruoyi-plus-soybean-master/src/service/api/auth.ts`；`packages/RuoYi-Vue-Plus-5.X/ruoyi-admin/src/main/java/org/dromara/web/controller/AuthController.java`]
- 租户与验证码辅助接口
  - `GET /auth/tenant/list`
  - `GET /auth/code`
  - 两者是否启用都必须做显式兼容，不要把“多租户/验证码永远存在”写死在 UI。  
  [Source: `packages/ruoyi-plus-soybean-master/src/service/api/auth.ts`；`packages/ruoyi-plus-soybean-master/src/views/_builtin/login/modules/pwd-login.vue`]

### Encryption and Transport Guardrails

- RuoYi 当前认证接口使用 `@ApiEncrypt`，现有 Soybean 请求层在 `VITE_APP_ENCRYPT=Y` 时，会用 AES + RSA 组合加密请求体，并通过自定义 header 传递加密密钥。用户端新实现如果接入同一网关/后端，必须沿用这一套机制，不允许自己发明第二套加密协议。  
  [Source: `packages/RuoYi-Vue-Plus-5.X/ruoyi-admin/src/main/java/org/dromara/web/controller/AuthController.java`；`packages/ruoyi-plus-soybean-master/src/service/request/index.ts`]
- 如果开发环境暂时关闭加密，也必须保持“接口字段、Bearer 头、成功码判定、错误码清理”这几项语义不变，避免 dev/prod 两套认证行为。

### Latest Technical Notes

- React 19 已稳定发布；新建 `packages/landing-page/` 时必须按 React 19 约束初始化，避免从旧 React 18 模板直接复制。  
  [Source: https://react.dev/blog/2024/12/05/react-19]
- shadcn/ui 官方当前支持 Vite 场景，建议通过 CLI 或 existing project 手册初始化，再添加 `dialog`、`form` 等组件，不要手写一套平行 UI 基础库。  
  [Source: https://ui.shadcn.com/docs/installation]  
  [Source: https://ui.shadcn.com/docs/components/dialog]  
  [Source: https://ui.shadcn.com/docs/components/form]

  [Source: https://fastapi.tiangolo.com/tutorial/security/oauth2-jwt]
