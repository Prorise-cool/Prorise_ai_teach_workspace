# Story 1.1: 统一认证入口中的注册与登录

Status: in-progress
Story Key: 1-1-统一认证入口中的注册与登录

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 学生或教师，
I want 通过统一认证入口完成注册和登录，
so that 我可以快速进入小麦平台开始学习。

## Acceptance Criteria

1. **Given** 访客位于首页并打开认证入口，**When** 访客提交有效注册信息，**Then** 系统通过 RuoYi 承接的认证流程完成账户创建，**And** 在不强制跳转到无关页面的前提下返回清晰成功反馈。
2. **Given** 已注册用户打开认证入口，**When** 用户提交有效登录凭证，**Then** 系统完成登录并返回已认证的首页上下文，**And** 为后续受保护访问建立可复用的认证会话。

## 审查回写（2026-03-27）

- 本 Story 后续执行重点是复用既有 seam 打通 RuoYi 契约、补齐规范中文注释、收敛测试目录，并把“逻辑验证页”和“正式页面交付”边界写死。
- 正式首页布局、正式认证入口视觉还原、Story 文案 / 调试信息清理，应与 Story 1.4 / Story 1.6 配套完成。

## Dev Notes

### Story Context Summary

- 本 Story 是整个产品链路的起点。它不只是“做一个登录页”，而是为后续首页双入口、FastAPI 受保护接口访问、历史记录、个人中心等能力建立统一认证基线。
- 当前仓库已经有 `sprint-status.yaml` 与 implementation story 文档，也已有一版 `student-web` 认证原型；问题已从“空白起步”变成“如何把逻辑验证 seam 收敛成可维护、可验证、可交接的正式实现前置”。
- 当前真正的现实约束不是“需求不清楚”，而是验证页与正式页面边界尚未写实、业务文件缺少规范中文注释、测试仍然散落，以及 RuoYi 本地联调尚需持续确认。
- 因此本 Story 的第一责任不是重造页面，而是先固化现有认证 seam、对齐真实接口契约、补足中文注释与测试收敛规则，再完成可复用的认证闭环。

### Reuse First, Do Not Reinvent

- 用户管理与认证源头在 `packages/RuoYi-Vue-Plus-5.X/`，不要在 FastAPI 或用户端重新发明用户表、密码校验、RBAC 规则或长期认证状态模型。
- `packages/ruoyi-plus-soybean` 是现成的 RuoYi + Soybean 前端参考实现，当前最有价值的是复用其接口契约、参数字段、加密方式、Token 存取与 `401` 清理语义，而不是直接复用其整页 `/login/:module` 路由体验。
- `packages/fastapi-backend/` 当前仍是占位状态，没有现成的用户管理实现。本 Story 不应该把注册/登录主流程落到 FastAPI。

### Delivery Strategy

- 交付顺序应调整为：`本地依赖与 RuoYi 启动` → `student-web 最小认证基础设施` → `首页认证入口壳层` → `注册 / 登录闭环` → `测试与联调证明`。
- 如果 `RuoYi + MySQL + Redis` 没有先跑起来，开发者只能“猜接口”，这会直接导致 Story 偏离真实契约；因此服务可用性验证属于 Story 1.1 的明确工作内容，不是额外工作。
- `FastAPI` 在 Story 1.1 中只作为未来集成边界存在；当前最多保留 base URL 和接口分层，不要求任何业务实现或联调成功。

### API Contract Requirements

- 登录接口：`POST /auth/login`
  - 请求头：`isToken: false`、`isEncrypt: true`、`repeatSubmit: false`
  - 关键字段：`username`、`password`、`tenantId`、`clientId`、`grantType`
  - 后端会校验 `clientId` 与 `grantType` 是否匹配可用客户端，并校验租户。  
  [Source: `packages/ruoyi-plus-soybean/src/service/api/auth.ts`；`packages/ruoyi-plus-soybean/src/typings/api/auth.d.ts`；`packages/RuoYi-Vue-Plus-5.X/ruoyi-admin/src/main/java/org/dromara/web/controller/AuthController.java`]
- 注册接口：`POST /auth/register`
  - 请求头：`isToken: false`、`isEncrypt: true`、`repeatSubmit: false`
  - 关键字段：`tenantId`、`username`、`password`、`confirmPassword`、`code`、`uuid`、`userType`
  - 后端会校验注册功能开关、验证码与用户名重复，并使用 BCrypt 持久化密码。  
  [Source: `packages/ruoyi-plus-soybean/src/service/api/auth.ts`；`packages/ruoyi-plus-soybean/src/views/_builtin/login/modules/register.vue`；`packages/RuoYi-Vue-Plus-5.X/ruoyi-admin/src/main/java/org/dromara/web/service/SysRegisterService.java`]
- 用户信息接口：`GET /system/user/getInfo`
  - 登录成功后必须调用，用于初始化 `user`、`roles`、`permissions`。  
  [Source: `packages/ruoyi-plus-soybean/src/service/api/auth.ts`；`packages/ruoyi-plus-soybean/src/typings/api/auth.d.ts`]
- 登出接口：`POST /auth/logout`
  - 前端先走接口，再清本地认证存储。  
  [Source: `packages/ruoyi-plus-soybean/src/service/api/auth.ts`；`packages/RuoYi-Vue-Plus-5.X/ruoyi-admin/src/main/java/org/dromara/web/controller/AuthController.java`]
- 租户与验证码辅助接口
  - `GET /auth/tenant/list`
  - `GET /auth/code`
  - 两者是否启用都必须做显式兼容，不要把“多租户/验证码永远存在”写死在 UI。  
  [Source: `packages/ruoyi-plus-soybean/src/service/api/auth.ts`；`packages/ruoyi-plus-soybean/src/views/_builtin/login/modules/pwd-login.vue`]

### Encryption and Transport Guardrails

- RuoYi 当前认证接口使用 `@ApiEncrypt`，现有 Soybean 请求层在 `VITE_APP_ENCRYPT=Y` 时，会用 AES + RSA 组合加密请求体，并通过自定义 header 传递加密密钥。用户端新实现如果接入同一网关/后端，必须沿用这一套机制，不允许自己发明第二套加密协议。  
  [Source: `packages/RuoYi-Vue-Plus-5.X/ruoyi-admin/src/main/java/org/dromara/web/controller/AuthController.java`；`packages/ruoyi-plus-soybean/src/service/request/index.ts`]
- 如果开发环境暂时关闭加密，也必须保持“接口字段、Bearer 头、成功码判定、错误码清理”这几项语义不变，避免 dev/prod 两套认证行为。

### Integration Boundaries

- 不要把登录态判断只放在前端。后续 FastAPI 访问必须能基于共享 JWT / Redis 在线态给出一致结论。  
  [Source: `_bmad-output/planning-artifacts/epics.md`，Story 1.3；`_bmad-output/planning-artifacts/architecture.md`，章节 “认证鉴权”]
- 本 Story 只建立“注册 + 登录 + 首页已认证上下文”最小闭环，不承担 Story 1.3 的完整受保护路由与 401 统一拦截实现，但必须预留可扩展的 auth hook / store 入口。
- 由于当前 FastAPI 尚未落仓，本 Story 的联调定义应收敛为：`student-web ↔ RuoYi` 完成认证闭环，`student-web` 仅保留 `VITE_FASTAPI_BASE_URL` 和未来受保护接口的分层占位，不要求调用成功。
- 不要在本 Story 中顺手实现社交登录、短信验证码登录、FastAPI 自建用户中心、完整个人中心或老师风格配置面板。

### Testing Requirements

- 在表单与鉴权测试之前，先验证最小开发底座：
  - `student-web` 能正常启动；
  - 本地环境变量能指向可用的 RuoYi 服务；
  - 至少 1 个认证相关请求能在 mock 或真实联调环境下正确返回。
- 测试覆盖至少包含：
  - 登录表单校验：空用户名、空密码、密码长度、验证码缺失
  - 注册表单校验：确认密码不一致、验证码缺失、租户切换
  - 请求参数拼装：`clientId`、`grantType=password`、`tenantId` 默认值
  - 注册成功后切回登录态
  - 登录成功后调用 `getInfo`
  - 401 / token 失效后清理认证存储并回到未认证状态
- Mock 外部依赖：
  - RuoYi `/auth/login`
  - RuoYi `/auth/register`
  - RuoYi `/auth/tenant/list`
  - RuoYi `/auth/code`
  - RuoYi `/system/user/getInfo`
- 当前前端包测试框架与部分认证测试已存在，但目录仍然散落；后续应迁移到稳定的 `src/test/` 镜像边界上，避免围着页面大组件继续堆脆弱测试。
