# 视频流水线补丁：占位降级 + RuoYi 回写鉴权修复 + 发布 description 截断

## 背景

在长耗时视频生成任务中观察到两类“系统级”问题：

1) **Worker 回写 RuoYi 时缺少鉴权上下文**

- 报错：`RUOYI_REQUEST_AUTH_REQUIRED`
- 触发点：流水线完成后执行 `sync_artifact_graph`（`/internal/xiaomai/video/session-artifacts`）时，worker 链路未显式传入用户 token / request_auth。

2) **公开视频发布写库失败（MySQL DataTruncation）**

- 报错：`Data too long for column 'description'`（`xm_user_work.description` 为 `varchar(500)`）
- 触发点：发布接口同步 `description=detail.result.summary` 时 summary 超过 500 字符。

同时，在 `patch_retry_max_retries=1` 的“及时止损”策略下，少数分镜修复失败会被跳过，导致最终合成视频缺少片段（用户感知为“残次品”）。

## 变更

### 1) Worker 的 `sync_artifact_graph` 显式带上 request_auth

- 在流水线持久化阶段，从 Redis 运行态读取 `xm_video_runtime_auth:{taskId}` 中保存的 `accessToken/clientId`，并以 `request_auth` 透传给 `VideoService.sync_artifact_graph(...)`。
- 结果：worker 在非 HTTP 路由上下文下也能带鉴权调用 RuoYi 内部接口，不再触发 `RUOYI_REQUEST_AUTH_REQUIRED`。

涉及文件：

- `packages/fastapi-backend/app/features/video/pipeline/orchestration/orchestrator.py`

### 2) 发布 title/description 截断，避免 RuoYi 写库溢出

双侧兜底（FastAPI + RuoYi）保证不会再把超长字符串写入 `xm_user_work`：

- FastAPI：在 `video_publication_to_ruoyi_payload` 中对
  - `title` 截断到 `<=200`
  - `description` 截断到 `<=500`
- RuoYi：在 `XmPersistenceSyncService.normalizeVideoPublication` 中同样做 `title/description` 最大长度截断。

涉及文件：

- `packages/fastapi-backend/app/features/video/long_term/records.py`
- `packages/RuoYi-Vue-Plus-5.X/ruoyi-modules/ruoyi-xiaomai/src/main/java/org/dromara/xiaomai/integration/service/XmPersistenceSyncService.java`

### 3) 分镜渲染失败时生成“占位片段”，保证最终视频片段数完整

- 当某 section 在 `patch_retry_max_retries` 耗尽后仍渲染失败，不再直接抛错导致该分镜被跳过；
- 改为尝试渲染一个**极短（1s）的占位 Manim 片段**（包含标题/失败提示/错误摘要/若干旁白行），随后在 FFmpeg 合成阶段会自动用 `tpad clone` 把画面补齐到音频时长。

涉及文件：

- `packages/fastapi-backend/app/features/video/pipeline/engine/agent.py`

### 4) 默认渲染质量回落到 `l`（降低隐形耗时）

- `FASTAPI_VIDEO_RENDER_QUALITY` 默认值改为 `l`，仍允许通过环境变量或输入参数覆盖到 `m/h`。

涉及文件：

- `packages/fastapi-backend/app/core/config.py`
- `packages/fastapi-backend/.env.example`
