# Story 1.3: 登出、401 处理与受保护访问一致性

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 已登录用户，
I want 前端、FastAPI 与 RuoYi 对认证态给出一致判断，
so that 我访问受保护资源时不会遇到前后端状态不一致的异常体验。

## Acceptance Criteria

1. 用户持有有效 Token 且 Redis 在线态存在对应记录时，前端、FastAPI 与 RuoYi 对受保护页面和接口的会话判断一致。
2. Token 失效、被撤销或用户主动登出后，系统统一返回未授权结果并清理本地认证态，前端回到可恢复未认证状态。
3. 用户已登录但权限不足时返回 `403` 而不是伪装成 `401`，前端可区分“未登录”和“已登录但无权限”。

## Dev Notes

### Business Context

- Story 1.3 是 Epic 1 的后端收口 Story。完成后，后续受保护页面都应复用同一套登录态判断逻辑。
- 它依赖 Story 1.1 的认证契约，也会反向约束 Story 1.2、1.4、1.6 的跳转与权限显隐逻辑。
- 当前版本目标是“认证语义一致”，不是在 FastAPI 内重做一套完整用户系统。

### Technical Guardrails

- FastAPI 不自建独立用户体系，认证主宿主仍是 RuoYi。
- FastAPI 与前端共享 JWT 认证头，Redis 作为在线态校验基础。
- `401` 表示未登录或登录失效，`403` 表示已登录但无权限。
- 前端 HTTP 客户端统一消费 `{ code, msg, data }`，不要为不同模块再写一套异常语义。
- Redis 仅用于运行态与在线态，不承担长期业务存储。

### Project Structure Notes

- 当前 `packages/fastapi-backend` 还没有实际应用骨架，开发时需要按架构目标目录先补最小结构，避免写入临时脚本或散落文件。
- 当前前端 `apiClient` 已存在，统一未授权处理应优先在这里扩展，而不是每个页面单独捕获 `401`。
- RuoYi 现有仓库已在 `packages/RuoYi-Vue-Plus-5.X` 中，认证行为应尽量通过既有 `/auth/*` 能力承载，不要引入第二条认证主链。

### Testing Requirements

- 前端覆盖 route guard、client interceptor、logout action。
- 后端至少准备认证依赖单测和最小集成测试夹具。
- 验证合法登录态访问成功，失效 / 撤销登录态返回 `401`，权限不足返回 `403`。
