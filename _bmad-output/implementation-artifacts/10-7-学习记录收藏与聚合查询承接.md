# Story 10.7: 学习记录、收藏与聚合查询承接

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 学习中心前端，
I want 有稳定的长期数据查询入口承接历史、收藏与聚合结果，
so that `/learning`、`/history` 与 `/favorites` 能以统一分页语义工作。

## Acceptance Criteria

1. 学习中心需要聚合视频、课堂、Companion、Evidence 与 Learning Coach 结果时，后端提供统一或可组合的分页查询结构，返回格式与全局 `{code, msg, rows, total}` 约定保持一致。
2. 用户执行收藏、取消收藏或删除历史时，变更被写入长期业务表，学习中心刷新后能看到一致结果，而不是运行态和长期态不一致。
3. 页面按类型、时间或状态进行过滤时，后端至少支持最小可行的筛选与分页，不要求前端自己在全量历史上做危险的本地拼装分页。

## Dev Notes

### Business Context

- 该 Story 是 Epic 9 学习中心域后端承接的核心基础，没有它，`/learning`、`/history`、`/favorites` 就只能停留在 mock 或前端本地拼装。
- 学习中心聚合的是多个业务域结果，因此长期查询边界必须先稳定，前端才可能做统一卡片与二次打开。
- 收藏与历史不只是 UI 状态，而是长期业务行为，需要进入 RuoYi 业务表。

### Technical Guardrails

- 学习中心域接口优先由 RuoYi 承接，标准业务 CRUD 不默认落在 FastAPI。
- 分页请求与响应必须对齐 `pageNum / pageSize` 和 `{code, msg, rows, total}`。
- 聚合查询至少支持类型、时间、状态等最小筛选，不允许把全量数据丢给前端后再本地分页。
- 收藏 / 取消收藏 / 删除历史必须写入长期业务表并立即反映到查询结果。
- 聚合数据来源必须来自长期宿主，不依赖 Redis 运行态或 SSE 回放。
