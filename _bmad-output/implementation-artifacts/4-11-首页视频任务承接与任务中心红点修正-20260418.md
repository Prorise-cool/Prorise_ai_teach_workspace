## 背景

上一轮补齐只把“当前任务状态卡”接到了 `/video/input`，但实际使用里暴露出两个断点：

1. bell 红点无论是否存在活跃任务都会显示，误导用户以为还有任务。
2. 用户从等待链路回到首页 `/` 时，看不到任何视频任务状态，导致“当前任务承接”只在视频输入页成立，首页仍是断开的。

这会直接削弱“任务在后台继续、用户随时可回来接着看”的感知，等于把承接体验做成了半截。

## 本次修复

### 1. bell 红点只在有活跃任务时显示

- `VideoTaskCenter` 将右上角红点改为 `total > 0` 时才渲染。
- bell 按钮本身仍然保留，空队列时继续可打开并展示“暂无进行中的任务”空态。

### 2. 首页也承接视频当前任务

- 从输入页抽出共享组件 `VideoActiveTaskCard`，把任务标题、阶段、进度、继续查看、取消任务统一到一套实现里。
- 首页 `/` 接入 `useVideoWorkspaceTasks`，在存在登录态时读取活跃视频任务。
- 首页 Hero 右侧新增“当前视频任务”卡，直接展示首个活跃任务状态。
- 首页顶栏也接入同一个 `VideoTaskCenter`，保证 bell 队列与主体任务卡共用同一批数据。

### 3. 状态与标题读取保持同源

- 新增 `video-workspace-task-shared.ts`，统一承接：
  - draft 标题读取
  - 进度裁剪
  - 生命周期 / 阶段文案解析
- 输入页和首页不再各自复制一份“当前任务状态”判断逻辑。

### 4. 公共首页安全接入登录态任务查询

- `useVideoWorkspaceTasks` 增加 `enabled` 选项。
- 首页只在存在有效登录态时才发起活跃任务查询，避免公开页在未登录场景下误请求受保护任务接口。

## 结果

- 没有活跃任务时，bell 不再误亮红点。
- 只要用户仍有 `pending` / `processing` 的视频任务，不管是在 `/video/input` 还是回到首页 `/`，都能直接看到任务状态并继续进入等待页。
- bell 队列和主体任务卡保持同源，不再出现“这里有、那里没有”的分裂行为。

## 约束

- 后续如果继续扩展首页任务承接，只允许复用 `VideoActiveTaskCard` / `video-workspace-task-shared.ts`，不能再复制第三套任务状态展示逻辑。
- bell 是否显示红点的语义已经固定为“存在活跃任务”，不能退回“按钮存在就亮点”的错误实现。
