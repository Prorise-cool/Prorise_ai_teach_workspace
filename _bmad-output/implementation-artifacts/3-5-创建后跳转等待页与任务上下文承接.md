# Story 3.5: 创建后跳转等待页与任务上下文承接

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 刚提交题目的用户，
I want 创建成功后立即进入正确的等待页并承接当前任务上下文，
so that 我不会在任务已经开始执行时还停留在输入页失去进度感。

## Acceptance Criteria

1. 用户成功创建视频任务后，前端自动跳转到 `/video/:id/generating`，`:id` 为后端返回的 `taskId`；跳转使用 `react-router-dom` `navigate` 并替换当前历史条目（`replace: true`），避免回退到已提交的输入页。
2. 等待页加载后能正确识别 URL 中的 `taskId`，并建立 SSE 连接 `GET /api/v1/video/tasks/{id}/events` 消费任务进度事件，使用 Story 2.5/2.6 冻结的统一 SSE parser 与断线恢复机制。
3. 等待页展示统一等待壳层（UX-DR-007/008），至少包含任务标题/摘要、当前阶段指示、进度条/百分比、预估剩余时间提示，视觉与课堂等待页保持统一壳层结构。
4. 用户在 mock 模式下创建任务后，等待页能加载 mock 任务状态流（mock SSE 事件序列），展示 pending → processing（含 progress 递增）→ completed 的完整动画，不依赖真实视频流水线。
5. 用户刷新等待页或从浏览器历史直接访问 `/video/:id/generating` 时，页面根据 `taskId` 调用 `GET /api/v1/video/tasks/{id}/status` 恢复当前任务上下文，不回到空白输入页。
6. 任务失败时等待页展示失败信息（错误码映射到用户可读文案）、重试入口（返回输入页并保留上次输入）与反馈入口，不弹出无法理解的技术错误。
7. 等待页 URL、任务 ID 与 SSE 连接参数在 mock / real 模式下行为一致，前端不做模式分支判断。

### Business Context

- 等待页是用户在提交题目后看到的第一个"系统在工作"的反馈，决定了用户是否有信心继续等待。如果等待页白屏、卡死或丢失进度，用户会关闭页面——任务虽然还在跑，但用户体验已经失败。
- 视频生成目标时长 90-180 秒（P95 < 5min），等待页需要在这段时间内持续给予用户有意义的进度反馈，而不是只有一个无限转圈。
- 等待页的壳层结构应与课堂等待页（Epic 5）保持统一，避免两个域各画一套等待 UI。如果已有统一等待组件则复用，否则本 Story 应创建可被课堂域复用的基础壳层。
- 失败时的重试体验很重要——用户不应该重新输入题目，而是"一键回到输入页并恢复上次内容"。

### Technical Guardrails

- SSE 连接必须使用 `packages/student-web/src/services/sse/index.ts` 的统一 parser，不在等待页内手写 SSE 解析。
- SSE 断线恢复使用 `Last-Event-ID`（Story 2.6），等待页不需要自行维护事件历史。
- 跳转使用 `replace: true` 替换输入页历史条目，避免用户回退到"已提交表单"的脏状态。
- 等待页必须是 lazy loaded 路由，不打入主包。
- 状态查询 `GET /api/v1/video/tasks/{id}/status` 使用 `useQuery` 管理，支持 SSE 连接前的初始快照加载。
- 错误码到文案的映射应统一在一处维护（如 `src/lib/error-messages.ts`），不在等待页内硬编码。
- 视频阶段映射（progress → stage）应在配置中定义，不在组件内硬编码区间值。
