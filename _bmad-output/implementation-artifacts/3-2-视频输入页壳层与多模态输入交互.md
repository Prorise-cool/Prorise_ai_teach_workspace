# Story 3.2: 视频输入页壳层与多模态输入交互

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 想快速搞懂一道题的用户，
I want 在一个核心输入区里完成打字、上传图片或粘贴图片，
so that 我能以最低操作成本发起讲解视频生成。

## Acceptance Criteria

1. `/video/input` 页面以单一核心输入区承接文本输入、上传图片与粘贴图片三种方式，用户不需要在多个互斥页面或多层弹窗之间切换输入方式。
2. 三种输入方式共享同一个主 CTA（"开始生成讲解视频"），页面在提交前明确展示当前输入来源（纯文本 / 图片附件 / 图片+OCR文本）与可编辑状态。
3. 当输入为空、图片格式不支持、内容明显不足时，页面返回明确、可恢复的 inline 错误提示；用户可在当前页修正后再次提交，不被带离流程。
4. 主 CTA 点击后调用 Story 3.1 冻结的 `POST /api/v1/video/tasks` adapter，mock 模式下使用 mock handler；创建成功后自动跳转等待页 `/video/:id/generating`。
5. 提交过程中主 CTA 进入 loading 状态，禁用重复提交；失败后恢复可提交态并展示错误信息。
6. 焦点在输入区域、上传触发器或主 CTA 上时，焦点可见、操作可达、不会因视觉样式消失焦点轮廓，满足桌面端主流程键盘可达（`NFR-UX-003`）。
7. 页面使用 `react-hook-form + zod` 管理输入校验，不使用 `useState + 手写校验`；表单 schema 与 Story 3.1 冻结的请求 schema 对齐。
8. 图片上传使用已有的 `useFileDropzone` hook，支持拖拽、点击选择、粘贴三种途径；图片预览区展示文件名、大小、可清除操作。
9. 页面在 `mock` 与 `real` 模式下状态闭环一致，前端不依赖真实后端即可完成输入 → 校验 → 提交 → 跳转的完整流程。

### Business Context

- 视频输入页是小麦产品"核心链路前半段"的入口，用户在此页完成从"想搞懂一道题"到"发起视频生成"的全部输入操作。
- 当前 `video-input-page.tsx` 已有壳层实现（标题、textarea、工具栏、引导卡片、社区瀑布流），但提交按钮仅有 `console.log` 占位，且使用 `useState` 管理文本——本 Story 需将其升级为正式的 `react-hook-form + zod` 管理。
- 输入页的核心体验是"一个输入区搞定一切"，UX-DR-004 明确规定单一核心输入区，不允许多页面或多弹窗切换输入方式。
- 本 Story 不负责公开视频发现区（3.6 负责）和等待页内容（3.5 负责），但必须确保创建成功后跳转路径正确。

### Technical Guardrails

- 必须使用 `react-hook-form + zod` 管理表单，禁止继续用 `useState + 手写校验`（CLAUDE.md Frontend Guardrails §4.1）。
- 必须使用 `@tanstack/react-query` 的 `useMutation` 管理提交状态，禁止手写请求状态机。
- 错误反馈使用 `sonner` toast 或 inline field error，禁止手写 toast 队列。
- 样式新增必须在 feature SCSS 中完成，不允许向 `theme.css` 写入页面私有变量。
- 所有 `className` 必须通过 `cn()` 合并，禁止字符串拼接。
- 图片上传后的 `imageRef` 生成逻辑：MVP 阶段可使用 `URL.createObjectURL` 作为本地引用，真实上传由 Story 3.3/3.4 在后端完成。
- 组件拆分：页面容器（`video-input-page.tsx`）只负责路由参数、表单 submit 和导航跳转；输入卡片、工具栏、预览区应拆为子组件或 hooks。
