# Story 10.8: 后台查询、导出与审计边界

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 平台运营人员或教师管理员，
I want 关键学习数据能够被稳定查询与审计，
so that 平台可以长期运营，同时又不把当前版本扩展成独立 ToB 产品。

## Acceptance Criteria

1. 视频、课堂、Companion、Evidence、quiz、收藏等结果产生后，运营人员通过后台可以按边界查看需要的学习记录、摘要和状态，当前版本只承接管理、查询和审计边界，不额外扩展成新的独立 ToB 产品域。
2. 某类记录需要导出或审计时，数据字段与状态语义足以支撑最小导出或审计，不需要重新回到 Redis 运行态拼装历史信息。
3. 后台查询权限受 RuoYi RBAC 控制时，权限不足会按统一规则拒绝访问，小麦不会在后台再自建一套平行权限系统与其冲突。

## Tasks / Subtasks

- [ ] 定义后台查询边界与最小字段集（AC: 1, 2）
  - [ ] 梳理视频、课堂、Companion、Evidence、Learning Coach、学习记录、收藏等后台需要的最小查询字段。
  - [ ] 明确哪些是只读查询、哪些允许最小管理操作，避免无界扩张成新产品域。
  - [ ] 为状态、摘要、来源、时间、用户归属等审计字段建立统一口径。
- [ ] 设计导出与审计清单（AC: 1, 2）
  - [ ] 输出最小导出字段清单，确保能满足运营复盘和基础统计而不泄露过多敏感数据。
  - [ ] 明确导出与审计必须基于长期数据主表 / 关联表，不回放 Redis 过程数据。
  - [ ] 约束导出格式、筛选条件和敏感字段处理边界。
- [ ] 对齐 RuoYi RBAC 与操作日志（AC: 1, 3）
  - [ ] 为后台查询、导出、敏感查看等动作配置权限标识与菜单 / 按钮权限。
  - [ ] 复用 RuoYi 操作日志、登录日志与权限拒绝语义，不另建平行审计体系。
  - [ ] 明确权限不足时返回 `403` 或标准拒绝结果，不伪装为资源不存在。
- [ ] 增加查询 / 导出 / 审计测试（AC: 1, 2, 3）
  - [ ] 覆盖有权限与无权限的后台查询路径。
  - [ ] 覆盖最小导出字段、状态字段和时间字段的正确性。
  - [ ] 覆盖后台查询无需 Redis 回放即可完成审计定位。

## Dev Notes

### Business Context

- `10.8` 是 Epic 10 的收口 Story，它决定小麦后台到底承接到什么边界为止。
- 该 Story 的目标是让运营、教师或管理员具备必要的查询、导出和审计能力，而不是演化出新的独立 ToB 业务产品。
- 前面的 `10.4` 到 `10.7` 负责沉淀长期数据，`10.8` 负责把这些长期数据转换成后台可消费的稳定管理边界。

### Technical Guardrails

- 后台查询、导出和审计都必须基于长期业务数据，不依赖 Redis 运行态拼装历史。
- 权限体系必须复用 RuoYi RBAC，权限不足统一走 `403` / 标准拒绝语义。
- 导出字段要满足运营和审计最小需要，同时符合学生数据隐私边界，不无控制地暴露敏感内容。
- 当前阶段只做查询、记录、审计和导出准备，不做与需求无关的复杂后台工作流扩展。
- 操作日志、登录日志和权限拒绝应尽量复用 RuoYi 已有基础设施，不另造一套审计系统。

### Project Structure Notes

- 当前 RuoYi Java 后端和 Vue 管理端前端都已作为基座存在，但 `ruoyi-xiaomai` 业务模块尚未创建；因此后台查询边界应优先落在新增业务模块，而不是硬塞进系统模块。
- `packages/ruoyi-plus-soybean/` 已存在，可作为后台查询页、导出入口和权限可见性的消费端，但本 Story 的核心仍是后端边界先稳定。
- RuoYi 自带操作日志、权限校验和 Generator 体系，应优先复用这些现成能力，而不是为小麦自建平行后台机制。

### Testing Requirements

- 覆盖后台查询在有权限、无权限与无结果时的表现。
- 覆盖最小导出字段的完整性、状态语义与时间格式。
- 覆盖查询与导出不依赖 Redis 运行态即可完成。
- 覆盖敏感字段最小暴露、权限拒绝与操作日志记录。
