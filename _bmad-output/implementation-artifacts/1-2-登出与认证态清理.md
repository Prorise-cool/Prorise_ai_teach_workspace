# Story 1.2: 登出与认证态清理

Status: ready-for-dev
Story Key: 1-2-登出与认证态清理

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 已登录用户，
I want 在主动退出时立即清理当前认证态，
so that 我离开设备后不会留下可继续访问受保护资源的会话。

## Acceptance Criteria

1. **Given** 已登录用户正在使用平台，**When** 用户主动执行登出，**Then** 本地认证态立即被清除，**And** 后续受保护访问必须重新登录。
2. **Given** 用户已经登出，**When** 其再次访问受保护页面或接口，**Then** 系统不再把旧 Token 视为有效，**And** 页面会回到未认证可恢复状态。

## Tasks / Subtasks

- [ ] 建立统一的认证清理原语与状态模型（AC: 1, 2）
  - [ ] 在 `packages/student-web/` 复用或补齐 Story 1.1 计划中的 auth store / hook / storage / service seam；如果 Story 1.1 尚未合并，按同一职责边界一次性落位，避免把清理逻辑散落到 `App.tsx`、按钮组件或任意页面里。
  - [ ] 实现单一 `clearAuthState` / `resetAuthState` 原语，至少清除 `token`、`refreshToken`、当前用户信息、鉴权标记、可能的受保护重定向上下文，以及正在进行中的认证相关 loading 标识，并保证幂等。
  - [ ] 用户主动登出、应用启动 rehydrate 失败、统一 `401` 清理三条路径必须共用同一个状态重置入口，不能出现多套 cleanup 分支。

- [ ] 实现主动登出交互与未认证回退（AC: 1）
  - [ ] 在首页浮动导航或最小用户胶囊中提供显式“退出登录”动作；该动作只在已登录态显示，未登录态继续显示登录 / 注册 CTA。
  - [ ] 登出完成后当前界面立即切回未认证首页壳层，隐藏受保护用户上下文，不跳转到无关页面，也不遗留“已登录用户名 / 头像 / 按钮”残影。
  - [ ] 交互需支持键盘操作、可见焦点、重复点击保护与最小 loading 反馈；本 Story 不扩展完整个人中心、设置页或复杂下拉菜单。

- [ ] 对齐 RuoYi `/auth/logout` 合同与旧 Token 失效路径（AC: 1, 2）
  - [ ] 调用 `POST /auth/logout` 作为服务器侧会话撤销入口，沿用 `Authorization: Bearer <token>` 与既有请求语义；不要自造 FastAPI 登出接口，也不要把登出逻辑塞进 RuoYi Admin 前端页面。
  - [ ] 对用户主动退出，前端必须在 `finally` 或等价收口中清理本地认证态；即使接口返回 `401`、用户已失效或网络异常，也不得保留本地 token。
  - [ ] 若本 Story 或 Story 1.1 引入了统一 `ky` 客户端，请在 `beforeRequest` 注入 Bearer token，在统一响应 / 错误处理里复用同一个 cleanup primitive；不要在每个 API 调用点手写 401 分支。

- [ ] 建立应用启动与受保护访问的最小回收机制（AC: 2）
  - [ ] 应用初始化时若发现本地残留 token，使用 `GET /system/user/getInfo` 做最小 rehydrate；失败则立即清理并停留在未认证可恢复状态。
  - [ ] 使用最小受保护接口或受保护 UI 状态验证“已登出后旧 token 不再可用”；在本 Story 中可用 `getInfo` 作为最小受保护接口，不要求提前实现完整受保护路由矩阵。
  - [ ] 避免出现无限跳转、重复 toast、重复 logout 请求，或“本地已登出但界面仍显示已登录”的状态撕裂。

- [ ] 补齐测试与联调证明（AC: 1, 2）
  - [ ] 为 auth cleanup primitive、logout service、应用启动 rehydrate、`401` 清理、首页未认证 fallback 编写 `Vitest + Testing Library` 测试。
  - [ ] Mock `POST /auth/logout`、`GET /system/user/getInfo`，覆盖成功、`401`、网络异常、token 缺失、重复点击退出、已失效 token 等边界。
  - [ ] 增加最小手工验证清单：登录成功 → 进入首页已认证态 → 点击退出 → 刷新页面 → 再次访问最小受保护接口 / 区域必须要求重新登录。

### Auth Cleanup Semantics

- 认证清理必须只有一个可信入口，例如 `clearAuthState(reason)`。推荐把 `reason` 建模为 `user-logout`、`unauthorized`、`bootstrap-failed`、`token-missing` 等有限枚举，方便 toast、日志与测试断言统一。
- 用户主动退出的顺序建议为：
  1. 锁定退出按钮，防止重复提交；
  2. 若存在 token，则发起 `/auth/logout`；
  3. 不论接口成功、`401` 还是网络失败，都执行本地 cleanup；
  4. 清除内存态与本地存储；
  5. 回到未认证首页壳层，并仅显示一次退出 / 失效反馈。
- `401` 自动清理与用户主动退出必须共享同一 cleanup primitive，但用户文案不同：
  - 用户主动退出：提示“已退出登录”；
  - token 失效 / 未授权：提示“登录状态已失效，请重新登录”。
- 不要过度清理与认证无关的本地状态，例如主题偏好、非敏感页面展示设置或用户尚未提交的公共输入草稿；本 Story 聚焦 auth session，而不是“清空整个浏览器侧缓存”。
- 如果 Story 1.1 引入了 auth dialog 草稿或 redirect query，cleanup 也要收口这类“仅在已认证流程中有意义”的状态，避免用户看到错误恢复目标。

### Integration Boundaries

- Story 1.2 的目标是“主动退出 + 认证态清理 + 最小未认证回退”，不是完整的跨前端 / FastAPI / RuoYi 鉴权一致性收口；后者属于 Story 1.3。
- 可以为 Story 1.3 预留统一 client、cleanup primitive、最小 rehydrate 逻辑，但不要在本 Story 顺手实现完整受保护路由矩阵、FastAPI token 校验页面、个人中心或学习记录页。
- 不要在本 Story 中自行开启 refresh-token 自动续期闭环。参考实现里的 refresh token 重试逻辑仍是占位，若强行补全，会把范围从“登出清理”膨胀为“全量会话管理”。

- 不要引入 cookie-based auth、跨域 iframe 登出、社交登录登出或教师端并行前台；这些都不属于当前验收范围。

### Testing Requirements

- 单元 / 组件测试至少覆盖：
  - cleanup primitive 幂等：连续调用两次不会抛错，也不会残留 token；
  - 用户主动退出：`/auth/logout` 成功后清理本地状态并回到未认证 UI；
  - 已失效 token：`/auth/logout` 返回 `401` 时仍完成本地清理；
  - 网络异常：请求失败后仍完成本地清理，但保留可解释提示；
  - 启动 rehydrate：本地有 token 但 `getInfo` 失败时自动清理；
  - 重复点击退出：只发一次有效退出流程或被安全去重；
  - 已登出后再次调用最小受保护接口：不会重新带上旧 token。
- 手工联调至少覆盖：
  - 登录成功后首页显示已认证态；
  - 点击退出后首页立刻恢复公共态；
  - 浏览器刷新后不应自动恢复为已登录；
  - 手动访问最小受保护能力时提示重新登录，而不是静默失败或无限跳转；
  - 若后端 Redis 在线态已删除，对应旧 token 再次访问接口返回未授权。
