# Story 4.10: 视频结果公开发布与输入页复用卡片

Status: in-progress

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 产出讲解结果的用户，
I want 在视频结果页显式公开发布可复用的视频结果，
so that 输入页可以发现这些公开结果并为其他用户提供快速复用入口。

## Acceptance Criteria

1. 视频任务完成且结果满足公开条件时，用户在 `/video/:id` 结果页执行公开发布操作，系统记录公开状态、最小卡片元数据与稳定结果标识；默认私有结果不会在未确认的情况下自动进入公开发现区。
2. 公开发布 API `POST /api/v1/video/tasks/:id/publish` 接受确认请求，返回公开后的最小卡片元数据（`title`、`summary`、`knowledgePoints`、`coverUrl`、`duration`、`resultId`、`publishedAt`）；取消公开 `DELETE /api/v1/video/tasks/:id/publish` 撤回公开状态。
3. 已公开的结果可被 `/video/input` 页面的公开视频发现区（Story 3.6）请求，输入页可获得标题/摘要、知识点、封面/时长与复用所需最小字段，不需要额外拼装社区详情数据。
4. 用户取消公开、公开失败或权限不足时，结果页返回明确反馈（`sonner` toast），公开发现区同步移除该卡片，不会把失败或未授权结果伪装成已公开。
5. 公开结果卡片数据通过 `GET /api/v1/video/published` 列表 API 提供，支持分页与排序（默认按发布时间倒序），Story 3.6 的公开视频发现区消费此 API。
6. 结果页的公开/取消操作按钮状态与后端实时同步：未公开 → 显示"公开发布"按钮；已公开 → 显示"取消公开"按钮；操作进行中 → 按钮 loading 状态。

### Business Context

- 公开发布是视频生成闭环中"用户贡献内容回流到社区"的关键链路，直接影响输入页公开视频发现区的内容丰富度。
- 默认私有是产品策略：用户必须显式操作才能公开，防止低质量或错误结果自动进入公开池。
- `PublishedVideoCard` 是输入页发现区和未来可能的视频广场的基础数据单元。
- 公开状态的实时同步确保用户操作后立即生效，不出现"已取消但仍显示在发现区"的不一致。

### Technical Guardrails

- 公开/取消操作必须校验任务归属（`userId` 匹配），不能信任前端传递的 userId。
- 公开状态变更需同时更新 RuoYi 长期存储和 Redis 缓存，保持双源一致；Redis 缓存设置合理 TTL（如 10min），降级到 RuoYi 查询。
- 公开列表 API 的性能目标：p95 < 200ms，建议使用 Redis 缓存 + 分页游标。
- 前端使用 `useMutation` + `invalidateQueries`，不手动操作 cache；确保 publish/unpublish 后 `useVideoResult` 和 published list 的 query 都被刷新。
- 公开结果的 `coverUrl` 应为 CDN 地址（与 Story 4.6 一致），不使用签名 URL（公开内容无需签名）。
- 并发控制：同一任务的 publish/unpublish 操作应有乐观锁或版本号，防止并发冲突。
- "查看讲解"动作（输入页卡片点击）跳转到 `/video/:id`，由 Story 3.6 前端实现。
