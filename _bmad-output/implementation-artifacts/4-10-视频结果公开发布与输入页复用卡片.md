# Story 4.10: 视频结果公开发布与输入页复用卡片

Status: backlog

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 产出讲解结果的用户，
I want 在视频结果页显式公开发布可复用的视频结果，
so that 输入页可以发现这些公开结果并为其他用户提供快速复用入口。

## Acceptance Criteria

1. 用户可在 `/video/:id` 显式执行公开发布或取消公开，默认私有结果不会自动进入公开发现区。
2. 公开成功后，系统产出可供 `/video/input` 消费的最小卡片元数据与稳定结果标识。
3. 公开失败、取消公开或权限不足时，结果页返回明确反馈，并同步影响输入页的公开发现区可见性。

## Dev Notes

### Business Context

- 视频结果默认私有，公开视频必须经过显式发布动作。
- 当前版本只承认“公开发布 + 发现复用”，不扩展评论、点赞、关注等社区能力。
- 输入页公开卡片区与结果页公开动作属于同一条业务链路。

### Technical Guardrails

- 公开状态与视频结果主体必须解耦，避免“视频可播放但公开元数据失败”被误判为完全成功。
- 输入页消费的是最小卡片元数据，不依赖完整结果详情结构。
- 取消公开必须能及时同步到输入页发现区，避免出现脏卡片。

### Project Structure Notes

- 需要与 `Story 3.6` 对齐公开卡片 payload。
- 当前前端真实结构采用 `src/features/*` + `src/app/routes/index.tsx`，不再使用旧的 `src/pages/*` 或 `src/router/*` 落点。
- 长期公开状态与最小元数据应进入长期业务宿主，不应只停留在 Redis 运行态。
- 结果页公开动作应兼容 mock 模式和真实接口模式。
- 本 Story 依赖 `4.6` 的结果元数据与 `4.8` 的结果页壳层；在这两个前置未稳定前，状态保持 `backlog`。

- 覆盖结果页在 mock / real 两种模式下的公开状态反馈。
