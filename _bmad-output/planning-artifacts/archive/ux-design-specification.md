# UX Design Specification — 小麦（AI 教学视频智能体）

**Author:** Prorise
**Date:** 2026-03-22

---

## Executive Summary

### Project Vision

小麦采用“双内容引擎 + 会话伴学 + 证据检索 + 学习教练”的产品骨架，为高职教育提供兼顾沉浸、解释与巩固的学习体验：
- **内容引擎 1（Classroom Engine）**：用户输入“我想学 XXX”，系统生成主题课堂、幻灯片、讨论与白板讲解
- **内容引擎 2（Video Engine）**：用户上传题目照片或输入描述，系统生成 Manim 单题讲解视频
- **Companion**：围绕当前视频秒点或课堂步骤进行追问与解释白板联动
- **Evidence / Retrieval**：承接证据补充、来源引用与依据回看；当前版本仅以来源抽屉 / 证据面板承载，不提供学生端独立页面
- **Learning Coach**：承接会话后的巩固练习、推荐与路径规划

边界冻结：
- **Companion**：解释当前这一步
- **Evidence / Retrieval**：提供资料依据与来源引用
- **Learning Coach**：负责学后练习、推荐与路径规划

核心价值主张：将传统 10+ 小时的教学视频制作压缩到 5 分钟，同时通过可切换的预设老师风格提供基础个性化体验。

### Target Users

| 用户类型 | 核心痛点 | 技术素养 | 主要设备 | 使用场景 |
|----------|----------|----------|----------|----------|
| **高职学生** | 抽象概念难理解、课后无人答疑、碎片时间学习 | 中等偏低，移动端为主 | 手机 > 电脑 | 课后复习、考前冲刺、作业难题 |
| **高职教师** | 视频制作耗时（10+小时/条）、数字素养不足 | 中等，桌面端为主 | 电脑 > 平板 | 备课、翻转课堂、资源建设 |
| **自学者** | 知识盲点、缺乏系统指导、按需学习 | 中等 | 混合 | 技能提升、职业转型 |

### Key Design Challenges

**挑战 1：双入口的认知负担**
- 用户进入产品时面临"我想学"还是"我有问题"的选择焦虑
- 高职学生（尤其是大一新生）可能不清楚自己的需求属于哪种模式
- **UX 目标**：让入口选择变得直觉化、零焦虑，减少用户思考时间

**挑战 2：等待体验设计**
- 视频生成需要最多 5 分钟，课堂生成也需要数分钟
- 对于习惯即时反馈的 Z 世代学生来说，这段等待是流失风险点
- **UX 目标**：设计有意义的进度反馈，让等待过程本身有价值感

**挑战 3：教师数字素养门槛**
- 研究显示高职教师数字素养参差不齐
- 任何需要教师理解技术概念的交互都是风险点
- **UX 目标**：做到"极简操作"，零学习成本上手

### Design Opportunities

**机会 1：风格选择的低干扰配置**
- 4 种 AI 老师首先是业务预设选项，不是独立页面或重点视觉对象
- 用户只需要在输入前通过简单下拉框完成选择，不应被额外设计打断
- 默认风格应始终可直接使用，降低决策成本

**机会 2：渐进式引导降低门槛**
- 首次使用时通过智能推荐（而非让用户自己选）降低决策成本
- 根据输入内容自动推荐入口，风格默认保持系统预设即可
- 逐步教育用户理解不同模式的适用场景

**机会 3：学习成就感的即时反馈**
- 课堂测验、视频观看后的"你学会了"反馈是强化动机的关键触点
- 分享功能可以促进口碑传播，增强社交认同感
- 做好微交互设计，能显著提升留存和满意度

## Core User Experience

### Defining Experience

小麦的核心用户体验围绕"进入-输入-等待-获得结果"的主链路展开：

**核心用户操作循环：**
1. 用户选择学习入口 → **快速理解路径**
2. 用户输入学习需求 → **极简操作，零思考**
3. 用户按需切换老师风格 → **简单下拉，不打断流程**
4. 系统生成内容 → **等待透明，进度可见**
5. 用户获得结果 → **速度+质量的双重惊喜**

**关键 UX 诉求：**
- 风格选择只是辅助配置，不是主流程障碍
- 等待不是"空白时间"，而是"期待过程"
- 简单不是"少功能"，而是"懂用户"

### Platform Strategy

| 阶段 | 目标平台 | 交互模式 | 关键约束 |
|------|----------|----------|----------|
| **MVP** | 桌面端（≥1024px） | 鼠标/键盘 | 单一断点，功能优先 |
| **Growth** | 平板适配（768-1024px） | 触控+鼠标 | 响应式布局优化 |
| **Vision** | 移动端（<768px） | 触控为主 | 独立移动端体验 |

**浏览器支持矩阵：**
- Chrome/Edge 90+：✅ 完全支持（主要目标）
- Firefox 88+：✅ 完全支持
- Safari 14+：⚠️ 基础支持（需测试 WebAudio 兼容性）

### Effortless Interactions

**必须做到零负担的交互：**

| 交互场景 | 零负担设计 |
|----------|------------|
| **入口选择** | 智能推荐 + 一键确认，而非让用户自己判断 |
| **风格选择** | 通过输入框附近的简单下拉框完成切换，默认值始终可直接使用 |
| **题目输入** | 支持"打字/拍照/粘贴"三种方式，无门槛 |
| **等待过程** | 分阶段进度 + 预计剩余时间 + 有意义的阶段描述 |
| **视频播放** | 一键播放，自动推荐相关内容 |

**消除的摩擦点：**
- ❌ 不让用户理解"LLM/Manim/TTS"是什么
- ❌ 不让用户担心"选错入口会怎样"
- ❌ 不让用户在等待时焦虑"是不是卡住了"
- ❌ 不让用户思考"这个功能在哪里"

### Critical Success Moments

**关键时刻 1：风格设置的无干扰理解**
- 用户点击输入框附近的老师风格下拉框，立即看到：
  - 当前会话老师风格名称
  - 其他可选老师风格及其选中态
  - 一个轻量且不会打断输入任务的配置入口
- **成功标准**：用户 3 秒内理解“这里可以切换老师风格”，且不会被迫离开输入任务

**关键时刻 2：输入后的"我在工作"确认**
- 用户提交需求后，立即看到：
  - 清晰的阶段划分："理解题目 → 生成脚本 → 渲染动画 → 合成视频"
  - 当前进度 + 预计剩余时间
  - 阶段性的"小成就"提示（如"脚本已生成，正在画动画..."）
- **成功标准**：用户不焦虑，知道系统在认真工作

**关键时刻 3：结果呈现的"超预期"瞬间**
- 视频生成完成，第一眼看到：
  - 流畅的 Manim 动画（数学公式"活"了）
  - 清晰的 TTS 配音
  - 整体视觉质量远超预期
- **成功标准**：用户产生"这居然是 5 分钟生成的"惊叹

### Experience Principles

**原则 1：配置轻量性**
- 4 种 AI 老师是可切换的业务预设，前端交互只需简单下拉框
- 风格选择不得升级为独立页面、重点视觉模块或主题切换系统

**原则 2：等待透明性**
- 进度反馈必须真实、有意义
- 避免模糊的"加载中"，要告诉用户具体在做什么
- 预计剩余时间要尽量准确

**原则 3：极简至上**
- 任何需要用户"想一想"的交互都是设计失败
- 首次使用应该"一看就会"，无需教程
- 智能推荐优于让用户选择

**原则 4：质量惊喜**
- 速度和质量的反差是核心差异化
- 在视觉上强化"高质量"的感知
- 让用户忍不住想分享给朋友

## Desired Emotional Response

### Primary Emotional Goals

**核心情感目标：被理解 + 惊喜感 + 分享欲**

小麦的情感设计围绕三个核心情感展开：

1. **被理解感** —— "这个产品懂我"
   - 首次使用时提供清晰入口和合理默认值
   - 用户感觉"这是为我准备的产品"

2. **惊喜感** —— "这也太厉害了"
   - 速度 + 质量的双重超预期
   - Manim 动画流畅呈现，视觉冲击力强

3. **分享欲** —— "要分享给室友"
   - 结果好到忍不住想展示
   - 分享功能便捷，成为传播引擎

### Emotional Journey Mapping

| 旅程阶段 | 用户情绪 | 设计响应 |
|----------|----------|----------|
| **首次打开** | "这是为我准备的" | 入口清晰、默认值可直接使用 |
| **选择风格** | "这里改一下就行" | 简单下拉、当前值明确 |
| **提交需求** | "开始了，有点期待" | 即时确认，阶段透明 |
| **等待生成** | "系统在认真工作" | 真实进度，不搞花哨（技术加速优先） |
| **看到结果** | "太厉害了，要分享" | 视觉冲击，一键分享 |
| **遇到失败** | "没关系，再试一次" | 失败可控，原因明确 |
| **再次使用** | "还是好用" | 记住最近使用配置，更顺滑 |

### Micro-Emotions

| 微情感 | 重要性 | 设计应对 |
|--------|--------|----------|
| **被理解感** | ⭐⭐⭐⭐⭐ | 清晰入口，智能推荐 |
| **期待感** | ⭐⭐⭐⭐ | 进度透明，阶段可见 |
| **惊叹感** | ⭐⭐⭐⭐⭐ | 视觉质量超预期 |
| **分享欲** | ⭐⭐⭐⭐⭐ | 结果优质 + 分享便捷 |
| **宽容感** | ⭐⭐⭐⭐ | 失败可控，不焦虑 |

**要避免的负面情绪：**
- ❌ 困惑感 —— "我该选哪个？"
- ❌ 焦虑感 —— "是不是卡住了？"
- ❌ 挫败感 —— "又失败了，算了"
- ❌ 失望感 —— "就这？"

### Design Implications

**情感 → 设计连接：**

| 情感目标 | UX 设计决策 |
|----------|-------------|
| **被理解感** | 首次使用先给出清晰入口和合理默认值，而不是强制偏好表单 |
| **风格匹配感** | 4 种 AI 老师通过名称与生成语气体现差异，前端只保留简单下拉选择 |
| **等待信任感** | 真实进度反馈，阶段描述具体，预计时间准确；**重点：技术加速 > UX 技巧** |
| **结果惊叹感** | Manim 动画质量第一眼就惊艳，TTS 配音清晰专业 |
| **分享欲** | 生成完成时突出分享按钮，支持一键复制链接/生成海报 |
| **失败宽容感** | 失败时明确告知原因（如"Manim 渲染失败，正在重试"），提供"再试一次"的明确路径 |

**关键设计决策：**
- 等待体验优化以技术加速为主，UX 不搞花哨的等待动画
- 分享功能是增长引擎，必须在结果页突出
- 风格选择保持轻量，默认值优先，不引入额外偏好收集流程

### Emotional Design Principles

**原则 1：被理解优先**
- 首次使用先提供清晰入口和合理默认值
- 智能推荐优于增加额外配置步骤

**原则 2：惊喜驱动分享**
- 结果必须超出预期，才能激发分享欲
- 分享功能必须便捷，一键完成

**原则 3：透明建立信任**
- 等待时真实反馈进度，不模糊处理
- 失败时明确告知原因，不隐藏问题

**原则 4：技术解决根本**
- 等待焦虑的最好解药是更快的生成速度
- UX 配合技术，而非掩盖技术瓶颈

## UX Pattern Analysis & Inspiration

### Inspiring Products Analysis

**B站（哔哩哔哩）**

| 维度 | UX 优点 | 对小麦的启示 |
|------|---------|-------------|
| **内容发现** | 推荐算法精准，用户快速找到想要的内容 | 智能推荐入口，减少选择负担 |
| **播放体验** | 流畅播放，倍速、进度控制便捷 | 视频播放器对标 B站，符合用户习惯 |
| **社区互动** | 弹幕让学习不孤单，评论区可互动 | 用"AI 老师对话"替代弹幕，提供即时互动 |
| **创作者连接** | 用户可关注 UP 主，建立持续连接 | 用户可持续使用自己习惯的老师风格默认值 |

**微信**

| 维度 | UX 优点 | 对小麦的启示 |
|------|---------|-------------|
| **即时反馈** | "发送中"状态，用户知道没卡住 | 生成进度透明，阶段反馈清晰 |
| **极简交互** | 核心功能一键可达 | 首页聚焦双入口，不做复杂导航 |
| **分享机制** | 一键分享，传播顺畅 | 结果页突出分享按钮，降低分享门槛 |

**作业帮/小猿搜题（反模式分析）**

| 痛点 | 用户体验 | 小麦的解决方案 |
|------|----------|----------------|
| **VIP 付费墙** | 核心功能需要付费，用户被迫离开 | 初期免费，积累用户后再考虑商业化 |
| **教师质量不确定** | 不同教师讲解质量差异大 | AI 老师质量稳定，风格可选 |
| **无法直接沟通** | 只能看视频，无法追问 | 支持"继续提问"，AI 老师持续讲解 |
| **体验割裂** | 需要多个 App 配合使用 | 一个产品搞定"学-问-练"全流程 |

| 模式 | 小麦的适配方式 |

**从微信借鉴：**

| 模式 | 小麦的适配方式 |
|------|----------------|
| **即时反馈状态** | 生成进度实时更新，阶段描述具体 |
| **极简首页** | 聚焦核心功能，不做复杂导航 |
| **一键分享** | 结果页突出分享，支持复制链接/生成海报 |

**从小红书借鉴（轻量反馈）：**

| 模式 | 小麦的适配方式 |

| **分享激励** | 分享后获得积分或成就 |

### Anti-Patterns to Avoid

| 反模式 | 为什么避免 | 小麦的做法 |
|--------|------------|------------|
| **VIP 付费墙挡核心功能** | 阻碍用户体验，降低传播 | 核心功能免费，增值服务再收费 |
| **模糊的"加载中"** | 用户焦虑，不知道是否卡住 | 具体阶段描述："正在生成脚本..." |
| **复杂的导航结构** | 增加认知负担，用户迷路 | 极简首页，双入口清晰 |
| **无法追问的死板视频** | 不满足个性化需求 | 支持"继续提问"，AI 老师持续互动 |
| **教师质量不确定** | 用户信任度低 | AI 老师质量稳定，风格可控 |

### Design Inspiration Strategy

**直接采用：**
- B站视频播放器交互模式（倍速、进度、全屏）
- 微信式的极简首页设计
- 微信"发送中"式的透明进度反馈

**适配改造：**
- 用"AI 老师对话"替代 B站弹幕（更直接的互动）
- 用"智能推荐"替代 B站推荐算法（基于内容而非行为）
- 用简单风格下拉替代复杂角色化选择组件

**坚决避免：**
- VIP 付费墙挡核心功能
- 模糊的进度反馈
- 无法追问的单向视频
- 复杂的导航结构

**核心差异化：**
- 大模型是最好的老师 —— 质量稳定、可对话、个性化
- 一个产品搞定全流程 —— 不需要多 App 切换
- 风格可选 —— 通过简单下拉完成会话配置

## Design System Foundation

### Design System Choice

**小麦采用 Shadcn/ui CLI v4 + Tailwind CSS v4 作为设计系统基础**

| 技术选型 | 版本/来源 | 用途 |
|----------|-----------|------|
| **UI 组件库** | Shadcn/ui CLI v4 | 基础组件、交互模式 |
| **样式系统** | Tailwind CSS v4 + UnoCSS | 主题定制、响应式 |
| **底层组件** | Radix UI | 无障碍访问基础 |
| **构建工具** | Vite 6.x | 快速开发体验 |
| **框架** | React 19 + TypeScript 5.7+ | 类型安全、性能 |

### Rationale for Selection

**选择 Shadcn/ui 的核心原因：**

| 优势 | 对小麦的价值 |
|------|-------------|
| **可复制而非安装** | 组件代码直接复制到项目中，完全可控，支持深度定制 |
| **基于 Radix UI** | 无障碍访问内置，符合 WCAG AA 标准（高职教育场景必需） |
| **Tailwind 原生** | 样式完全自定义，支持统一品牌风格与少量局部点缀 |
| **TypeScript 支持** | 类型安全，减少运行时错误，提升开发效率 |
| **轻量级** | 只复制需要的组件，保持包体积小，加载快 |
| **React 19 原生** | 充分利用最新特性，性能最优 |

**与项目需求的匹配度：**

| 项目需求 | 设计系统解决方案 |
|----------|------------------|

## 2. Core User Experience

### 2.1 Defining Experience

**小麦的定义性体验："输入即惊喜"**

用户输入题目 → 选择喜欢的风格 → 获得超预期的高质量讲解视频

这是一个端到端的体验魔法，不是单一交互。三个支柱缺一不可：

1. **极简输入**：输入题目就行，打字/拍照/粘贴，不要思考
2. **风格匹配**：4 种 AI 老师风格鲜明，用户感觉"这个老师懂我"
3. **超预期结果**：速度快到惊人（5 分钟内），质量高到恐怖（Manim 级别）

**用户会这样向朋友描述：**
"你输入题目，5 分钟就有讲解视频，这个 AI 老师很懂我！"

### 2.2 User Mental Model

**用户现有的心智模型：**

| 来源 | 心智模型 | 小麦的适配 |
|------|----------|------------|
| **B站** | 搜索视频 → 选择UP主 → 播放 | 输入题目 → 选择风格 → 生成播放 |
| **作业帮** | 拍题 → 等待 → 查看答案 | 输入题目 → 快速生成 → 完整讲解 |
| **线下补课** | 找老师 → 约时间 → 上课 | 选择风格 → 即时生成 → 随时观看 |

**用户期望：**
- 输入方式要简单（打字/拍照）
- 老师风格可选，但默认值可直接使用
- 生成速度要快（比等老师回复快）
- 质量要稳定（不像作业帮那样随机）

**可能困惑的点：**
- 初次使用不知道选哪个入口 → 智能推荐
- 不知道各风格有什么区别 → 在下拉中用简短名称区分，默认值可直接使用
- 担心生成失败 → 透明进度 + 失败可控

### 2.3 Success Criteria

**核心体验成功指标：**

| 指标 | 成功标准 | 测量方式 |
|------|----------|----------|
| **输入成功率** | > 95% | 用户成功提交需求的比例 |
| **风格配置理解率** | > 90% | 用户知道在哪里切换风格且不会被打断 |
| **生成速度** | < 5 分钟 | 端到端生成时间 |
| **质量满意度** | > 4.5/5.0 | 用户评分 |
| **分享率** | > 30% | 生成后分享的比例 |

**"哇"的瞬间触发条件：**
- 用户 3 秒内理解风格下拉框用途
- 生成进度比预期快
- 视频第一眼看到 Manim 动画流畅

### 2.4 Novel UX Patterns

**小麦采用的交互模式：**

| 模式类型 | 模式描述 | 小麦的实现 |
|----------|----------|------------|
| **成熟模式** | 视频播放器交互 | 对标 B站，倍速/进度/全屏 |
| **成熟模式** | 图片上传识别 | 腾讯云 OCR，拍题即识别 |
| **创新模式** | AI 风格选择器 | 输入框附近的轻量下拉配置入口，只展示名称列表和选中态 |
| **创新模式** | 透明生成进度 | 分阶段+预计时间+小成就 |
| **创新模式** | 与 AI 老师对话 | 生成后可继续提问 |

**需要教育用户的模式：**
- 双入口的区别 → 首次使用时智能推荐+说明
- AI 老师风格的差异 → 通过输入框附近的轻量下拉框快速理解
- 生成进度的含义 → 分阶段描述让用户理解

### 2.5 Experience Mechanics

**核心交互流程：**

**阶段 1：输入触发**
- 用户看到清晰的输入框："输入题目或知识点"
- 支持三种输入方式：打字 / 拍照 / 粘贴
- 输入框下方有智能推荐："检测到数学题，推荐使用问答入口"

**阶段 2：风格选择**
- 输入框右上角或近输入框位置显示当前老师风格下拉框
- 用户点击后展开当前会话的风格选项列表
- 下拉列表默认展示 4 种老师风格名称和选中态
- 组件结构必须支持未来扩展更多老师风格，但不把它设计成独立视觉重点

**阶段 3：生成等待**
- 提交后立即跳转到进度页面
- 分阶段展示：理解题目 → 生成脚本 → 渲染动画 → 合成视频
- 每个阶段有进度条和预计剩余时间
- 阶段完成时有"小成就"提示："脚本已生成！"

**阶段 4：结果呈现**
- 视频生成完成后自动跳转到播放页面
- 第一眼看到的是 Manim 动画的高质量画面
- 播放器下方有突出的分享按钮
- 右侧有"继续提问"入口，支持与 AI 老师对话

**错误处理：**
- 生成失败时明确告知原因（如"Manim 渲染失败"）
- 提供"再试一次"按钮，一键重试
- 失败不超过 3 次时保持友好，超过后转人工

**颜色使用原则：**
- 亮色模式以暖黄色调为主，营造温暖友好氛围；暗色模式保持温暖感，避免过冷的蓝灰色
- 4 种老师风格色在品牌色基础上叠加，通过 CSS 变量实时无缝切换
- 所有文字颜色对比度严格符合 WCAG AA 标准（4.5:1）

### 3. Layout & Structure (布局与结构)

引入 OpenMAIC 架构理念，页面从"传统的图文左右布局"进化为**"以对话/输入为核心的沉浸式布局"**。

**核心布局原则：**

- **悬浮导航 (Floating Nav)**：摒弃笨重的全宽顶部导航栏，采用右上角悬浮药丸形 (Pill-shape) 工具栏，释放最大垂直空间给内容
- **视觉聚焦 (Centered Hero)**：核心交互区（输入框/对话框）水平居中，最大宽度限制为 800px，确保视线聚焦，减少眼球左右移动
- **留白与呼吸感**：增加模块外边距（以 vh 为单位计算垂直边距），营造轻松、不压抑的对话环境

**间距系统（4px 基础单位）：**
- `--spacing-2` (8px): 紧凑元素间距（如图标与文字）
- `--spacing-4` (16px): 默认间距（如按钮内边距）
- `--spacing-6` (24px): 快捷指令间的气泡间距
- `--spacing-10` (40px): 核心输入框的内部上下留白

### 4. Material & Depth (材质与层级系统)

结合现代 Web 设计趋势，引入**毛玻璃（Glassmorphism）**与多层级阴影，让界面产生通透感与层次感。

**圆角系统 (Radius)：**
- `--radius-sm` (4px): 内部小图标、复选框
- `--radius-md` (8px): 标准卡片、弹窗内元素
- `--radius-lg` (16px): 核心对话框主体 (Main Chat Box)
- `--radius-full` (9999px): 悬浮导航条、快捷指令药丸标签、头像

**材质与阴影 (Shadow & Blur)：**

| Token | 值 (实现细节) | 用途 |
|-------|---------------|------|
| **Glass Nav** | bg-white/60 backdrop-blur-md | 右上角悬浮导航，透出底层光晕 |
| **Glass Surface** | bg-white/80 backdrop-blur-xl | 核心对话框，高级通透感 |
| **Shadow SM** | 0 1px 2px rgba(0,0,0,0.05) | 快捷指令、导航栏 |
| **Shadow MD** | 0 4px 12px rgba(0,0,0,0.08) | 按钮悬浮、活跃态卡片 |
| **Shadow LG** | 0 8px 24px rgba(0,0,0,0.12) | 核心对话框默认状态 |
| **Shadow XL (Glow)** | 0 20px 25px -5px rgba(114,46,209,0.1) | 核心对话框 Focus 状态发光阴影 |

### 5. Animation System (动画与动效系统)

动效是赋予"小麦"生命力的关键，设计需遵循**"克制、流畅、有呼吸感"**的原则。

**1. 氛围光晕 (Background Glows) - 呼吸感**
- **实现方式**：在底层背景上放置超大圆角的绝对定位元素，使用 blur-3xl（极度模糊）和低透明度（10%-20%）
- **动画定义**：animate-pulse (周期 3~4 秒)：模拟呼吸的频率
- **色彩搭配**：左上角为主色 (var(--color-primary))，右下角为风格色或品牌点缀色（如 blue-500），形成对角线色彩呼应

**2. 交互反馈 (Interaction Transitions) - 流畅性**
- **全局过渡**：所有的颜色切换、边框变化、阴影变化默认使用 `transition-all duration-300 ease-in-out`
- **按钮与卡片悬停**：
  - 浮动：`hover:-translate-y-0.5`（轻微上浮 2px）配合阴影加深（SM -> MD）
  - 按压：`active:scale-[0.97]`（轻微下陷，提供真实的点击物理反馈）
- **焦点状态**：当用户在核心聊天框输入时，外层容器阴影由 shadow-lg 扩散至 shadow-2xl，边框颜色平滑过渡为主色（Primary），视觉完全聚焦

**3. 图标微动效 (Icon Micro-interactions)**
- 悬停在特殊操作（如 OpenMAIC Star 按钮或笔形编辑图标）时，图标触发微缩放 `group-hover:scale-110`，增加趣味性

### 6. Accessibility Considerations (无障碍设计规范)

| 要求 | 标准 | 实现 |
|------|------|------|
| **色彩对比度** | WCAG AA (4.5:1) | 无论亮暗色模式，文本与背景对比度强制达标。按钮文字使用深色（#3b1701）确保在黄底上清晰可见 |
| **键盘导航** | 全功能可键盘操作 | 输入框、按钮均支持 Tab 切换，Focus 状态除了阴影变化，强制保留系统 Outline |
| **屏幕阅读器** | ARIA 标签完整 | 图标按钮（如日月切换、录音键）均包含 aria-label 属性 |
| **动画控制** | 尊重系统设置 | 使用 `@media (prefers-reduced-motion)` 媒体查询，在需要时关闭脉冲光晕，防止光敏不适 |

**冻结规则**

- 正式页面进入开发前，必须同时冻结高保真视觉稿、关键状态、交互说明、稳定接口契约。
- `/learning` 承担学习结果聚合；`/history`、`/favorites` 属于学习中心域视图。
- `/profile` 只负责个人资料；`/settings` 只负责平台设置与账号偏好。
- 当前版本不保留学生端独立 `/knowledge` 主路由；Evidence / Retrieval 仅以内嵌来源抽屉或证据面板承载。
- 如后续需要独立证据深挖页，需作为后续版本需求单独立项，不属于当前 MVP 冻结口径。

### 7.2 首页与认证页

**首页 `/`**

- 目标：让用户在 3 秒内理解“主题课堂”和“单题视频”两个入口差异。
- 模块：Logo、双入口卡片、推荐提示区、最近学习入口、登录入口。
- 入口文案：
  - `我想系统地学`
  - `我有道题不会`
- 推荐语义：只帮助用户选对入口，不提前暴露复杂系统结构。

**统一认证页 `/login`**

- 独立页面承接登录 / 注册切换，不在首页弹框完成。
- 成功后必须回跳原上下文。
- 未授权重定向要保持可恢复，而不是无上下文跳转。

### 7.3 Video Engine 会话页

**输入页 `/video/input`**

- 一个核心输入区承接打字、拍照、粘贴。
- 老师风格以下拉方式附着在输入区域旁，不使用大卡片角色选择。
- 主 CTA 明确表达“开始生成讲解视频”。

**等待页 `/video/:id/generating`**

- 复用统一等待壳层。
- 阶段粒度至少包含：题目理解、分镜、Manim、渲染、TTS、合成。
- 失败、超时、断线恢复必须在同页闭环。

**结果页 `/video/:id`**

- 主区域为播放器。
- 侧区域为 `Companion` 入口、问答流与解释白板。
- 用户可一边播放一边追问，主叙事不被伴学劫持。
- `Evidence / Retrieval`、`Learning Coach` 只作为后续动作入口，不占据主叙事。

### 7.4 Classroom Engine 会话页

**输入页 `/classroom/input`**

- 输入主题并发起课堂任务。
- 会话配置与视频输入页保持一致，但文案偏主题学习。

**等待页 `/classroom/:id/generating`**

- 与视频等待页共用统一任务体验。
- 阶段文案偏课堂生成、幻灯片、讨论与白板。

**结果页 `/classroom/:id`**

- 主内容区：幻灯片 / 讲解 / 讨论。
- 白板区：公式、结构图、步骤拆解。
- 伴学侧栏：围绕当前 slide、当前步骤或当前白板段落发问。
- 正式 quiz 不得硬插在课堂主叙事中。
- Classroom Engine 只产出课后练习触发信号；正式 checkpoint / quiz 由 Learning Coach 承接。

### 7.5 学习沉淀与个人域页面

**Evidence / Retrieval（非路由来源抽屉 / 证据面板）**

- 面向资料接入、文档解析、来源引用与证据回看。
- 只作为 `/video/:id`、`/classroom/:id`、`/learning` 内的来源抽屉或证据面板出现，不再保留学生端独立页面。
- 未来若需要独立证据深挖页，视为后续版本需求，不属于当前 MVP / 当前冻结口径。

**Learning Coach `/checkpoint/:sessionId`、`/quiz/:sessionId`、`/path`**

- `checkpoint`：轻量检查理解，不打断沉浸主流程。
- `quiz`：正式作答、判题、解析与结果回写。
- `path`：长期目标、周期、计划与后续行动。

**学习中心 `/learning`**

- 聚合历史、收藏、问答记录、错题、推荐与学习路径。
- 是“结果回看与继续学”的统一入口。

**个人中心域 `/profile`、`/settings`**

- `/profile` 只承接资料编辑。
- `/settings` 承接偏好与平台设置。
- 学习结果不回流个人中心主页。

### 7.6 页面级边界冻结规则

- `Companion` 解释“当前这一步”，`Evidence / Retrieval` 解释“资料里怎么说、依据是什么”，`Learning Coach` 负责“接下来怎么学”。
- 视频与课堂页面都必须支持不中断主叙事的追问入口。
- `Companion` 的展开态属于视频结果页 / 课堂结果页的一部分，不再单独算一类页面目录。
- 任何共享等待态、toast、error、confirm，统一归入共享状态资产，不拆成伪路由。

## 8. Companion Layer UX（会话伴学层）

### 8.1 设计原则

- 所有提问必须绑定上下文锚点：视频时间点、课堂步骤、slide 或白板段落。
- 回答优先解释当前上下文，再决定是否补充外部资料。
- 白板解释是 Companion 的核心差异化，而不是附加彩蛋。
- 展开 Companion 后，用户仍能继续看视频或浏览课堂。

### 8.2 视频页 Companion

**触发方式**

- 点击“问这一秒”。
- 暂停时针对当前帧提问。
- 从推荐追问中快速发起。

**最小结构**

- 当前时间锚点。
- 提问框。
- 问答流。
- 白板标签页 / 折叠区。

**体验要求**

- 提问后默认保持播放器上下文，不自动跳离视频。
- 白板解释优先围绕当前镜头、当前公式或当前推导。

### 8.3 课堂页 Companion

**触发方式**

- 针对当前 slide。
- 针对当前白板步骤。
- 针对当前讨论片段中的某一轮发言。

**最小结构**

- 当前步骤锚点。
- 问答流。
- 白板解释。
- 返回课堂主叙事的显式收起动作。

**体验要求**

- 用户追问后，课堂内容浏览仍可继续。
- Companion 回答不能重写课堂主叙事，只能补充解释。

### 8.4 失败与降级

| 场景 | 处理方式 |
|------|----------|
| Companion 暂时不可用 | 保留主内容区，侧栏显示明确失败与重试入口 |
| 白板解释生成失败 | 回退为结构化文本解释，不阻断问答 |
| 当前锚点缺失 | 提示用户基于当前播放位置或当前课堂步骤重新发问 |
| 资料不足 | 提示打开证据来源抽屉，或在当前页的来源抽屉 / 证据面板内继续深挖依据 |

## 9. Evidence / Retrieval UX（证据面板层）

### 9.1 载体定位

- 回答“资料、教材、讲义、术语里怎么解释，依据是什么”。
- 强调来源与引用，而不是当前会话时刻。
- 默认作为 Companion / Learning Coach 的后台证据层；学生端前台只以结果页或学习中心里的来源抽屉 / 证据面板呈现。
- 当前版本不设计学生端独立 Evidence 页面。

### 9.2 面板结构

- 顶部：来源范围、课程范围或资料接入上下文。
- 顶部工具区：文档上传、解析状态、历史切换。
- 主区域：来源摘要、引用片段、证据回看流。
- 回答下方：相关知识点、下一步动作、回到原会话入口。

### 9.3 关键交互

| 元素 | 交互 | 反馈 |
|------|------|------|
| 范围选择 | 切换 | 当前选中高亮、刷新当前证据上下文 |
| 文档上传 | 上传文件 | 显示上传中、解析中、成功或失败状态 |
| 证据深挖输入 | Enter / 点击发送 | 在来源抽屉 / 证据面板内流式回答、来源逐步出现 |
| 来源引用 | 点击 | 跳转原文片段或展开引用摘要 |
| 相关知识点 | 点击 | 作为下一问种子或跳去学习中心推荐区 |

### 9.4 与 Companion 的关系

- `Evidence / Retrieval` 不解释“视频第 42 秒为什么这样”。
- `Companion` 若发现当前会话上下文不足，可以异步拉取证据，或在当前页内引导用户在来源抽屉 / 证据面板继续深挖。
- `Evidence / Retrieval` 的结果可以反向成为 `Learning Coach` 的推荐来源，但不承担练习闭环。

## 10. Learning Coach Layer（学习教练层）

### 10.1 设计原则

- 所有学习教练能力都发生在会话后，而不是会话中强插。
- 重点不是“再解释一遍”，而是“检验理解、指出薄弱点、给下一步动作”。
- 结果必须回流学习中心，可回看、可继续。

### 10.2 Checkpoint

**角色**

- 轻量理解检查。
- 在课堂或视频结束后给用户一个低打扰入口。

**交互要求**

- 题量少、反馈快。
- 可以直接进入 quiz，也可以先结束本次会话。
- 不要求独立大页面视觉复杂度，但要有独立状态闭环。

### 10.3 Quiz

**角色**

- 正式作答、判题、解析、结果回写。

**MVP 题型边界**

- 单选题优先。
- 多选、填空、主观题后置。

**关键状态**

- 作答前。
- 已选择未提交。
- 判题中。
- 单题解析。
- 全部完成后的总结页。

### 10.4 Learning Path

**角色**

- 结合目标、周期与近期学习结果，生成阶段性学习路径。

**关键模块**

- 目标选择。
- 周期选择。
- 周计划 / 阶段任务。
- 保存、调整、开始学习。

### 10.5 与学习中心关系

- Learning Coach 的结果默认进入 `/learning`。
- `/learning` 是查看 quiz 结果、错题、推荐、路径的聚合入口。
- 个人中心不承担 Learning Coach 结果聚合。

### 11.3 认证机制

**JWT 共享 Redis 方案**：

```
1. 用户登录 → RuoYi 生成 JWT Token
2. RuoYi 将 Token 存入 Redis（key: online_tokens:{tokenValue}）
3. 前端持有 Token，存储在 localStorage/cookie
4. 前端请求 FastAPI 时携带 Token
5. FastAPI 从 Redis 验证 Token 有效性
6. Token 有效 → 放行；无效 → 返回 401
```

**前端处理**：

```typescript
// HTTP 客户端配置
const apiClient = ky.create({
  prefixUrl: API_BASE_URL,
  hooks: {
    beforeRequest: [
      (request) => {
        const token = localStorage.getItem('token');
        if (token) {
          request.headers.set('Authorization', `Bearer ${token}`);
        }
      },
    ],
    afterResponse: [
      (response) => {
        if (response.status === 401) {
          // Token 过期，跳转登录
          window.location.href = '/login';
        }
      },
    ],
  },
});
```

### 11.4 SSE 连接管理

**连接策略**：

| 场景 | 策略 |
|------|------|
| **首次连接** | 创建 EventSource，监听事件 |
| **断线重连** | 携带 Last-Event-ID 重连 |
| **超过重试上限** | 切换到轮询 `/status` |
| **页面切换** | 关闭连接，清理资源 |

**前端实现**：

```typescript
// SSE 客户端封装
class TaskEventSource {
  private eventSource: EventSource | null = null;
  private reconnectAttempts = 0;
  private maxReconnectAttempts = 5;

  connect(taskId: string) {
    this.eventSource = new EventSource(
      `/api/v1/video/tasks/${taskId}/events`
    );

    this.eventSource.onmessage = (event) => {
      const data = JSON.parse(event.data);
      this.handleEvent(data);
    };

    this.eventSource.onerror = () => {
      this.reconnect();
    };
  }

  private reconnect() {
    if (this.reconnectAttempts < this.maxReconnectAttempts) {
      this.reconnectAttempts++;
      setTimeout(() => this.connect(), 1000 * this.reconnectAttempts);
    } else {
      // 切换到轮询
      this.startPolling();
    }
  }
}
```

## 12. Unified Waiting Experience（统一等待体验设计）

> **架构对齐**：视频任务和课堂任务遵循统一任务模型，等待体验采用统一的 UX 模式。

### 12.1 设计原则

1. **一致性**：视频生成和课堂生成使用相同的进度组件
2. **透明性**：真实反馈进度，不模糊处理
3. **可控性**：用户可以取消任务
4. **可恢复性**：断线重连后恢复进度状态

### 12.2 统一进度组件

**组件 Props**：

```typescript
interface TaskProgressProps {
  taskId: string;
  taskType: 'video' | 'classroom';
  stages: Stage[];
  currentStage: number;
  progress: number;
  estimatedTimeRemaining: number;
  onCancel: () => void;
}

interface Stage {
  id: string;
  name: string;
  status: 'pending' | 'running' | 'completed' | 'failed';
  progress: number;
  message?: string;
}
```

**视觉规范**：

| 状态 | 颜色 | 图标 |
|------|------|------|
| **pending** | `--text-secondary` | ⏳ |
| **running** | `--primary` | 🔄 |
| **completed** | `--success` | ✅ |
| **failed** | `--error` | ❌ |

### 12.3 错误处理统一

| 错误类型 | 通用处理 | 特殊处理 |
|----------|----------|----------|
| **网络错误** | 显示重试按钮 | - |
| **超时** | 显示超时提示 | 建议检查网络 |
| **服务不可用** | 显示降级提示 | 自动切换 Provider |
| **渲染失败** | 显示失败原因 | 自动修复提示 |

## 13. Agent Style Data Specification（Agent 风格数据规范）

> **架构对齐**：4 种 AI 老师差异通过 AgentConfig 数据预设实现，不是页面级 CSS 主题切换。
> **交互边界**：前端 MVP 只要求在输入框附近提供简单下拉框；风格选择不是独立 UX 重点。

### 13.1 AgentConfig 数据结构

```typescript
interface AgentConfig {
  id: 'serious' | 'humorous' | 'patient' | 'efficient';
  name: string;
  persona: string;        // System Prompt
  avatar?: string;        // URL 或 emoji，可选局部展示
  color: string;          // 局部点缀色 HEX
  ttsVoice?: string;      // TTS 音色 ID
  examples?: string[];    // 示例对话
  description: string;    // 一句话描述
}
```

### 13.2 4 种 Agent 预设

| 风格 | name | avatar | color | persona 要点 |
|------|------|--------|-------|-------------|
| **严肃型** | 严谨教授 | 🎓 | `#4A6FA5` | 专业严谨、逻辑清晰、不废话 |
| **幽默型** | 风趣老师 | 😄 | `#FF9500` | 轻松有趣、举例生动、有梗 |
| **耐心型** | 温和导师 | 🌸 | `#52C41A` | 步骤详细、反复解释、有鼓励 |
| **高效型** | 干练讲师 | ⚡ | `#722ED1` | 直击要点、省时高效、无废话 |

### 13.3 CSS 变量使用规范

**正确使用（局部点缀）**：

```css
/* 仅用于头像边框、指示器、标签等局部元素 */
.agent-card[data-style="serious"] {
  --agent-color: #4A6FA5;
}

.agent-avatar {
  border-color: var(--agent-color);
}

.agent-tag {
  background-color: var(--agent-color);
}
```

**错误使用（页面级主题）**：

```css
/* ❌ 不要这样做 - 全局主题切换 */
:root[data-style="serious"] {
  --primary: #4A6FA5;  /* 错误：改变全局主色 */
}
```
