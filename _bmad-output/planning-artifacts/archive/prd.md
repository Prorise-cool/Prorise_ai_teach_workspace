# 2. 产品概述

## 2.1 产品愿景
小麦是一款面向中国高职教育场景的 AIGC 原生教学平台，通过双内容引擎、会话伴学、证据检索与学习教练能力，帮助学生更快理解知识点、题目与学习路径，帮助教师更高效地产出可播放、可复用的教学内容。

## 2.2 产品定位
小麦当前提供五类核心能力层：

边界三分法冻结如下：
- Companion：解释当前这一步
- Evidence / Retrieval：提供资料依据与来源引用
- Learning Coach：负责学后练习、推荐与路径规划

### 核心能力层 A：双内容引擎（独立）
- **Video Engine**
  - 用户输入题目文本或上传题目图片
  - 系统生成单题讲解视频并支持播放复习
- **Classroom Engine**
  - 用户输入主题
  - 系统生成基础课堂内容、幻灯片、讨论与白板讲解

### 核心能力层 B：会话伴学 Companion（共享）
- 围绕“当前学习上下文”进行即时追问与解释
- 绑定视频时间点、课堂步骤或白板步骤
- 支持侧边答疑与解释白板联动

### 核心能力层 C：Evidence / Retrieval Service（后台，可插拔）
- 面向教材、讲义、术语与上传文档建立可检索、可引用的证据层
- 为 Companion 与 Learning Coach 提供来源引用、术语解释与资料依据补充
- 当前版本不作为学生端独立主入口，仅通过 `/video/:id`、`/classroom/:id`、`/learning` 与 Learning Coach 的来源抽屉 / 证据面板承载；如未来新增独立页，视为新版本需求

### 核心能力层 D：学习教练 Learning Coach（独立）
- 课后 checkpoint / quiz、错题解析、推荐、学习路径规划
- 以“学后巩固”为目标，不打断课堂与视频的沉浸主流程

### 核心能力层 E：结果回看与数据闭环
- 历史记录
- 收藏管理
- 会话问答回写
- 学习信号沉淀
- 学习中心聚合展示

## 2.3 当前版本目标
当前版本目标是在赛事周期内交付一个可稳定演示、可持续扩展的 MVP / 公测版，满足：
- 用户可登录并进入双入口
- 视频生成主链路稳定打通
- 基础课堂链路稳定打通
- 会话伴学、证据检索与学习教练具备可演示闭环
- 长任务过程可见、状态可恢复、结果可回看
- 长期业务数据有明确持久化归属
- 外部能力具备基础 Failover 与降级策略

## 2.4 核心价值
### 对学生
- 降低抽象知识理解难度
- 缩短单题学习时间
- 提供更具解释性的学习体验
- 提供从“提问—理解—练习—记录”的连续学习闭环

### 对教师
- 降低教学视频制作成本
- 提高备课与资源生成效率
- 快速产出可播放、可管理的教学内容
- 逐步形成可复用的知识资源与学习记录体系

## 2.5 当前版本的产品边界变化
相较于旧版 PRD，本版本已不再只聚焦“课堂 + 视频 + 历史记录”，而是正式纳入：
- 会话伴学 Companion 服务
- Evidence / Retrieval 服务
- Learning Coach 学习教练服务
- 学习中心
- 统一任务框架
- SSE 恢复机制
- Provider 抽象与外部能力编排
- Async Worker 执行边界
- RuoYi 业务表数据承接边界

# 3. 架构对齐摘要

## 3.1 系统分层
系统采用双后端分层：

### FastAPI
负责：
- AI 功能执行
- 异步任务协调
- 视频与课堂生成
- Provider 调度
- SSE 进度推送
- 外部能力编排
- 运行时状态管理

### RuoYi
负责：
- 用户、角色、权限
- 小麦长期业务数据持久化
- 学习记录、收藏、任务元数据、问答日志等业务表承接
- 查询、审计、导出与管理宿主能力

## 3.2 数据边界
- Redis 只用于运行时状态、Token 在线态、SSE 事件缓存、短期上下文、Provider 健康状态
- Redis 中的数据必须设置 TTL
- 长期业务数据必须进入 RuoYi/MySQL 或 COS
- FastAPI 不承担长期业务主库存储职责
- 文件产物统一进入 COS

## 3.3 模块边界
- 课堂服务与视频服务是两个独立内容引擎，禁止互相耦合生成实现细节
- Companion 为共享会话层，消费内容引擎上下文，不承担内容生成流水线
- Evidence / Retrieval Service 为后台证据层，不承担“当前时刻解释”职责
- Learning Coach 为学后巩固层，不承担会话内即时答疑职责
- 各模块可共享任务框架、Provider 抽象层、SSE 基础设施、RuoYi 防腐层、COS 客户端
- 禁止跨业务模块直接耦合具体实现逻辑

术语冻结说明：`FR-KQ` 为历史编号保留，当前语义统一对应 Evidence / Retrieval 服务层，不代表学生端存在独立 Knowledge QA 主页面。

边界三分法冻结如下：
- Companion：解释当前这一步
- Evidence / Retrieval：提供资料依据与来源引用
- Learning Coach：负责学后练习、推荐与路径规划

| 能力域 | 主问题 | 主数据源 | 禁止承担 |
|---|---|---|---|
| Companion | 我现在这一步为什么这样？ | 会话产物源 | 通用资料库主问答 |
| Evidence / Retrieval | 这个知识点在资料里怎么定义？依据是什么？ | 内容知识源 + 证据索引 | 当前秒点解释 |
| Learning Coach | 我下一步该怎么练？ | 学习行为源 + 结果数据 | 会话内即时答疑 |

## 3.4 外部能力边界
- 腾讯云 ADP 当前作为默认外部 AI 能力实现接入
- 主链路稳定性不得依赖单一外部平台
- 用户前端体验全部自研，不直接暴露外部平台原生前端组件
- 外部依赖均需具备降级策略

## 3.5 Agent 风格边界
- Agent 风格属于数据层人格预设，不是 UI 全局主题
- 前端整体品牌风格统一
- 不同风格仅体现在内容语气、头像、局部点缀、语音参数等层面
- 前端中的风格选择仅作为输入框附近的简单下拉配置，不承载独立风格页、角色卡片矩阵或主题切换设计
- MVP 固定 4 种老师风格点缀色：严谨教授 `#4A6FA5`、风趣老师 `#FF9500`、温和导师 `#52C41A`、干练讲师 `#722ED1`

# 4. 用户与核心场景

## 4.1 目标用户
| 用户类型 | 核心需求 | 典型场景 |
|---|---|---|
| 高职学生 | 快速理解知识点与难题，获得练习与推荐 | 课后复习、作业难题、考试冲刺 |
| 高职教师 | 快速生成教学视频与课堂内容 | 备课、课堂资源制作、翻转课堂 |
| 自学者 | 定向突破知识盲点 | 补基础、专项练习、技能提升 |
| 管理角色（预留） | 查看记录、审计与运营数据 | 后台查询、导出、统计 |

## 4.2 核心场景
### 场景 A：主题课堂学习
用户生成课堂内容，在学习过程中可随时追问当前步骤，并在课后进入 checkpoint / quiz。

### 场景 B：单题视频学习
用户生成讲解视频，在播放过程中可针对当前秒点发问，并获得文本 + 白板解释。

### 场景 C：证据回看与来源补充
用户在会话解释不足时，通过来源抽屉 / 证据面板查看教材、讲义、术语或上传文档中的来源依据，并回看引用来源。

### 场景 D：学习教练
系统基于会话结果与学习行为，提供课后练习、错题解析、推荐与路径规划。

### 场景 E：教师备课
教师输入知识点或题目，快速获得可播放视频、基础课堂结果及后续学习建议。

## 4.3 当前版本聚焦场景
MVP / 当前版本优先聚焦：
1. 学生单题讲解
2. 学生主题学习
3. 会话解释后的证据补充与来源回看
4. 学习中心中的基础记录聚合
5. 教师快速备课中的单题视频生成与课堂生成

## 4.4 当前版本不重点解决
- 完整 ToB 后台产品体系
- 深度院校运营能力
- 自定义 Agent 市场
- 社交关系与分享生态
- 多语言完整商业化体系
- 高级推荐算法

# 6. 功能需求

## 6.1 FR-UM 用户与认证

### FR-UM-001 用户注册/登录
**描述**  
用户可以通过统一登录页（页面内支持切换到注册态）完成注册、登录、登出，并进入小麦前端功能页面。

**优先级**  
P0

**依赖**
- RuoYi 用户体系
- 前端统一认证页 `/login` 与鉴权拦截

**业务规则**
- 统一认证采用独立登录页（`/login`），页面内支持登录 / 注册态切换，不使用首页弹框
- 未登录用户访问受保护页面必须被拦截
- 登录态需在前端、FastAPI 与 RuoYi 层面保持一致
- FastAPI 不自建独立用户体系

**验收标准**
- AC1：用户输入正确凭证后，可在 2 秒内完成登录并进入首页
- AC2：未登录用户访问受保护页面时，自动跳转登录页
- AC3：登录失败时返回明确错误提示，不泄露敏感账户信息
- AC4：用户登出后原登录态立即失效

### FR-UM-002 Token 鉴权一致性
**描述**  
前端、FastAPI 与 RuoYi 对用户登录态保持一致判断。

**优先级**  
P0

**依赖**
- Redis 在线态
- 统一鉴权头

**验收标准**
- AC1：合法登录态可访问受保护的 FastAPI 接口
- AC2：失效或非法 Token 调用接口返回 401
- AC3：用户登出后再次调用受保护接口应失败

### FR-UM-003 个人资料查看与修改
**描述**  
用户可在个人中心查看并修改基础个人信息。

**优先级**  
P1

**验收标准**
- AC1：用户可查看昵称、头像、学校等基础信息
- AC2：用户可修改昵称和头像，并即时看到更新结果
- AC3：非法格式或超限文件上传被拒绝并提示原因

### FR-UM-004 权限分层
**描述**  
系统需基于 RuoYi 角色体系区分学生、教师和管理角色的能力边界。

**优先级**  
P1

**验收标准**
- AC1：学生角色仅可访问 ToC 学习功能
- AC2：教师角色可访问教师范围内的结果管理能力
- AC3：管理角色能力不直接暴露在 ToC 主导航中

## 6.2 FR-UI 前端页面与导航

[Rule] 本节定义的是产品最终必须具备的页面能力，并要求采用“契约先行、mock 先行、前后端并行”的实施方式。
[Rule] 当前执行口径为：先冻结 API 契约、状态模型、错误码、示例 payload 与 mock 样例；后端负责真实端点、数据回写与接口测试，前端基于 `adapter + mock handler` 并行完成正式页面与状态闭环。
[Rule] 高保真视觉稿、关键状态、交互说明、稳定接口契约四项门禁用于真实接口联调、主分支合并与发布收口；在视觉稿与契约稳定时，正式页面可先基于 mock 开发，不必等待真实后端完成。
[Rule] 若当前仅有线框图或结构草图，允许建设临时验证壳层，但不得把该壳层视为 FR-UI 的完成交付。
[Rule] 路由口径固定：`/learning` 为学习中心聚合入口；`/history`、`/favorites` 是学习中心域结果管理视图；`/profile` 仅负责个人资料；`/settings` 仅负责平台设置与账号偏好。
[Rule] 当前版本学生端不保留独立 `/knowledge` 路由；Evidence / Retrieval 只通过结果页、学习中心与 Learning Coach 的非路由面板承载。

### FR-UI-R01 详细页面与路由清单（并行实施口径）
| 页面能力 | 路由 | 职责边界 | 对应 FR |

## 6.3 FR-VS 视频服务

### FR-VS-001 题目输入
**描述**  
用户可通过文本输入或图片上传提交题目。

**优先级**  
P0

**依赖**
- 文本输入组件
- 图片上传组件
- OCR 能力

**业务规则**
- 文本输入为默认方式
- 图片上传是补充输入方式
- OCR 失败时必须提供手动输入替代路径

**验收标准**
- AC1：用户可提交文本题目并创建视频任务
- AC2：用户可上传 JPG/PNG 图片并提交视频任务
- AC3：图片识别失败时系统提示用户改为手动输入

### FR-VS-002 题目理解
**描述**  
系统需将题目解析为结构化理解结果，用于后续分镜与视频生成。

**优先级**  
P0

**验收标准**
- AC1：系统可提取题型、知识点和解题结构
- AC2：无法理解题目时返回明确提示而非直接报错
- AC3：理解结果可供后续阶段继续使用

### FR-VS-003 分镜生成
**描述**  
系统基于题目理解生成讲解视频分镜方案。

**优先级**  
P0

**业务规则**
- 默认分镜目标总时长为 `120 秒`
- 允许时长区间为 `90-180 秒`
- 分镜节奏按题目复杂度动态分配，不采用固定均分

**验收标准**
- AC1：系统生成不少于 3 个有效分镜
- AC2：每个分镜包含场景描述、讲解重点和旁白内容
- AC3：分镜总时长默认控制在 `120 秒` 附近，且位于 `90-180 秒` 区间内，不明显偏离题目复杂度

### FR-VS-004 Manim 代码生成
**描述**  
系统基于分镜生成可渲染的 Manim 动画代码。

**优先级**  
P0

**验收标准**
- AC1：生成代码通过基础语法检查
- AC2：代码包含完整可执行场景定义
- AC3：生成结果可进入渲染阶段继续处理

### FR-VS-005 代码自动修复
**描述**  
渲染失败后，系统自动尝试修复 Manim 代码。

**优先级**  
P0

**业务规则**
- 系统执行预设次数的自动修复，上限固定为 `2 次`
- 修复失败后任务必须正确结束，不得悬挂
- 高质量方案失败时可降级返回较低质量结果

**验收标准**
- AC1：首次渲染失败后自动进入修复流程
- AC2：系统最多执行 `2 次` 自动修复
- AC3：修复失败后返回清晰失败结果并记录任务状态
- AC4：存在降级结果时优先返回可用结果

### FR-VS-006 动画渲染
**描述**  
系统在受限环境中执行 Manim 渲染，生成视频素材。

**优先级**  
P0

**验收标准**
- AC1：任务成功时产出可用视频文件
- AC2：渲染超时或执行失败时任务状态被正确标记
- AC3：前端能看到阶段性进度变化

### FR-VS-007 TTS 合成
**描述**  
系统根据讲解内容合成语音音频。

**优先级**  
P0

**业务规则**
- 至少支持主备两类 Provider
- 主 Provider 失败时自动切换到备 Provider

**验收标准**
- AC1：至少支持 2 个可切换 TTS Provider
- AC2：主 TTS 不可用时系统自动切换备选 Provider
- AC3：当所有 TTS 不可用时返回明确失败或降级结果

### FR-VS-008 视频合成与上传
**描述**  
系统将动画与音频合成为最终视频并上传对象存储。

**优先级**  
P0

**验收标准**
- AC1：视频合成成功后输出可播放 MP4
- AC2：成功上传至 COS 后返回可访问视频 URL
- AC3：上传失败时执行重试或降级策略，并正确反馈状态

### FR-VS-009 视频任务进度反馈
**描述**  
视频任务执行过程中，用户应实时看到任务状态和进度。

**优先级**  
P0

**验收标准**
- AC1：任务创建成功后前端可建立事件流连接并接收进度更新
- AC2：进度覆盖理解题目、分镜生成、代码生成、渲染、TTS、合成、上传等关键阶段
- AC3：连接短暂中断后可恢复任务当前状态
- AC4：任务完成后前端可展示播放器入口或跳转播放页

## 6.5 FR-CS 课堂服务

### FR-CS-001 输入主题生成基础课堂
**描述**  
用户输入一个主题后，系统生成一套基础课堂内容。

**优先级**  
P0

**业务规则**
- MVP 阶段聚焦课堂基础闭环
- 不以完整复刻所有复杂课堂机制为目标
- 课堂生成属于长任务

**验收标准**
- AC1：用户输入主题后，可在目标时间内生成可展示课堂内容
- AC2：生成失败时返回可理解错误并支持重试
- AC3：生成过程中前端实时看到当前步骤和进度

### FR-CS-002 Agent 风格选择
**描述**  
用户可在预设老师风格中选择一种用于课堂或视频生成。

**优先级**  
P1

**业务规则**
- MVP 固定 4 种风格点缀色：严谨教授 `#4A6FA5`、风趣老师 `#FF9500`、温和导师 `#52C41A`、干练讲师 `#722ED1`
- 风格差异只作用于老师相关局部元素与生成语气，不允许切换全局页面主题
- 风格选择器以输入框附近的简单下拉框承载；MVP 只要求展示风格名称列表与当前选中态，结构可扩展，不额外承载视觉化设计

**验收标准**
- AC1：系统提供严肃、幽默、耐心、高效四种风格选项
- AC2：不同风格生成结果在语气、讲解方式和举例倾向上存在可辨识差异
- AC3：若用户未选择，系统使用默认风格

### FR-CS-003 幻灯片展示
**描述**  
课堂内容应包含基础幻灯片结构，支持前端展示。

**优先级**  
P0

**验收标准**
- AC1：生成结果包含多张结构化幻灯片
- AC2：每张幻灯片包含标题和正文要点
- AC3：前端可按顺序切换浏览

### FR-CS-004 会话结束信号与课后练习触发
**描述**  
课堂引擎在内容结束后输出结构化学习信号，用于触发课后的 checkpoint / quiz，而不是把正式练习硬插入课堂主叙事。

**优先级**  
P0

**业务规则**
- 课堂主叙事流中不插入强制计分题
- 课堂只负责输出可供 Learning Coach 消费的知识点、难点、步骤摘要与推荐练习信号
- 正式 quiz 由 Learning Coach 承接并回写学习中心

**验收标准**
- AC1：课堂结果可产出章节摘要、关键知识点和后续练习触发信号
- AC2：系统可从课堂结果跳转到 `/checkpoint/:sessionId` 或 `/quiz/:sessionId`
- AC3：相关结果进入学习记录与学习中心闭环

### FR-CS-005 多 Agent 讨论
**描述**  
课堂中支持多角色 Agent 参与讨论。

**优先级**  
P2

**验收标准**
- AC1：课堂中至少可展示多角色参与的讨论片段
- AC2：讨论过程顺序可控且可呈现
- AC3：用户作为学生输入后可获得回应
- AC4：讨论片段在 `/classroom/:id` 可回看，并可作为学习记录的一部分进入 `/learning`

### FR-CS-006 课堂进度反馈
**描述**  
课堂生成过程中应向用户反馈当前阶段和进度。

**优先级**  
P0

**验收标准**
- AC1：课堂生成时可建立实时进度连接
- AC2：前端可显示当前阶段描述和进度条
- AC3：连接断开后可恢复当前任务状态

### FR-CS-007 白板布局管理
**描述**

**优先级**  
P2

## 6.7 FR-KQ Evidence / Retrieval 服务

> 说明：`FR-KQ` 为历史编号保留，当前统一对应 Evidence / Retrieval 服务层，不代表学生端仍存在独立 Knowledge QA 主页面。

### FR-KQ-001 证据检索入口
**描述**

P0

- AC2：系统返回可阅读的答案结果，并尽可能附带来源依据

P1

### FR-KQ-005 术语解释

## 6.11 FR-PV Provider 与外部能力编排

P1

P0

**验收标准**
- AC1：EvidenceProvider、QuizFlowProvider、PathPlanningProvider 或 EvaluationProvider 可调用外部平台能力完成
- AC2：前端体验保持统一品牌风格

# 8. 数据与集成约束

## 8.1 长期业务数据
以下数据必须可持久化、可查询、可追溯：
- 视频任务元数据
- 课堂会话摘要
- 会话伴学问答记录
- 白板解释结果或动作日志
- 会话产物索引
- 学习记录
- 收藏记录
- 测验结果
- 问答记录
- 错题本数据
- 学习路径结果
- 推荐结果摘要（如需长期回看）

## 8.2 运行时数据
以下数据属于运行时状态，不要求作为长期业务表实时承接：
- 任务阶段进度
- SSE 过程事件
- 短期会话上下文
- 当前锚点窗口
- Companion 运行态窗口
- Provider 健康状态
- 临时恢复快照

## 8.3 集成边界
- 前端通过 HTTP / SSE 调用 FastAPI 获取功能能力
- 用户与基础业务能力与 RuoYi 保持一致
- FastAPI 与 RuoYi 之间通过防腐层交互
- 外部平台能力通过抽象层接入，不直接暴露原生前端组件
- Companion 优先消费会话产物源，必要时才补充调用 Evidence / Retrieval Service
- Learning Coach 消费学习行为源与结果数据，不在会话内直接承担即时答疑

## 8.4 数据归属原则
- 用户在个人中心、学习中心可见的数据属于长期业务数据，应进入 RuoYi/MySQL
- 文件产物进入 COS
- Redis 不作为学习记录、收藏、问答记录、会话伴学记录、测验结果的长期存储宿主

# 11. 测试与上线门禁

## 11.1 E2E 必过场景
### P0 必过
1. 用户登录 → 进入首页 → 进入单题视频场景
2. 文本题目输入 → 视频生成 → 播放视频

7. 长任务过程中进度展示正常，连接中断后状态可恢复

4. 白板高级布局

- 所有 P0 E2E 场景 100% 通过

- 关键性能指标达到基线
- 任务状态、进度、完成、失败四类路径均验证通过
- 长期业务数据落表与文件归档路径验证通过

# 12. Definition of Ready / Definition of Done

## 12.1 Definition of Ready
一个需求准备启动开发时，先满足基础条件：
1. 需求描述清晰
2. 优先级明确
3. 验收标准明确
4. 依赖已识别
5. 技术边界已确认不冲突
6. 可拆分为开发任务

后端 / 接口 Story 额外需满足：
1. API 契约、错误码、状态枚举、示例 payload、mock 样例与数据归属明确
2. 上下游依赖、测试夹具、接口测试边界明确
3. 若引用前端场景，线框图或页面结构说明足以表达触发路径即可

正式前端页面 Story 额外需满足：
1. 已有批准的高保真视觉稿，而不是只有线框图
2. 已覆盖空态、错态、加载态、禁用态与关键交互说明
3. 对应 API 契约、错误码、状态枚举、示例 payload 与 mock 样例已稳定，不处于高频变更期
4. 路由、组件边界、可访问性要求与验收截图范围已明确
5. 可基于 `adapter + mock handler` 先行完成页面状态闭环；真实接口联调、主分支合并与发布收口仍受四项门禁约束，但不要求等待后端真实实现全部完成

## 12.2 Definition of Done
一个需求完成前需满足：
1. 代码已合并
2. 所有 AC 通过
3. 对应测试通过
4. 无未关闭 P0 Bug
5. 接口与页面变更已同步文档
6. 测试环境验证通过
7. 不违反当前架构边界
8. 不破坏学习中心与个人中心职责边界，不引入路由口径漂移

# 13. 版本计划

## 13.1 阶段 A：契约冻结、mock 建立与并行启动
- 冻结统一 API 响应、错误码、状态枚举、示例 payload 与 mock 样例

## 13.2 阶段 B：双端并行实现与主链路闭环
- 后端完成视频 / 课堂任务主流程、SSE 恢复语义、Provider 基础能力验证与 RuoYi 业务表回写

- 仅在高保真视觉稿、关键状态、交互说明、稳定接口契约四项门禁齐备后执行主分支合并与发布收口

- P0 E2E 回归
- 稳定性优化
- 性能优化
- 演示环境部署
- 演示脚本固化

# 14. 变更控制

## 14.1 冻结规则
- 正式前端页面在高保真视觉稿、关键状态、交互说明已明确且 API 契约、错误码、状态枚举、示例 payload、mock 样例稳定时即可启动开发，不要求真实后端已全部完成
- 高保真视觉稿、关键状态、交互说明、稳定接口契约四项门禁用于真实接口联调、主分支合并与发布收口
- 接口契约冻结后，若需变更字段或状态枚举，必须评估并同步影响到的页面与测试基线
- 发版冻结阶段仅允许高优先级缺陷修复，不得引入跨模块新需求

## 14.2 变更原则
若新增需求满足以下任一条件，应默认不进入当前 MVP：
- 跨模块引入新依赖
- 影响主链路稳定性
- 需要新增复杂后台能力
- 无法在现有测试门禁下验证
- 对演示价值提升有限但开发成本高
