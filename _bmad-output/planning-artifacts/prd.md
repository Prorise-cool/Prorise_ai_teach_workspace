## 1. 执行摘要 (Executive Summary)

### 1.1 产品愿景

**小麦**是中国首个 AIGC 原生虚拟教室，通过双入口架构为高职教育提供沉浸式个性化学习体验：
- **入口 1（初学场景）**：完整虚拟教室体验，复刻 OpenMAIC 全部功能
- **入口 2（问答 Agent）**：单题讲解视频生成，基于 Manim 高质量动画

### 1.5 成功指标

| 指标类型 | 目标值 | 统计口径 | 采集方式 |
|----------|--------|----------|----------|
| **视频生成成功率** | > 80% | 分子：Manim 渲染成功且输出可播放 MP4 的请求数；分母：所有触发视频生成的请求数 | 后端 `video_render_result` 事件日志，按自然日统计 |
| **端到端延迟** | < 5 分钟（P95） | 从用户点击"生成"到前端收到可播放视频 URL 的时间差 | APM 链路追踪（请求 ID 贯穿），取 P95 值 |
| **学生满意度** | > 4.0/5.0 | 课堂/视频结束后弹出 5 分制评分，分子：所有评分总和；分母：有效评分次数（排除 < 3s 即提交的无效评分） | 前端 `satisfaction_submit` 埋点，按周统计 |
| **教师效率提升** | > 50% | 对照组：教师手动录屏制作同等时长视频的平均耗时；实验组：使用小麦生成同等时长视频的平均耗时 | 用户调研问卷 + 系统日志对比，按月统计 |

### 1.6 时间规划

| 里程碑 | 时间节点 | 核心交付物 |
|--------|----------|------------|
| **M1: 技术验证** | 第 1 周 | 双模块 Demo 跑通 |
| **M2: MVP 开发** | 第 2-4 周 | 完整 MVP 功能 |
| **M3: 测试上线** | 第 5 周 | 公测版部署 |
| **M4: 赛事提交** | 4 月 25 日 | 演示视频 + 答辩 PPT |

### 1.7 关键设计原则

1. **模块松耦合**：课堂服务与视频服务独立开发、独立部署
2. **共享层抽象**：LLM/TTS 通过抽象工厂模式，两边复用但互不依赖
3. **前端重做**：功能组件参考复用，风格组件必须重写（避免抄袭）
4. **管理端不动**：RuoYi 仅通过 API 对接，不修改其代码
5. **早期 i18n**：W2 阶段即集成国际化，降低后期改造成本

### 4.2 用户旅程

#### 旅程 1：高职学生 — 课后复习（入口 1 初学场景）

**角色：李明**
- 背景：珠海城职院大二学生，计算机网络专业
- 痛点：数据结构课程跟不上，递归算法怎么都理解不了
- 目标：期末考试前搞懂递归

**故事：**

> **开场**：周三晚上 10 点，李明对着数据结构课本发愁。明天的实验课要用递归实现快速排序，但他连递归的基本概念都没搞懂
>
> **上升**：他打开小麦 App，选择"初学场景"，输入"我想学递归算法"。系统提示选择 AI 老师风格，他选了"幽默型"——希望能轻松一点
>
> 3 分钟后，一个完整的虚拟课堂生成了。AI 老师"小幽默"开始用"俄罗斯套娃"的比喻讲解递归，配合动画展示函数调用栈的变化
>
> **高潮**：课堂进行到一半，AI 老师问："李明，你现在能告诉我递归的终止条件是什么吗？"李明犹豫了一下，打字回答。AI 老师给出即时反馈，又举了一个新的例子帮他巩固
>
> **结尾**：课后测验，李明 5 道题对了 4 道。他第一次觉得递归"原来这么简单"。他把课堂链接分享给了室友

**需求揭示**：课堂生成、AI 风格选择、交互式测验、即时反馈、分享功能

---

#### 旅程 2：高职学生 — 作业难题（入口 2 问答 Agent）

**角色：王芳**
- 背景：珠海城职院大一学生，会计专业
- 痛点：数学作业的微积分题目完全不会
- 目标：理解这道题的解题过程

**故事：**

> **开场**：周日下午，王芳盯着数学作业的最后一题——求函数的极值。课本上的例题和这道题完全不一样
>
> **上升**：她打开小麦 App，切换到"问答 Agent"模式，用手机拍了题目照片上传。系统识别后问她想要什么风格，她选了"耐心型"——希望能讲得详细一点
>
> 4 分钟后，一段 3 分钟的讲解视频生成了。视频里，AI 老师用 Manim 动画一步步画出函数图像，标注极值点，配合清晰的语音讲解
>
> **高潮**：王芳看到动画演示的一瞬间，突然理解了"导数为零"和"极值点"的关系。她调到 0.5 倍速，又看了一遍关键步骤
>
> **结尾**：她不但完成了这道题，还试着用同样的方法解出了下一题。她把视频收藏到了"微积分"文件夹

**需求揭示**：图片识别上传、题目理解、视频生成、倍速播放、收藏管理

---

#### 旅程 3：高职教师 — 备课场景

**角色：张老师**
- 背景：珠海城职院计算机系教师，负责《Python 程序设计》课程
- 痛点：每学期都要更新教学视频，传统制作太耗时
- 目标：一周内完成"文件操作"章节的 5 个教学视频

**故事：**

> **开场**：新学期还有一周开始，张老师看着去年的教学视频发愁。去年的视频是用录屏软件做的，画质一般，讲解也有点干巴
>
> **上升**：她登录小麦教师端，选择"视频服务"模块。她把 5 个知识点的提纲输入系统，每个知识点选择不同的 AI 风格——基础概念用"严肃型"，实战案例用"幽默型"
>
> **高潮**：系统并行生成 5 个视频，总计 25 分钟的 Manim 动画讲解。张老师抽查了其中一个，发现代码演示的动画效果比她自己录屏强多了
>

### 5.2 虚拟教室机制（复用 OpenMAIC）

**核心架构：**
- **DirectorGraph**：LangGraph 状态图，调度 Agent 发言顺序
- **Agent Registry**：智能体注册表，存储角色配置
- **Agent Roles**：teacher / student / assistant 三种角色

**"小麦" 的角色可塑性：**
- 通过替换**装扮**（头像、颜色、名字）和**角色定位**
- "小麦"可变成不同的**学生**或**教师**
- 支持用户自定义 Agent 人设（长期目标）

**运行机制：**
```
用户进入虚拟课堂
    ↓
Director 调度 Agent 发言
    ↓
Agent 生成内容（文本 + 动作）
    ↓
动作执行（语音/白板/幻灯片）
    ↓
多 Agent 对话互动
    ↓
用户作为"学生"参与课堂
```

| **Redis** | 连接超时 3s | JWT 验证降级为直接解码验证（不校验黑名单）；缓存降级为直接查库 | 自动重连，恢复后切回正常模式 |

## 13. 变更控制机制 (CR 流程)

### 13.1 需求冻结规则

| 阶段 | 冻结时间点 | 冻结范围 |
|------|-----------|----------|
| **M1（技术验证）** | 第 1 周开始 | P0 功能列表冻结 |
| **M2（MVP 开发）** | 第 2 周开始 | 全部 FR 冻结，仅允许 Bug 修复 |
| **M3（测试上线）** | 第 5 周开始 | 代码冻结，仅允许 P0 Bug 热修复 |
| **M4（赛事提交）** | 4/23 | 完全冻结，仅允许演示环境配置调整 |

### 13.2 变更请求流程

```
发起人提交 CR
    ↓
影响评估（技术负责人，≤ 1 个工作日）
    ├── 评估内容：工期影响、风险等级、关联模块、测试影响
    ↓
审批决策
    ├── 小变更（≤ 0.5 人天，无跨模块影响）→ 技术负责人审批
    ├── 中变更（0.5-2 人天，或跨模块）→ 技术负责人 + 产品负责人审批
    └── 大变更（> 2 人天，或影响里程碑）→ 项目负责人审批
    ↓
审批通过 → 更新 PRD/Epic/Story → 调整排期 → 执行
审批拒绝 → 记录原因 → 归档
```

### 13.3 变更请求模板

| 字段 | 说明 |
|------|------|
| **CR 编号** | CR-YYYY-NNN（自增） |
| **发起人** | 姓名 + 角色 |
| **发起日期** | YYYY-MM-DD |
| **变更描述** | 具体要改什么，为什么要改 |
| **影响范围** | 涉及的 FR/NFR 编号、模块、页面 |
| **工期影响** | 预估增加的人天数 |
| **风险等级** | 低/中/高 |
| **审批人** | 根据 13.2 规则确定 |
| **审批结果** | 通过/拒绝 + 原因 |
| **执行状态** | 待执行/执行中/已完成/已取消 |
