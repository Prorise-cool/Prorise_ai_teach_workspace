# 3. 架构对齐摘要

## 3.1 系统分层

系统采用双后端分层：

### FastAPI

负责：

* AI 功能执行
* 异步任务协调
* 视频与课堂生成
* Provider 调度
* SSE 进度推送
* 外部能力编排
* 运行时状态管理

### RuoYi

负责：

* 用户、角色、权限
* 小麦长期业务数据持久化
* 学习记录、收藏、任务元数据、问答日志等业务表承接
* 查询、审计、导出与管理宿主能力

## 3.2 数据边界

* Redis 只用于运行时状态、Token 在线态、SSE 事件缓存、短期上下文、Provider 健康状态
* Redis 中的数据必须设置 TTL
* 长期业务数据必须进入 RuoYi/MySQL 或 COS
* FastAPI 不承担长期业务主库存储职责
* 文件产物统一进入 COS

## 3.3 模块边界

* 课堂服务与视频服务是两个独立内容引擎，禁止互相耦合生成实现细节
* Companion 为共享会话层，消费内容引擎上下文，不承担内容生成流水线
* Evidence / Retrieval Service 为后台证据层，不承担“当前时刻解释”职责
* Learning Coach 为学后巩固层，不承担会话内即时答疑职责
* 各模块可共享任务框架、Provider 抽象层、SSE 基础设施、RuoYi 防腐层、COS 客户端
* 禁止跨业务模块直接耦合具体实现逻辑

术语冻结说明：`FR-KQ` 为历史编号保留，当前语义统一对应 Evidence / Retrieval 服务层，不代表学生端存在独立 Knowledge QA 主页面。

边界三分法冻结如下：

* Companion：解释当前这一步
* Evidence / Retrieval：提供资料依据与来源引用
* Learning Coach：负责学后练习、推荐与路径规划

| 能力域 | 主问题 | 主数据源 | 禁止承担 |
|---|---|---|---|
| Companion | 我现在这一步为什么这样？ | 会话产物源 | 通用资料库主问答 |
| Evidence / Retrieval | 这个知识点在资料里怎么定义？依据是什么？ | 内容知识源 + 证据索引 | 当前秒点解释 |
| Learning Coach | 我下一步该怎么练？ | 学习行为源 + 结果数据 | 会话内即时答疑 |

## 3.4 外部能力边界

* 腾讯云 ADP 当前作为默认外部 AI 能力实现接入
* 主链路稳定性不得依赖单一外部平台
* 用户前端体验全部自研，不直接暴露外部平台原生前端组件
* 外部依赖均需具备降级策略

## 3.5 用户配置系统与 Agent 动态生成

* 用户配置系统收集用户的个人简介和学习偏好，由后端动态生成适配的 AI agents
* 前端整体品牌风格统一，Agent 差异通过生成内容的语气、讲解方式体现，不允许按 Agent 切换全局页面主题
* 用户配置包含：昵称（可选）、个人简介（200字符以内）、语言选择
* 系统根据用户配置由后端生成适配的 Agent persona，而非前端提供预设风格选项
* i18n 与亮 / 暗色属于横切基础能力，必须从首个正式页面 Story 起进入开发与验收流程
