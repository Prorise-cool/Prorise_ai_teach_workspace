# 6.10 FR-PV Provider 与外部能力编排

## FR-PV-001 Provider 抽象与可替换性
**描述**  
系统需通过统一抽象层接入 LLM、TTS 等外部能力，避免业务逻辑与具体厂商实现强耦合。

**优先级**  
P0

**验收标准**
- AC1：业务模块通过统一接口调用 Provider 能力
- AC2：替换具体 Provider 时不需要重写业务主流程
- AC3：至少支持主备 Provider 切换能力

## FR-PV-002 Provider Failover
**描述**  
当主 Provider 不可用时，系统应自动切换备用 Provider。

**优先级**  
P0

**验收标准**
- AC1：主 Provider 故障时系统可切换到备 Provider
- AC2：切换过程对用户尽量透明
- AC3：全部 Provider 不可用时系统返回明确失败结果

## FR-PV-003 Provider 健康状态缓存
**描述**  
系统应缓存 Provider 健康状态，用于快速决策和降级。

**优先级**  
P1

**验收标准**
- AC1：系统可维护基础 Provider 健康状态
- AC2：健康状态为短期运行态数据，不进入长期业务表
- AC3：健康状态失效后可自动刷新

## FR-PV-004 腾讯云 ADP 外部能力编排
**描述**  
系统可通过抽象层调用腾讯云 ADP 的知识库、工作流、多智能体等能力，为知识问答和学习辅助提供支撑。

**优先级**  
P0

**业务规则**
- 前端不直接调用腾讯云原生组件
- 外部能力应以平台能力补充形式接入，而非接管主产品 UI
- 核心长期业务数据仍由 RuoYi 承担持久化

**验收标准**
- AC1：知识问答、课后小测或学习路径等能力可调用外部平台能力完成
- AC2：前端体验保持统一品牌风格
- AC3：外部平台不可用时系统具备明确失败或降级路径
