# 5. 版本范围

## 5.1 MVP 范围

> **实施节奏调整（2026-03-28）**
> 当前 MVP 采用“契约先行、mock 先行、前后端并行、联调与发布收口”的执行口径。
> API 契约、错误码、状态枚举、示例 payload 与 mock 样例先冻结；后端按契约提供真实 API 端点，前端可基于 `adapter + mock handler` 并行实现正式页面，不等待后端全部完成后再启动。
> 高保真视觉稿、关键状态、交互说明、稳定接口契约四项门禁继续保留，但仅用于真实接口联调、主分支合并与发布收口，不作为前端编码启动门槛。
> 在 UI 只有线框图或结构草图时，允许建设临时验证壳层，但不视为正式页面交付；当视觉稿、关键状态与交互说明已批准且契约稳定时，首页、输入页、等待页、结果页、学习中心域页面与个人中心域页面可直接基于 mock 启动实现。
> 学习中心与个人中心职责边界固定：`/learning` 承接学习结果聚合；`/history` 与 `/favorites` 属于学习中心域结果管理视图；`/profile` 仅承接个人资料；`/settings` 承接平台设置与账号偏好。

## 5.2 非 MVP 范围

当前版本明确不做：

* 自定义 Agent 市场/运营系统
* 完整 ToB 管理产品开发
* 社交互动生态
* 移动端原生 App
* SSR / SSG
* 高级学习分析系统
* 全平台深度国际化

## 5.3 MVP 验收原则

MVP 不是堆功能，而是满足：

* 双入口都能走通核心闭环
* 新增的 Companion、Evidence / Retrieval 与 Learning Coach 至少有可演示最小闭环
* 主链路稳定可演示
* 失败路径可解释
* 数据有归属
* 任务过程可见且可恢复
* 结果可回看
