# 6. 功能需求

## 6.1 FR-UM 用户与认证

### FR-UM-001 用户注册/登录

**描述**\
用户可以通过统一登录页（页面内支持切换到注册态）完成注册、登录、登出，并进入小麦前端功能页面。

**优先级**\
P0

**依赖**

* RuoYi 用户体系
* 前端统一认证页 `/login` 与鉴权拦截

**业务规则**

* 统一认证采用独立登录页（`/login`），页面内支持登录 / 注册态切换，不使用首页弹框
* 未登录用户访问受保护页面必须被拦截
* 登录态需在前端、FastAPI 与 RuoYi 层面保持一致
* FastAPI 不自建独立用户体系

**验收标准**

* AC1：用户输入正确凭证后，可在 2 秒内完成登录并进入首页
* AC2：未登录用户访问受保护页面时，自动跳转登录页
* AC3：登录失败时返回明确错误提示，不泄露敏感账户信息
* AC4：用户登出后原登录态立即失效

### FR-UM-002 Token 鉴权一致性

**描述**\
前端、FastAPI 与 RuoYi 对用户登录态保持一致判断。

**优先级**\
P0

**依赖**

* Redis 在线态
* 统一鉴权头

**验收标准**

* AC1：合法登录态可访问受保护的 FastAPI 接口
* AC2：失效或非法 Token 调用接口返回 401
* AC3：用户登出后再次调用受保护接口应失败

### FR-UM-003 个人资料查看与修改

**描述**\
用户可在个人中心查看并修改基础个人信息。

**优先级**\
P1

**验收标准**

* AC1：用户可查看昵称、头像、学校等基础信息
* AC2：用户可修改昵称和头像，并即时看到更新结果
* AC3：非法格式或超限文件上传被拒绝并提示原因

### FR-UM-004 权限分层

**描述**\
系统需基于 RuoYi 角色体系区分学生、教师和管理角色的能力边界。

**优先级**\
P1

**验收标准**

* AC1：学生角色仅可访问 ToC 学习功能
* AC2：教师角色可访问教师范围内的结果管理能力
* AC3：管理角色能力不直接暴露在 ToC 主导航中

## 6.2 FR-UI 前端页面与导航

\[Rule] 本节定义的是产品最终必须具备的页面能力，并要求采用“契约先行、mock 先行、前后端并行”的实施方式。
\[Rule] 当前执行口径为：先冻结 API 契约、状态模型、错误码、示例 payload 与 mock 样例；后端负责真实端点、数据回写与接口测试，前端基于 `adapter + mock handler` 并行完成正式页面与状态闭环。
\[Rule] 高保真视觉稿、关键状态、交互说明、稳定接口契约四项门禁用于真实接口联调、主分支合并与发布收口；在视觉稿与契约稳定时，正式页面可先基于 mock 开发，不必等待真实后端完成。
\[Rule] 若当前仅有线框图或结构草图，允许建设临时验证壳层，但不得把该壳层视为 FR-UI 的完成交付。
\[Rule] 路由口径固定：`/learning` 为学习中心聚合入口；`/history`、`/favorites` 是学习中心域结果管理视图；`/profile` 仅负责个人资料；`/settings` 仅负责平台设置与账号偏好。
\[Rule] 当前版本学生端不保留独立 `/knowledge` 路由；Evidence / Retrieval 只通过结果页、学习中心与 Learning Coach 的非路由面板承载。

### FR-UI-001 首页双入口展示

**描述**\
首页需清晰展示“主题课堂”和“单题视频”两个核心入口，并帮助用户理解差异。

**优先级**\
P0

**验收标准**

* AC1：首页展示两个核心入口
* AC2：每个入口包含简明文案说明适用场景
* AC3：用户点击入口后可在 1 秒内进入对应功能页面

### FR-UI-002 课堂页面

**描述**\
课堂页面承载课堂内容展示、幻灯片浏览、白板讲解与嵌入式 Companion 追问，不在主叙事中强插正式答题流程。

**优先级**\
P0

**验收标准**

* AC1：页面包含幻灯片区域、讲解区域和用户输入区域
* AC2：用户可顺序切换浏览幻灯片
* AC3：用户可在当前课堂步骤发起 Companion 追问并获得解释白板
* AC4：课堂完成后可进入 checkpoint / quiz 与结果反馈
* AC5：课堂相关路由至少覆盖 `/classroom/input`、`/classroom/:id/generating`、`/classroom/:id`

### FR-UI-003 视频生成页

**描述**\
视频生成页承载题目输入、图片上传、风格选择、任务创建与进度查看。

**优先级**\
P0

**验收标准**

* AC1：页面提供文本输入框与图片上传区域
* AC2：页面支持风格选择或默认风格
* AC3：任务提交后展示当前生成进度与阶段提示
* AC4：视频相关路由至少覆盖 `/video/input`、`/video/:id/generating`

### FR-UI-004 播放器页

**描述**\
播放器页承载视频播放、倍速、全屏、信息查看、结果操作与嵌入式 Companion 追问。

**优先级**\
P0

**验收标准**

* AC1：播放器位于页面主体区域
* AC2：播放器下方显示题目信息或知识点标签
* AC3：提供收藏、重新生成、返回等基础操作入口
* AC4：支持围绕当前播放时间点发起追问，并显示侧栏解释结果

### FR-UI-005 个人中心

**描述**\
个人中心只承接用户基础资料、账号安全入口，以及平台相关的选项设置；不再承担历史记录、收藏与学习结果聚合。

**优先级**\
P1

**验收标准**

* AC1：页面展示用户头像、昵称、学校信息
* AC2：页面提供设置入口，并能进入平台相关选项设置
* AC3：页面支持个人信息修改入口
* AC4：`/profile` 仅承接个人资料与资料修改，学习结果聚合必须放在 `/learning` 域

### FR-UI-006 资料证据面板与来源抽屉（非路由）

**描述**\
学生端不再提供独立资料证据路由；证据能力只以结果页或学习中心中的来源抽屉 / 证据面板形式出现，用于查看依据、来源片段与解析状态。

**优先级**\
P2

**验收标准**

* AC1：学生端证据能力只以非路由面板出现，不再新增独立 `/knowledge` 页面
* AC2：来源抽屉或证据面板可展示引用来源、文档片段与解析状态
* AC3：用户可从 `/video/:id`、`/classroom/:id`、`/learning` 或 Learning Coach 的“查看依据”动作打开
* AC4：学生端默认以查看与回看为主，资料上传解析入口不要求在学生端独立暴露

### FR-UI-007 学习中心页

**描述**\
学习中心用于聚合学习记录、收藏、问答记录、测验结果与推荐内容。

**优先级**\
P0

**验收标准**

* AC1：页面聚合展示至少历史记录、收藏、测验结果三个区域
* AC2：用户可从学习中心进入对应明细结果页
* AC3：页面展示结构清晰，不因聚合导致信息混乱
* AC4：`/history` 与 `/favorites` 作为学习中心域明细视图，不能反向绑定到 `/profile`

### FR-UI-008 国际化架构预留

**描述**\
前端需为中英双语切换保留架构能力。

**优先级**\
P2

**验收标准**

* AC1：静态文本具备国际化资源管理方式
* AC2：未来接入语言切换时不需要重写页面结构
* AC3：当前版本默认中文，不阻断 MVP 交付

## 6.3 FR-VS 视频服务

### FR-VS-001 题目输入

**描述**\
用户可通过文本输入或图片上传提交题目。

**优先级**\
P0

**依赖**

* 文本输入组件
* 图片上传组件
* OCR 能力

**业务规则**

* 文本输入为默认方式
* 图片上传是补充输入方式
* OCR 失败时必须提供手动输入替代路径

**验收标准**

* AC1：用户可提交文本题目并创建视频任务
* AC2：用户可上传 JPG/PNG 图片并提交视频任务
* AC3：图片识别失败时系统提示用户改为手动输入

### FR-VS-002 题目理解

**描述**\
系统需将题目解析为结构化理解结果，用于后续分镜与视频生成。

**优先级**\
P0

**验收标准**

* AC1：系统可提取题型、知识点和解题结构
* AC2：无法理解题目时返回明确提示而非直接报错
* AC3：理解结果可供后续阶段继续使用

### FR-VS-003 分镜生成

**描述**\
系统基于题目理解生成讲解视频分镜方案。

**优先级**\
P0

**业务规则**

* 默认分镜目标总时长为 `120 秒`
* 允许时长区间为 `90-180 秒`
* 分镜节奏按题目复杂度动态分配，不采用固定均分

**验收标准**

* AC1：系统生成不少于 3 个有效分镜
* AC2：每个分镜包含场景描述、讲解重点和旁白内容
* AC3：分镜总时长默认控制在 `120 秒` 附近，且位于 `90-180 秒` 区间内，不明显偏离题目复杂度

### FR-VS-004 Manim 代码生成

**描述**\
系统基于分镜生成可渲染的 Manim 动画代码。

**优先级**\
P0

**验收标准**

* AC1：生成代码通过基础语法检查
* AC2：代码包含完整可执行场景定义
* AC3：生成结果可进入渲染阶段继续处理

### FR-VS-005 代码自动修复

**描述**\
渲染失败后，系统自动尝试修复 Manim 代码。

**优先级**\
P0

**业务规则**

* 系统执行预设次数的自动修复，上限固定为 `2 次`
* 修复失败后任务必须正确结束，不得悬挂
* 高质量方案失败时可降级返回较低质量结果

**验收标准**

* AC1：首次渲染失败后自动进入修复流程
* AC2：系统最多执行 `2 次` 自动修复
* AC3：修复失败后返回清晰失败结果并记录任务状态
* AC4：存在降级结果时优先返回可用结果

### FR-VS-006 动画渲染

**描述**\
系统在受限环境中执行 Manim 渲染，生成视频素材。

**优先级**\
P0

**验收标准**

* AC1：任务成功时产出可用视频文件
* AC2：渲染超时或执行失败时任务状态被正确标记
* AC3：前端能看到阶段性进度变化

### FR-VS-007 TTS 合成

**描述**\
系统根据讲解内容合成语音音频。

**优先级**\
P0

**业务规则**

* 至少支持主备两类 Provider
* 主 Provider 失败时自动切换到备 Provider

**验收标准**

* AC1：至少支持 2 个可切换 TTS Provider
* AC2：主 TTS 不可用时系统自动切换备选 Provider
* AC3：当所有 TTS 不可用时返回明确失败或降级结果

### FR-VS-008 视频合成与上传

**描述**\
系统将动画与音频合成为最终视频并上传对象存储。

**优先级**\
P0

**验收标准**

* AC1：视频合成成功后输出可播放 MP4
* AC2：成功上传至 COS 后返回可访问视频 URL
* AC3：上传失败时执行重试或降级策略，并正确反馈状态

### FR-VS-009 视频任务进度反馈

**描述**\
视频任务执行过程中，用户应实时看到任务状态和进度。

**优先级**\
P0

**验收标准**

* AC1：任务创建成功后前端可建立事件流连接并接收进度更新
* AC2：进度覆盖理解题目、分镜生成、代码生成、渲染、TTS、合成、上传等关键阶段
* AC3：连接短暂中断后可恢复任务当前状态
* AC4：任务完成后前端可展示播放器入口或跳转播放页

## 6.5 FR-CS 课堂服务

### FR-CS-001 输入主题生成基础课堂

**描述**\
用户输入一个主题后，系统生成一套基础课堂内容。

**优先级**\
P0

**业务规则**

* MVP 阶段聚焦课堂基础闭环
* 不以完整复刻所有复杂课堂机制为目标
* 课堂生成属于长任务

**验收标准**

* AC1：用户输入主题后，可在目标时间内生成可展示课堂内容
* AC2：生成失败时返回可理解错误并支持重试
* AC3：生成过程中前端实时看到当前步骤和进度

### FR-CS-002 Agent 风格选择

**描述**\
用户可在预设老师风格中选择一种用于课堂或视频生成。

**优先级**\
P1

**业务规则**

* MVP 固定 4 种风格点缀色：严谨教授 `#4A6FA5`、风趣老师 `#FF9500`、温和导师 `#52C41A`、干练讲师 `#722ED1`
* 风格差异只作用于老师相关局部元素与生成语气，不允许切换全局页面主题
* 风格选择器以输入框附近的简单下拉框承载；MVP 只要求展示风格名称列表与当前选中态，结构可扩展，不额外承载视觉化设计

**验收标准**

* AC1：系统提供严肃、幽默、耐心、高效四种风格选项
* AC2：不同风格生成结果在语气、讲解方式和举例倾向上存在可辨识差异
* AC3：若用户未选择，系统使用默认风格

### FR-CS-003 幻灯片展示

**描述**\
课堂内容应包含基础幻灯片结构，支持前端展示。

**优先级**\
P0

**验收标准**

* AC1：生成结果包含多张结构化幻灯片
* AC2：每张幻灯片包含标题和正文要点
* AC3：前端可按顺序切换浏览

### FR-CS-004 会话结束信号与课后练习触发

**描述**\
课堂引擎在内容结束后输出结构化学习信号，用于触发课后的 checkpoint / quiz，而不是把正式练习硬插入课堂主叙事。

**优先级**\
P0

**业务规则**

* 课堂主叙事流中不插入强制计分题
* 课堂只负责输出可供 Learning Coach 消费的知识点、难点、步骤摘要与推荐练习信号
* 正式 quiz 由 Learning Coach 承接并回写学习中心

**验收标准**

* AC1：课堂结果可产出章节摘要、关键知识点和后续练习触发信号
* AC2：系统可从课堂结果跳转到 `/checkpoint/:sessionId` 或 `/quiz/:sessionId`
* AC3：相关结果进入学习记录与学习中心闭环

### FR-CS-005 多 Agent 讨论

**描述**\
课堂中支持多角色 Agent 参与讨论。

**优先级**\
P2

**验收标准**

## 6.7 FR-KQ Evidence / Retrieval 服务

> 说明：`FR-KQ` 为历史编号保留，当前统一对应 Evidence / Retrieval 服务层，不代表学生端仍存在独立 Knowledge QA 主页面。

### FR-KQ-001 证据检索入口

**描述**\
系统需支持从 Companion、Learning Coach、学习中心或结果页中的来源抽屉触发证据检索，用于补充课程知识点、教材内容或术语依据；入口仅为来源抽屉 / 证据面板 / 学习中心回看入口，不新增学生端独立 `/knowledge` 页面。

P0

**验收标准**

* AC1：系统支持以会话内补证据或会话后证据回看两种方式发起检索
* AC2：系统返回可阅读的答案结果，并尽可能附带来源依据
* AC3：失败时提供重试、改写问题或回退纯解释提示

### FR-KQ-002 文档上传与解析

**优先级**\
P1

**依赖**

* 文档解析能力
* 外部知识库能力

**业务规则**

**验收标准**

* AC1：支持上传规定格式文档进行解析
* AC2：解析完成后文档内容可用于问答检索
* AC3：解析失败时返回明确错误原因
* AC4：基于已解析文档产生的问答记录可在学习中心重新查看

### FR-KQ-003 证据问答与依据补充

**描述**\

**优先级**\
P0

**验收标准**

* AC1：系统可围绕课程知识返回针对性回答
* AC2：回答结果与问题语义保持相关
* AC3：无结果时系统返回明确说明而不是编造答案

### FR-KQ-004 引用来源展示

**描述**\
问答结果应尽可能展示引用来源，帮助用户理解答案依据。

**优先级**\
P1

**验收标准**

* AC1：有来源时可展示文档片段、章节或来源标记
* AC2：来源展示不影响主答案阅读
* AC3：来源缺失时系统明确说明来源不足

### FR-KQ-005 术语解释

**描述**\
系统支持对证据检索或会话补证据中的专业术语进行简明解释。

**优先级**\
P1

**验收标准**

* AC1：用户可对术语发起解释请求
* AC2：返回内容适合当前目标用户理解水平
* AC3：术语解释结果可与主问答结果联动展示

### FR-KQ-006 问答记录回写

**描述**\

P0

**验收标准**
