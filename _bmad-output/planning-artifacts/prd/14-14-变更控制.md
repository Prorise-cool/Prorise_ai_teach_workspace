# 14. 变更控制

## 14.1 冻结规则

* 正式前端页面在高保真视觉稿、关键状态、交互说明已明确且 API 契约、错误码、状态枚举、示例 payload、mock 样例稳定时即可启动开发，不要求真实后端已全部完成
* 高保真视觉稿、关键状态、交互说明、稳定接口契约四项门禁用于真实接口联调、主分支合并与发布收口
* 接口契约冻结后，若需变更字段或状态枚举，必须评估并同步影响到的页面与测试基线
* 发版冻结阶段仅允许高优先级缺陷修复，不得引入跨模块新需求

## 14.2 变更原则

若新增需求满足以下任一条件，应默认不进入当前 MVP：

* 跨模块引入新依赖
* 影响主链路稳定性
* 需要新增复杂后台能力
* 无法在现有测试门禁下验证
* 对演示价值提升有限但开发成本高
